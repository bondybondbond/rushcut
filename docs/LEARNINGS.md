## Workflow ‚Äî computer-use burns round-trips hunting for a window across multi-monitor setups; check `windowLocations` first (2026-07-23)

**Problem:** Confirmed live checking `rushcut.exe`'s console on this user's 2-monitor setup: several `screenshot`/`switch_display` round-trips were spent guessing which monitor actually held the target window before finding it, because `screenshot` only ever shows ONE display at a time and there's no window-enumeration tool in the computer-use toolset.
**Solution:** `request_access`'s response already includes a `windowLocations` array (`{bundleId, displays: [{id, label, isPrimary}]}`) for every granted app, populated at grant time ‚Äî read that FIRST and `switch_display` directly to the reported label, instead of screenshotting each display in turn to look for the window. Re-check `windowLocations` (call `request_access` again with the same app) if the app was just moved or relaunched, since the field reflects state at grant time, not live.
**Context:** Any computer-use task on a machine with 2+ monitors, especially when the target app isn't the one currently in focus.

## Workflow-GateMiss ‚Äî Round 4 (wrap-readiness) approved #149 on code review alone; live testing found 5+ real UI bugs it couldn't have caught (2026-07-22)

**What was approved:** `rushcut-pp-consultant`'s Round 4 wrap-readiness check for #149 (mid-roll cards) gave a clean APPROVE, explicitly noting "no live behavioral verification happened this session" was acceptable given the constraints (couldn't safely touch the user's live dev server), backed by careful static tracing + a WebSearch-based trap-check (Round 2.5) of the DnD mechanism specifically.

**Solution:** no rubric change needed yet (see wrapup Step 0.9 ‚Äî action threshold is ‚â•3 entries sharing a recurring pattern; this is the first). ## Workflow ‚Äî `enforce-skill-gate.js` false-triggers "wrapup" on background-notification/subagent text, not just genuine user messages (#149, 2026-07-22)

**Problem:** The hook is meant to force `Skill(rushcut-wrapup)` only when the *user* types a trigger phrase, but its trigger-detection scans the transcript rather than checking whether the current message is a genuine new user chat turn. Twice in one session: (1) a background task-notification carrying a subagent's result text (which happened to discuss "file the follow-up issue at wrapup, not now") falsely triggered it in the orchestrator ‚Äî caught and confirmed false via `AskUserQuestion`, cost one round-trip; (2) far worse, the SAME false trigger fired mid-task inside the `rushcut-real-pp-auditor` subagent itself, which has no `Skill` tool to clear the gate ‚Äî its entire competitor-research task was aborted with no recovery, losing all its work.
**Solution:** Not yet fixed ‚Äî filed as issue #153 (tooling, P1). Until fixed: treat any wrapup-trigger firing during a background-task notification as suspect; verify with the user via `AskUserQuestion` before invoking `Skill(rushcut-wrapup)` rather than assuming it's genuine. If a subagent (not the orchestrator) reports getting blocked by this hook mid-task, its work is unrecoverable for that invocation ‚Äî don't wait for it, re-invoke fresh once the transcript's false-trigger phrase has scrolled out of whatever window the hook scans.
**Context:** Applies to any session using subagents (Agent tool, background `run_in_background: true`) alongside the wrapup-trigger hook ‚Äî the risk is highest right after a subagent's own report discusses wrapup-adjacent topics (issue filing, session closing) without the user having actually said "wrap up."

## Workflow ‚Äî cleaning up a manual test-render's temp files deleted the real, shared render-timing-log.jsonl (#148, 2026-07-22)

**Problem:** After running a manual `wsl -- python3 run.py` test render (verifying #148's card-splice logic against a scratch manifest under `%TEMP%\rushcut\`), the cleanup command `rm -rf` targeted the test's own manifest/log/output by name ‚Äî but also included `render-timing-log.jsonl`, assuming it was per-test output. It isn't: `%TEMP%\rushcut\` is the SAME directory Rust writes every real render's manifest to, and `render-timing-log.jsonl` there is the one shared, cumulative history file every real render appends a line to (see `docs/speed-goal.md`, `.claude/rules/pipeline.md`'s render-cache/speed sections). The file was deleted with no git tracking and no recovery path. (In this specific incident the loss was low-stakes ‚Äî the user confirmed all meaningful measurements were already written up in `docs/speed-goal.md`, the jsonl was just raw source ‚Äî but the deletion itself was still a mistake, not a safe default.)
**Solution:** Before any `rm`/`Remove-Item` on a shared temp directory (`%TEMP%\rushcut\`, WSL `/tmp`, or any dir Rust/run.py also writes to), list the directory first and delete ONLY files whose name embeds the test's own job-id/scratch marker ‚Äî never a bare glob or a hardcoded "cleanup list" copied from wrapup's own Step 5 (that list is scoped to wrapup's own known test-artifact names, e.g. `eval-test-film-*`; a one-off manual test script needs its own equally-scoped delete list, not a superset that happens to include a real shared file by coincidence of directory).
**Context:** Applies to any manual pipeline verification run outside the normal dev-plan/wrapup flow (a scratch manifest + direct `run.py` invocation to prove a code change works) ‚Äî the cleanup step after such a test needs the same "list before delete, scope to files you created" discipline as any other destructive-action check, even though the directory is "just a temp folder."

## Workflow ‚Äî a formal scored self-improvement loop (per a generic "AI Loop Best Practices" framework) was evaluated and rejected for this project's dev-tooling gates

**Problem:** A generic framework for self-improving AI loops (Trigger ‚Üí Execute ‚Üí Score ‚Üí Gate(green/yellow/red) ‚Üí Memory) implies any recurring AI-gated routine should have a numeric/categorical score and threshold tiers, not just binary pass/fail. Applied literally to RushCut's 5 dev-tooling pieces (`rushcut-dev-plan`, `rushcut-wrapup`, `rushcut-pp-consultant`, `rushcut-real-pp-auditor`, `rushcut-qa-reviewer`), all 5 lack a Score and a tiered Gate ‚Äî reads like a real gap.
**Solution:** Evaluated 2026-07-21 by two independently-run reviewers of different architectures (`rushcut-pp-consultant`/Claude and real Perplexity/GPT-5.6 Terra via `rushcut-real-pp-auditor`) ‚Äî both converged on rejecting the formal loop, for reasons the framework itself predicts: (1) the framework's "score=automation-not-a-loop" warning assumes no human oversight; a solo builder reviewing every session already IS that oversight, so PASS/OBJECTION functions as verbally-encoded green/red. (2) Genuine LLM-as-judge calibration needs a ~100-200 example hand-labeled ground-truth set this project doesn't have and would cost real effort to build. (3) A single person acting as both labeler and rubric-author has a documented structural risk of unconsciously tuning the rubric to confirm their own pre-existing judgments (cited: Shankar et al., "Who Validates the Validators?", ICML 2024) ‚Äî exactly the "echo chamber" failure mode the source framework itself warns about, just via a mechanism the framework doesn't name. Kept the existing binary gates + bounded retries (already satisfies the framework's "bounded retries" escape hatch) and added one cheap fix instead: a `Workflow-GateMiss`-tagged log entry whenever a gate is later found wrong, audited every ~10 sessions (`rushcut-wrapup` Step 0.9) ‚Äî evidence-gated escalation (‚â•3 same-pattern misses) rather than building infrastructure ahead of data.
**Context:** Applies whenever a generic AI-orchestration/agent-loop best-practices framework is proposed for this project's own dev tooling (not user-facing product features) ‚Äî re-derive from this entry's reasoning before re-running the same evaluation from scratch. Two-architecture consultant convergence (same-architecture subagent + genuinely different foundation model via real Perplexity) was itself the signal that made this a trustworthy "no" rather than a single-model rubber-stamp.

## Workflow-gh ‚Äî backticks inside a double-quoted `gh issue comment --body "..."` trigger bash command substitution, silently corrupting the posted comment

**Problem:** A markdown-formatted comment body containing inline-code spans like `` `docs/DESIGN.md` `` or a quoted phrase like `"drag-left swipe"` passed via `--body "..."` in the Bash tool gets parsed by Git Bash itself ‚Äî backticks trigger command substitution (tries to execute the enclosed text as a shell command), silently dropping or mangling that segment of the posted comment. The `gh` call still succeeds and returns a valid comment URL, so nothing visibly fails; the corruption is only visible by re-reading the posted comment back.
**Solution:** Write the comment body to a file (e.g. in the scratchpad dir) and pass it via `gh api repos/<owner>/<repo>/issues/comments/<id> -X PATCH -f body="$(cat file.md)"` (or `--body-file` on `gh issue comment` at creation time) instead of an inline double-quoted string with backticks. After any `gh issue comment`/`gh issue create` call whose body contains backticks, re-fetch it (`gh issue view <n> --json comments --jq '.comments[-1].body'`) to confirm it wasn't silently mangled, since the command's own success/URL output does not indicate corruption.
**Context:** Any `gh issue create`/`gh issue comment` call with a markdown body containing inline code spans, run via the Bash tool (Git Bash). Confirmed on #35 wrapup (2026-07-16) ‚Äî a comment segment referencing a quoted gesture name was silently dropped.

## Workflow-WSL ‚Äî piping a `wsl ...` command's output into a PowerShell-side cmdlet fails silently on Unix-only tools

**Problem:** `wsl -- grep "..." /mnt/c/.../file.log | tail -30` runs `grep` inside WSL but the `|` is parsed by PowerShell itself, not passed through to WSL ‚Äî the pipe target (`tail`) then executes as a PowerShell command, which doesn't exist there ("tail: command not recognized"). Any Unix-only filter (`tail`, `head`, `wc`, `sort`) chained onto a `wsl -- <cmd>` call this way fails the same way.
**Solution:** Wrap the entire pipeline inside a single WSL invocation: `wsl -- bash -c "grep '...' /path/to/file | tail -30"`. Everything after `bash -c "..."` stays in the WSL shell, so `|` and any Unix tool resolve correctly.
**Context:** Any PowerShell-tool call that reads/filters a WSL-side log or file (`pipeline-latest.log`, `proxy-bg.log`, etc.) using more than one command.

## Workflow-WSL ‚Äî PowerShell's `>` redirect on a `wsl -- <cmd>` call treats a `/mnt/c/...` argument as a Windows path, not WSL output

**Problem:** `wsl -d Ubuntu-24.04 -u root -- python3 run.py ... > "/mnt/c/Users/.../out.txt" 2>&1` fails with `Could not find a part of the path 'C:\mnt\c\Users\...'`. The `>` is parsed by PowerShell (same class of bug as the `|`-pipe trap above), so PowerShell tries to open a file at the literal string `/mnt/c/Users/...` relative to its own drive-rooted semantics -- prepending `C:\`, producing the nonsensical `C:\mnt\c\...` path.
**Solution:** Either wrap the whole thing in `wsl -- bash -c "... > /path/to/out.txt 2>&1"` so the redirect happens inside WSL against a real WSL path, or drop the redirect entirely and read the already-written per-job log file instead (`run.py` always writes `pipeline-<job_id>.log` and `ffmpeg-stderr-last.log` next to the manifest -- for a failed ffmpeg command, the full stderr is already captured there without needing to redirect anything).
**Context:** Any attempt to capture full stdout/stderr from a `wsl -- <cmd>` call for later inspection, e.g. debugging an ffmpeg filter_complex error. Confirmed while diagnosing #136's zoom+open/close timebase bug -- `ffmpeg-stderr-last.log` (already written by the pipeline's own ffmpeg wrapper) turned out to be the right file to read anyway, making the redirect attempt unnecessary in the first place.

## Workflow ‚Äî check `render-timing-log.jsonl` before designing a perf refactor off an issue's estimated %

**Problem:** GitHub issues with a numeric performance estimate (e.g. "40-60% wall-clock reduction") can be stale the moment a different, unrelated batch ships ‚Äî the estimate was made against the pipeline as it existed when filed, not as it exists now. Designing a refactor straight off that estimate risks building real complexity for a bottleneck that no longer exists.
**Solution:** Before designing any perf-optimization batch, read `%TEMP%\rushcut\render-timing-log.jsonl` (already populated with `t_normalise_s`/`t_render_s`/`t_total_s`/`proxy_used` per real render since Batch T-era) and check whether the targeted stage is actually a meaningful fraction of `t_total_s` on representative real data. Issue #20 (V4.2 decode/encode overlap) estimated 40-60%; the log showed `t_normalise_s` at 1.9-4% of total across every real render, proxy-covered or not ‚Äî proxy caching (unrelated work) had already captured the win. One log read avoided a multi-session refactor of `render.py`'s Step 1-5 orchestration.
**Context:** Any session starting a `V4.x`/pipeline-performance issue, or any issue whose body cites an estimated speedup/slowdown percentage.

## Workflow ‚Äî grep the WSL pipeline log for a specific tag, don't `tail` a large range first

**Problem:** `tail -N` on `pipeline-latest.log` (or any real-render log) with N large enough to "probably" cover the section of interest returns tens of KB of noise (per-clip sync-check lines, ffmpeg command dumps) before the actually-relevant line is visible, wasting context on a read that a targeted search would have answered directly.
**Solution:** Grep the log for the specific tag first (`grep -n -i encoder`, `grep -n -i cache`, `grep -n -i fallback`) ‚Äî these tags are already stable, documented log prefixes (`[encoder]`, `[cache]`, `[U1g]`) used consistently across the pipeline. Only fall back to `tail`/broader reads if the grep comes up empty and the stage genuinely hasn't been reached yet.
**Context:** Any session reading `pipeline-latest.log` or `pipeline-{job_id}.log` via WSL to verify a real render's behavior (encoder choice, cache hit/miss, fallback, timing). Also note: PowerShell's `wsl -- grep -E "a|b|c"` mangles the `|` (WSL's `wsl <args>` reconstructs and re-shells the argv on the Linux side, so an unescaped `|` becomes a real pipe even inside a PowerShell-quoted string) ‚Äî use single-term greps or a properly escaped `\|` instead of `-E` alternation.

## Workflow ‚Äî a render-cache HIT during back-to-back A/B verification silently skips the thing being tested

**Problem:** Verifying an encoder/pipeline change by re-rendering the same project with identical settings right after a baseline render can hit the V4.1 render cache (`[cache] HIT` in the log) ‚Äî Steps 1-5 (including the actual video encode) are skipped entirely, and only the cheap music/loudnorm remux reruns (~30-40s instead of several minutes). The render "succeeds" and completes suspiciously fast, which reads as a bug (or a huge unexplained speedup) if the cache hit isn't checked for. Confirmed on #110: an hevc_amf A/B test's second render silently reused the h264_amf-encoded cached intermediate ‚Äî encoder never ran, real comparison never happened, until the log was checked.
**Solution:** Always grep the log for `[cache] HIT` vs `[cache] MISS` before trusting a suspiciously-fast real-render verification. To force a fresh encode for A/B testing on the same project/settings, delete the specific cached file (`grep [cache] sig=` for the hash, then `rm` it from the cache dir) rather than changing settings, which would also change what's being compared. Same trap applies to the "hand-written manifest + direct `wsl python3 run.py`" repro pattern (re-running the exact same manifest JSON used for an earlier real render to "re-verify a fix" also silently hits `TIMING:cache=hit`) ‚Äî there, mutate one cache-signature field (e.g. bump a clip's `in_ms` by 1) to force a miss, or check the `render_cache=hit/miss` field in the `ANALYSIS:` line, and always give the copied manifest a fresh `job_id` + throwaway `output_path` so it doesn't clobber the original job's temp dir or the user's real output file.
**Context:** Any real-render A/B verification (encoder swap, bitrate change, filter change) on the same project without an intervening settings change, or any manual-manifest pipeline re-verification (see #98/#99 precedent) re-running the SAME manifest twice. Cache dir location: see the `USERPROFILE`-not-inherited-into-WSL gap below (it's often NOT at the intended `%APPDATA%\rushcut\render-cache\` path).

## Workflow ‚Äî a manifest-injected debug flag with zero effect may mean the binary was never rebuilt, not that the flag is broken

**Problem:** #118 Run B set `RUSHCUT_FORCE_NORMALISE=1` correctly (confirmed via env var + launch order) and the render still took the proxy-reuse path every time, with no error. Reading `pipeline-latest.log` alone couldn't distinguish "flag not wired correctly" from "flag never reached the process" ‚Äî both look identical from the log's perspective (same `[C-proxy] ... using proxy` line either way). Root cause: the `force_normalise` code (#120, commit `24ed8ae`) shipped in a session that only ran `cargo check` ‚Äî `cargo build` was silently skipped because the running binary held the file lock (see the MEMORY.md note from that session). The `.exe` on disk was still the pre-#120 build, so the manifest's `settings` object never had a `force_normalise` key at all ‚Äî not `false`, just absent.
**Solution:** When a manifest-injected debug/config flag appears to have zero effect despite verified end-to-end code wiring (grep confirms the Rust read + Python read + decision-point check), don't debug the launch mechanism first ‚Äî check two things in this order: (1) read the actual manifest JSON (`%TEMP%\rushcut\<job_id>.json`) for the key's literal presence, not just the log's derived message; (2) compare the running binary's file timestamp (`ls -la --time-style=full-iso target/debug/rushcut.exe`) against `git log -1 --format=%ci -- src-tauri/src/lib.rs` (or whichever source file owns the flag) for the commit that added it. If the binary predates the commit, `cargo build` never actually happened for that change ‚Äî kill all `rushcut.exe` instances to release the file lock, rebuild, then retest.
**Context:** Any session testing a Rust-injected manifest field (`force_normalise`, `use_hevc_amf`, or any future `RUSHCUT_*` debug env var) that appears not to take effect. Also relevant any time a prior session's summary says a Rust change was verified via `cargo check` rather than `cargo build` ‚Äî that phrasing specifically means the binary was never relinked.

## Workflow ‚Äî chained `Select-String | Select-String` does not AND-match across separate log lines

**Problem:** Piping one `Select-String` into another to combine two patterns (e.g. find lines matching pattern A, then filter for pattern B) silently returns nothing when A and B live on *different* log lines that are logically related but not textually adjacent ‚Äî e.g. `proxy-bg.log`'s `[PROXY_BG] done clip_id=... elapsed=...` and the following `[PROXY_BG] fan-out local_path=... proxy_path=...` line are two separate emits per clip, so filtering "done lines" then re-filtering for a proxy-hash substring (which only appears on the fan-out line) always returns empty.
**Solution:** Use a single regex with alternation/lookaround over the raw file instead of chaining match objects across two `Select-String` calls, or grep for the unique identifier that actually appears on the line you need (e.g. the source clip filename on `encode-start` lines) rather than assuming two conceptually-linked facts share one line.
**Context:** Any PowerShell log-parsing task combining two conditions from a structured multi-line log emitter.

## E2E ‚Äî msedgedriver pinned version can drift out from under WebView2/Edge auto-update

**Problem:** `pnpm test:e2e` failed with `session not created: This version of Microsoft Edge WebDriver only supports Microsoft Edge version 148. Current browser version is 150.0.4078.48` ‚Äî the msedgedriver binary at `C:\Users\<you>\.cargo\bin\msedgedriver.exe` (hardcoded path in `wdio.conf.ts`, not npm-managed) is pinned to whatever version was downloaded once; Windows auto-updates the system Edge/WebView2 runtime independently and silently drifts ahead of it. Not caused by any app code change ‚Äî pure environment drift between sessions.
**Solution:** Download the exact matching build from `https://msedgedriver.microsoft.com/<version>/edgedriver_win64.zip` (the version string must match `msedge.exe`'s `FileVersion` exactly ‚Äî check via `(Get-Item "C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe").VersionInfo.FileVersion`), extract, and overwrite the binary at the hardcoded path (`wdio.conf.ts:177`). **`msedgedriver.azureedge.net` (the old documented CDN) no longer resolves** ‚Äî use `msedgedriver.microsoft.com` instead. Confirmed fix 2026-07-09: v148‚Üív150, smoke suite went from session-create failure to 9/9 passing. Keep a `.bak` of the old binary before overwriting (cheap rollback, zero cost).
**Context:** Any session running `pnpm test:e2e*` that fails with "session not created" + a version mismatch message. Will recur whenever Windows auto-updates Edge past the pinned driver ‚Äî check `msedgedriver --version` vs the installed Edge version first before assuming a real test regression, then re-run this same download-and-swap fix.

## Workflow ‚Äî background `until` monitor output is unreliable; read the log directly when user says "done"

**Problem:** A background Bash `until` poll loop writing to a temp file produces an empty output file when the loop exits ‚Äî the final `tail` command runs but its output goes to the background task file which is already "done" from the harness perspective. When the user says "it's finished", the background task's output file is empty or stale.
**Solution:** When the user says a long-running operation (render, build, proxy gen) is done, always read the source log directly (`wsl -- tail -N /mnt/c/.../pipeline-latest.log`) rather than checking the background task output file. The source log is authoritative; the monitor is just a convenience trigger.
**Context:** Any session using `run_in_background: true` with an `until` polling loop. The monitor is useful for being notified when the condition is first met ‚Äî but for reading results, go to the source.

## Workflow ‚Äî WebView2 cold-start race with Vite

**Problem:** If the Tauri binary starts before Vite is serving on port 1420, WebView2 navigates to localhost:1420, gets a connection-refused response, shows a chrome-error page, and does NOT retry. The window shows black forever. `wdio.conf.ts`'s `ensureViteRunning()` guards against this for WDIO runs, but manual binary launches are vulnerable.
**Solution:** Always confirm Vite is serving (`curl -s http://localhost:1420/ -o /dev/null`) before launching the binary. Use `pnpm dev` (which sequences Vite first, then cargo run) rather than launching the binary directly when possible.
**Context:** Manual debugging sessions where binary is launched from PowerShell or bash separately from Vite.

## E2E ‚Äî msedgedriver.azureedge.net CDN fully decommissioned; auto-download must use msedgedriver.microsoft.com with a major-version fallback

**Problem:** `msedgedriver.azureedge.net` (the CDN host used by most msedgedriver "download the matching driver" tooling/examples, including the obvious first implementation) is permanently dead ‚Äî DNS resolution fails outright ("name does not exist"), confirmed 2026-07-13. Separately, even after switching to the correct host `msedgedriver.microsoft.com`, an exact-version download can still 404: a live Tauri app's WebView2 process can report a browser build (via CDP `/json/version`, e.g. `Chrome/150.0.7871.101`) that has no published driver at all, while `LATEST_RELEASE_<major>` (e.g. `LATEST_RELEASE_150`) resolves to a same-major but very different patch/build (`150.0.4078.65`) that DOES exist. WebView2 Evergreen runtime builds and the msedgedriver catalog don't always publish in lockstep, even on the same major version.
**Further wrinkle (same session, CONFIRMED not a restart-lag artifact):** on this machine, the registry-tracked installed WebView2 Runtime version (`HKLM:\SOFTWARE\WOW6432Node\Microsoft\EdgeUpdate\Clients\*` `pv` value), the on-disk install folder (`C:\Program Files (x86)\Microsoft\EdgeWebView\Application\`), AND `pnpm tauri info`'s own reported WebView2 version all agree on `150.0.4078.65` (which has a published driver) ‚Äî while the actual LIVE `rushcut.exe` process reports `150.0.7871.101` via CDP `/json/version`, which has no published driver anywhere. The original hypothesis (background-update-in-progress lag, self-resolves on restart) is **disproven**: restarted the binary to a fresh PID and got the identical `150.0.7871.101` report. Ruled out: `webviewInstallMode` in `tauri.conf.json` (not set ‚Äî not a fixed/embedded runtime), a `WEBVIEW2_BROWSER_EXECUTABLE_FOLDER` env override (not set, machine or user scope), a per-user WebView2 install (`%LOCALAPPDATA%\Microsoft\EdgeWebView\Application\` doesn't exist), and a bundled `WebView2Loader.dll` in `target/debug/` (not present). Three independent, mutually-agreeing sources (registry, disk, `tauri info`) vs. one live-CDP outlier, with every known override path checked and ruled out ‚Äî treat this as accepted drift risk (per real Perplexity's call, 2026-07-13), not an unsolved mystery worth further live debugging. If it recurs, the new pretest warning below at least surfaces it clearly instead of a cryptic driver-session error.
**Solution:** `e2e/helpers/edgeDriver.ts` (`ensureMatchingEdgeDriver()`, shared by both `wdio.conf.ts` and `wdio.attach.conf.ts`) does: (1) compare only the first 3 version components between pinned driver and live browser (patch drift alone doesn't require a re-download), (2) on mismatch, try the exact-version zip at `https://msedgedriver.microsoft.com/<version>/edgedriver_win64.zip`, (3) on 404, fall back to `LATEST_RELEASE_<major>` (decode as UTF-16LE with BOM ‚Äî these pointer files are NOT UTF-8, a naive `fetch(...).text()` garbles every other byte) and download that version instead, (4) after installing, re-check whether the new driver actually compatibility-matches the live browser and print a clear human-readable warning if not, rather than letting a cryptic `unrecognized Microsoft Edge version` surface deep inside WebDriver session-creation retries with no context. This replaces the old approach of hand-pinning one `msedgedriver.exe` and manually re-downloading it every time WebView2 auto-updates.
**Context:** Any session hitting `session not created: cannot connect to microsoft edge` / `unrecognized Microsoft Edge version` when starting a WDIO session against a live Tauri WebView2. Check `msedgedriver.exe --version` vs the CDP `/json/version` report first ‚Äî the pretest warning above now does this automatically. Do not assume a restart will fix a registry-vs-live-CDP version disagreement ‚Äî confirmed it doesn't, on this exact scenario. If all three of registry/disk/`tauri info` agree with each other and only CDP disagrees, that's accepted drift, not a bug to keep chasing.

## Diagnostics ‚Äî AMD GPU TDR events appear in Application log (WER 1001), not System log

**Problem:** Windows Event ID 141 (GPU watchdog timeout / VIDEO_TDR_FAILURE) is stored as a kernel dump file (`WATCHDOG-YYYYMMDD-HHMM.dmp`) and reported to the **Application** log as WER Event 1001 with `Event Name: LiveKernelEvent` and `P1: 141`. Querying the System log for IDs 4101, 141, etc. returns nothing even when a TDR just occurred ‚Äî that is the wrong log.
**Solution:** To detect AMD TDR events: `Get-WinEvent -FilterHashtable @{ LogName='Application'; StartTime=... } | Where-Object { $_.Id -eq 1001 -and $_.Message -like '*141*' }`. The `Attached files:` field in the message contains the WATCHDOG dump path (e.g. `WATCHDOG-20260614-2208.dmp`) ‚Äî the filename timestamp is the exact crash time. `AMD_REPORT_UM-*.dmp` files in the same `LiveKernelReports\` directory are secondary confirmation. All WER 1001 events for recent crashes are batched and logged the minute after the crash, all with the same `TimeCreated` second.
**Context:** `scripts/diagnose-freeze.ps1` (fixed in U5c to also check Application log). Any "freeze investigation" on AMD hardware. Dump files at `C:\WINDOWS\LiveKernelReports\WATCHDOG\` ‚Äî requires admin to list directly.

## Pipeline ‚Äî U1g segmented render falls back to monolithic when WSL /tmp is cleared mid-job

**Problem:** `_render_segmented()` writes segment files (`u1g_seg_N.mp4`) and the concat manifest (`u1g_concat.txt`) to `/tmp/<job_id>/` (WSL tmpfs). Under memory pressure from two sequential large 4K encodes, WSL can clear the directory between the last segment encode completing and the concat-assembly write. `concat_list.write_text()` then raises `[Errno 2] No such file or directory`; the outer `except Exception` catches it and falls back to the monolithic path ‚Äî exactly the OOM scenario U1g was designed to avoid. Confirmed on job `db8f3aa6`: both batches completed (`drift=0`), then `/tmp/<job_id>/u1g_concat.txt` write failed.
**Solution:** Move U1g segment files and concat manifest to NTFS `%TEMP%\rushcut\<job_id>\` ‚Äî the same pattern already used by zoom-cache and `warm_zoom.py`. NTFS survives WSL memory events; tmpfs does not. FFmpeg can read NTFS via `/mnt/c/...` paths. U1g concat txt entries need the `/mnt/c/` prefix. This is Batch U4c.
**Context:** `pipeline/render.py` `_render_segmented()` ‚Äî `TMP_BASE = Path("/tmp")` line ~128; `concat_list.write_text()` line ~897. Symptom in logs: two `[U1g] segment frames=N expected=N drift=0` lines followed immediately by `[WARNING] [U1g] segmented render failed ([Errno 2] No such file or directory: '/tmp/.../u1g_concat.txt') -- falling back to monolithic`. **Issue #7 follow-up (2026-06-18):** U4c fixed the `[Errno 2]` path but the blanket `except Exception` still silently drops to monolithic on any failure (OOM/exit-15, drift, etc.). Added structured instrumentation (`[U1g][fallback] class=... exc_type=... ffmpeg_exit=... mem_avail_mb=... batch=.../... batch_len=... clips_total=... exc=%r`) and an explicit `[U1g] segmented render complete (no fallback) batches=... clips_total=... mem_avail_mb=...` success marker. Hardening (planner-vs-other classification + BATCH_SIZE retry 4‚Üí3‚Üí2) deferred until a real `[U1g][fallback]` line is captured.

## Pipeline ‚Äî U1g both-boundary (open+close) drift falls back to monolithic, which then OOMs and can crash WSL itself (#114, FIXED 2026-07-11)

**Problem:** A different trigger than the `/tmp`-cleared case above, same destination and much worse outcome. On a 20-clip 4K project with **both** `opening_transition` and `closing_transition` active, all 7 U1g batches and both boundary re-encodes (`idx=0 open`, `idx=6 close`) succeeded, but the post-encode frame-drift guardrail then fired: `[U1g] boundary-only open/close frames=4310 expected=4312 (base=4306 + oc_delta=6) drift=2 frame(s)` ‚Äî real gain was only 4 frames total vs. the formula's assumed 6 (`oc_delta_expected = round(0.1*fps_f)` per boundary, i.e. 3+3). The `RuntimeError` was caught by the same blanket `except Exception`, discarding all the already-successful segmented work and falling back to a full 20-input monolithic filter_complex. That OOM'd on `h264_amf` (`Cannot allocate memory`), retried on `libx264 -preset medium`, and THAT run drove WSL from ~11.3 GB available down to 89 MB / swap nearly full before being OOM-killed with no `ERROR:`/`DONE:` ever logged. WSL itself was left unstable afterward ‚Äî a subsequent unrelated `wsl` command hit `Catastrophic failure / WSL/Service/E_UNEXPECTED` and needed `wsl --shutdown` to recover. User only saw a generic "Pipeline timed out -- check WSL2 is running."
**Root cause (confirmed via per-boundary instrumentation):** the assumed formula was symmetric (3 frames per active boundary) but real ffmpeg `xfade` semantics are NOT symmetric between an opening and a closing transition. The opening xfade uses `offset=0` (black fades into inner starting at t=0); per ffmpeg's own xfade semantics (`output_duration = offset + second_stream_duration`), an offset of 0 means the output duration is governed entirely by the second stream (the inner content) ‚Äî the black source's own `black_dur - xfade_dur` tail never gets added. Measured: **open boundary delta = 0 frames** (not the assumed +3). The closing xfade starts at `offset = total_after_open - xfade_dur > 0`, so it DOES extend the timeline by roughly `black_dur - xfade_dur` ‚Äî measured **close boundary delta = 4 frames** (close to, but not exactly, the assumed +3). Real total delta for both-on = 0 + 4 = 4, not the assumed 6 ‚Äî that 2-frame gap was the entire bug.
**Solution:** Two independent fixes, both shipped: (1) the drift-check `RuntimeError` was given a distinct `_BoundaryDriftError(RuntimeError)` subclass (module-level, so `isinstance()` checks are reliable), and the `except Exception` block around `_render_segmented()` now special-cases it via `isinstance(e, _BoundaryDriftError)` **inside the same except block** (not a second `except` clause ‚Äî a more-specific clause ordered after a general `except Exception` would never fire) to skip the monolithic fallback entirely and raise a clean, user-facing error instead. This is a standing safety net for *any* future correctness-check trip on this path, independent of the numeric fix. (2) The `oc_delta_expected` formula was replaced with a live-measured accumulator (`oc_actual_frame_delta`) ‚Äî `_boundary_reencode()` now probes frame count before/after each individual boundary re-encode and sums the real per-boundary deltas, so the guardrail compares the final concat against **actual measured segment growth** instead of a theoretical duration-based formula. This also upgrades the check's meaning: it now validates concat/mux integrity rather than a fragile xfade-offset assumption, so it can't be wrong again for the same class of reason.
**Context:** `pipeline/render.py` ‚Äî `_BoundaryDriftError` class (~line 227), `oc_actual_frame_delta` accumulator (~line 914) mutated inside `_boundary_reencode()` (~line 1123), guardrail read at ~line 1310-1326, except-block carve-out at ~line 1360+. If a similar duration-based frame-count assumption ever needs revisiting elsewhere in the xfade/boundary code, remember: an `xfade`/`acrossfade` at `offset=0` does NOT extend total duration by `first_input_duration - transition_duration` ‚Äî only a non-zero offset does. Don't assume symmetry between an opening and a closing transition's frame contribution again.

## React ‚Äî `useRef` initial value is reset on every component remount

**Problem:** `useRef(initialValue)` is only called once per *component instance*. A route transition that unmounts and remounts a component creates a fresh instance ‚Äî every `useRef` reverts to its initial value. This silently resets any ref that was updated by effects or event listeners in the previous mount, including timestamps, flags, and accumulated state.
**Solution:** For any ref that represents "last known state of something that continues across navigations" (e.g. last-pipeline-activity timestamp, accumulated scroll offset), seed the correct value inside a `useEffect` that runs shortly after mount ‚Äî not as the `useRef` initial value. The effect fires after mount and before any polling interval can read the ref, so it always overwrites the stale default before it matters. Do NOT hoist the seed into the `useRef(...)` call or `useState` init, where it captures only the value at mount time (which may itself be stale).
**Context:** `Render.tsx` `lastProgressAtRef` stall detection (U1e). Any component with a ref that tracks "ongoing external state" across navigations ‚Äî the typical case is a timestamp or index seeded from DB/API data loaded by a `useEffect`.

## React ‚Äî double-advance guard for clip boundary race

**Problem:** When a clip ends at its `out_ms` boundary, both `onTimeUpdate` (which detects `v.currentTime >= outSec`) and `onEnded` (which fires at file end when `out_ms === duration_ms`) can both trigger `advanceClip` in the same JS event loop tick, causing double-advance (skips a clip).
**Solution:** Use a ref guard: `const isAdvancingRef = useRef(false)`. At the top of `advanceClip`, check `if (isAdvancingRef.current) return; isAdvancingRef.current = true;`. After loading the new clip, clear it: `setTimeout(() => { isAdvancingRef.current = false; }, 250)`. The 250ms window covers the time between the two events firing.
**Context:** Any sequential clip player where `onTimeUpdate` and `onEnded` both drive clip advance. Confirmed in `Sound.tsx` `advanceFilmClipRough`.

## React ‚Äî per-clip video player must seek to in_ms on loadedmetadata

**Problem:** Loading a raw clip or proxy into a `<video>` element with `video.load()` starts playback from `currentTime=0` (the beginning of the raw file). If the trimmed section starts at `in_ms > 0`, `currentMs` never reaches `in_ms`, so derived film-time formulas (`currentMs - in_ms`) stay negative (clamped to 0) and any playhead or timeline feature appears frozen.
**Solution:** In `handleLoadedMetadata`, after setting `durationMs`, immediately seek: `video.currentTime = inMs / 1000`. Also set `currentMs` to `inMs` so the scrubber initialises at the trim start. In `handleTimeUpdate`, check `ms >= outMs` and stop+clamp there. Set scrubber `min={inMs}` / `max={outMs}`, and display `currentMs - inMs` / `outMs - inMs` for elapsed/total. In `togglePlay`, if `currentTime * 1000 >= outMs`, seek back to `inMs` before calling `play()` to allow replay.
**Context:** Any screen with per-clip video playback that respects `clip.in_ms` / `clip.out_ms` (Arrange zoom tab).

## React ‚Äî conditional render unmounts media elements

**Problem:** `{condition && <video>}` unmounts the `<video>` element when `condition` becomes false, dropping the browser's decoded buffer and seek state. When `condition` becomes true again and a `useEffect` with that condition as a dependency re-fires, `video.load()` is called on a fresh element ‚Äî causing a full reload stutter even if the source file hasn't changed.
**Solution:** Replace `{condition && <div>...</div>}` with `<div className={condition ? "flex" : "hidden"}>...</div>` (or `display:none`). The element stays in the DOM with its src and `currentTime` intact. Add a loaded-src ref (`loadedSrcRef`) to the `useEffect` and skip `video.load()` when returning to the same clip.
**Context:** Any screen where a `<video>` (or `<audio>`) is inside a conditionally rendered block that gets toggled by tab switches, drawer toggles, or modal state.

## React ‚Äî IIFE in JSX causes stray-closing-tag compile errors

**Problem:** Wrapping JSX in an IIFE (`{(() => { const x = ...; return (<JSX />); })()}`) to define local variables inline is fragile ‚Äî the surrounding `return` and the JSX opening/closing tags must perfectly balance. A single extra or missing `</div>` inside the outer component produces a cryptic Babel "Adjacent JSX elements must be wrapped" error that points at the wrong line, making root cause non-obvious.
**Solution:** Derive local variables in the component body above `return` (or in a small helper function), then reference them in the render path normally. Only use an IIFE if the variable depends on other JSX-internal context that truly can't be hoisted; if so, isolate it to the smallest possible scope and comment the intent.
**Context:** Any JSX block where you need computed values (derived from state/props) that aren't worth a new component or a `useMemo`. Spotted during Batch M1 transition card-chip implementation.

## Tauri ‚Äî `window.confirm` silently fails without `dialog:allow-confirm`

**Problem:** In a Tauri WebView2, `window.confirm(...)` is internally routed to the `dialog` plugin. ## React ‚Äî a `setState` call earlier in the SAME async function is not visible to code later in that same continuation

**Problem:** `async function onMount() { ... setOutputFolder(x); ... await submitJob(...) }` ‚Äî code inside `submitJob` (or any function it calls) that reads the `outputFolder` state variable via closure still sees the value from the render that *defined* the closure, not `x`, because React state updates only become visible on the next render ‚Äî which hasn't happened yet within the same `await`-chained continuation. **Solution:** Either (a) thread the fresh value through as an explicit override parameter (the existing `is4kOverride` pattern), or (b) mirror the state into a `useRef` that's updated synchronously in the same setter call (`function setX(v) { xRef.current = v; setXState(v); }`), and read `xRef.current` from any code that might run in the same continuation the state was just set in. (b) is less plumbing when the value is read from many call sites.
**Context:** Any mount-effect or event handler that both sets state AND (directly or via a called function) reads that same state later in the same `async` body before the next render ‚Äî e.g. Render.tsx's #13 folder-gate resume path (`outputFolderRef` mirror added to fix this exact case).

## FFmpeg ‚Äî filter_complex

- **`crop` filter latches `iw`/`ih` from the FIRST FRAME when upstream `scale=eval=frame` produces variable-size output** ‚Äî `crop`'s `iw` variable is evaluated at filter init (the first frame) and frozen there. For gradual zoom-in (zf starts at 1.0): `iw_latched = src_w = out_w`, so `(iw-ow)*fx = 0` for every frame regardless of focal point ‚Üí always zooms top-left. For zoom-out (zf starts at ratio): `iw_latched = src_w*ratio`; as zoom shrinks the valid range narrows until x exceeds it ‚Üí rightward drift, then hard left snap. Fix: substitute the same `zf(t)` expression that drives scale, applied to known Python-constant source integers, directly into the crop origin: `crop=out_w:out_h:'(2*trunc(SRC_W*ZF/2)-out_w)*fx':'(2*trunc(SRC_H*ZF/2)-out_h)*fy'`. Because `zf` contains FFmpeg's `t` variable, the crop tracks the correct focal every frame. Bump the zoom cache key version constant whenever the formula changes so stale cached files are not reused.
- **Mixed portrait+landscape requires fixed-canvas pre-scale before concat/xfade** ‚Äî `scale=-2:{h}` appended after concat/xfade produces streams of different widths (e.g. 1920px landscape vs 540px portrait); FFmpeg aborts with exit 234. Fix: pre-scale every input stream to an exact canvas with named labels `[sv0]`, `[sv1]`... before any concat or xfade reference: `[{i}:v]scale=W:H:force_original_aspect_ratio=decrease,pad=W:H:(ow-iw)/2:(oh-ih)/2[sv{i}]`. Both the `"none"` (concat) path and xfade path must be updated ‚Äî missing one path means the crash survives for certain transition settings.
- **SAR mismatch between proxy clips and zoom-reencoded clips causes exit 234** ‚Äî DJI proxy files carry SAR 3321:3320 (non-square pixels from the container metadata). libx264 zoom-reencoded clips carry SAR 1:1. FFmpeg concat filter raises `Input link in0:v0 parameters (SAR 3321:3320) do not match output link (SAR 1:1)` and exits 234. Fix: add `,setsar=1` to the end of the canvas pad filter in the pre-scale string so ALL streams (proxy passthrough and reencoded) are normalised to SAR 1:1 before any concat or xfade. Must be added to BOTH `build_batch_video_fc` (U1g segmented path) and `build_filter_complex` (monolithic path) in `transitions.py` ‚Äî missing one path leaves the bug on that code branch. Canvas string: `scale=W:H:force_original_aspect_ratio=decrease,pad=W:H:(ow-iw)/2:(oh-ih)/2,setsar=1`.
- **xfade transition name is `fade` not `crossfade`** ‚Äî `crossfade` raises "Not yet implemented in FFmpeg". Use `xfade=transition=fade`. For dip-to-black use `xfade=transition=fadeblack` (native, no custom logic needed).
- **`scale` must be inside `-filter_complex`** ‚Äî using `-vf` alongside `-filter_complex` on the same output stream raises "Simple and complex filtering cannot be used together". Append scale as the final step inside the filter chain.
- **Get durations from trimmed paths, not normalised paths** ‚Äî xfade offset formula uses per-clip duration; if trim runs before transitions, re-run `get_duration()` on the trimmed files or offsets will be silently wrong.
- **Pairwise `acrossfade` chaining for 3+ clips requires `apad=whole_dur` first** ‚Äî WITHOUT apad, chained acrossfade misaligns because each clip's audio is 64-120ms shorter than its video duration, causing crossfade start points to drift off the xfade offsets. WITH `apad=whole_dur=durations[i]` per clip, audio duration matches video exactly, so chained acrossfade produces perfect alignment for all N. Using `concat` for 3+ clips instead causes audio to lag 1.5s behind video from the second cut onward (hard-cut at t=sum(durations[:2]) while video xfade ends at t=sum(durations[:2])-xfade_dur).
- **xfade offset formula** (port verbatim from spike): `offset = cumulative + duration[i-1] - xfade_dur * i`
- **xfade_dur must be clamped to half the shortest clip duration** ‚Äî a 1.5s xfade consumes a 1s clip entirely (both the preceding xfade and the next one eat the same clip). Clamp: `effective_dur = min(xfade_dur, min(durations) / 2.0)` before building the filter chain. Log a warning if clamping occurs.
- **`color` inline sources for `xfade` must have `settb` matching the clip's container timebase (exit 234)** ‚Äî A `color=c=black:r=30000/1001` source generates timebase `1001/30000` (reciprocal of the fps rational). libx264-encoded H264 clips use `tbn = fps_numerator`, so timebase `1/30000`. FFmpeg 6.1.1's `xfade` filter rejects mismatched timebases with `First input link main timebase (X) do not match the corresponding second input link xfade timebase (Y)` (exit 234). Fix: append `,settb=1/30000` to the color filter chain ‚Äî derive it from the fps rational via `_fps_to_tbn(fps_raw)`: `f"1/{fps_raw.split('/')[0]}"` (take the numerator). This applies to ALL color sources in xfade chains: post-pass open/close blacks, monolithic opening/closing sources, any synthetic source paired with an H264-encoded stream.
- **The same timebase mismatch (exit 234) also happens between two REAL streams, not just a color source vs. a real stream ‚Äî specifically when a Ken Burns zoom (`scale=eval=frame,crop`) chain feeds an `xfade`, AND separately when a `concat` filter sits between the zoom chain and the `xfade`.** Two independent renegotiation points, both needing their own fix (#137, found+fixed during #136, 2026-07-19): (1) `scale=eval=frame`'s dynamic per-frame re-evaluation renegotiates its output timebase to a generic `1/1000000` instead of preserving the input container's `1/30000` ‚Äî fix by appending `,settb={clip_tbn_str}` directly to the per-clip canvas node (after zoom+scale+format+setparams, before the `[sv{i}]` label). (2) Separately, `concat` ALSO renegotiates its own output timebase and discards any `settb` its inputs already declared ‚Äî even a `concat=n=1` (single-clip passthrough) does this ‚Äî so a second, distinct `settb` node is needed immediately after the between-clips concat/xfade stage, right before that combined stream is handed to a further xfade (e.g. an open/close transition). Confirmed via real ffmpeg stderr: `First input link main timebase (1/30000) do not match the corresponding second input link xfade timebase (1/1000000)`, reproduced identically on both a 1-clip and a 2-clip project ‚Äî this is a pre-existing bug in the general xfade path whenever per-clip zoom and an xfade (between-clip OR open/close) combine in one filter_complex pass, not something specific to single-clip renders. It went undetected because the two code paths that historically combined zoom+xfade both avoided combining them in ONE ffmpeg invocation (a two-pass design: bake zoom into a real intermediate file first, THEN xfade that file ‚Äî a concretely-encoded file's timebase is deterministic and never hits this).
- **The #137 bug requires a zoom/xfade chain's output to be wrapped in a SECOND xfade layer ‚Äî a single flat xfade chain (no wrap) never hits it, even with real dynamic zoom feeding directly into xfade (#138, verified 2026-07-21, not applicable).** `build_batch_video_fc()` (U1g per-batch path, `transitions.py`) never had the `#137` `settb` fix applied ‚Äî deliberately left untested. Live-tested with a real 8-clip/3-batch render, `kb_in_*` dynamic Ken Burns zoom (genuine `scale=eval=frame,crop`, not just static "medium" crop) present in every batch, feeding straight into `xfade` against non-zoomed clips: exit 0, `drift=0`, no crash. Root cause of the non-reproduction: `build_batch_video_fc` is a single flat pairwise-xfade chain writing straight to `[vout]` ‚Äî structurally identical to `build_filter_complex()`'s own safe plain-crossfade between-clips branch (also never the #137 crash site). #137 only manifests when that chain's output is handed to a *further* xfade layer (the open/close-to-black wrap) ‚Äî U1g's open/close is handled entirely separately via `build_open_close_post_fc()`/`_boundary_reencode()`, which already had `clip_tbn_str` wired with a `"1/30000"` default from #65/#114. Lesson: a timebase-renegotiation bug's trigger condition can be narrower than "zoom + xfade in one pass" ‚Äî confirm the EXACT graph shape (single chain vs. nested/wrapped xfade) before assuming a fix needs to spread to every code path that superficially looks similar.

## FFmpeg ‚Äî codec / output

- **Proxy reuse as normalise input (Batch C candidate)** ‚Äî after B-0 + parallel normalise, the remaining 80s normalise bottleneck is 4K HEVC software decode (hardware accel not viable ‚Äî `/dev/dxg` present, `VK_KHR_video_decode_queue` not supported). The Trimmer already generates H.264 1080p proxies per clip. If render.py uses those proxies as normalise input instead of raw 4K HEVC, it reads H.264 (fast decode, ~5√ó faster than HEVC software) and skips the HEVC decode entirely. Estimated normalise: 80s ‚Üí ~20-30s with no parallelism. Spec: proxy must match normalise output exactly (`-c:v libx264 -pix_fmt yuv420p -profile:v main -preset ultrafast -ar 48000 -r 25 -fps_mode cfr`). If proxy resolution is 480p (current), render.py scale step still upscales ‚Äî acceptable for 1080p output. Flag as Batch C candidate once B is shipped.
- **Concurrent FFmpeg in WSL2 causes contention** ‚Äî proxy generation fired immediately at project create ran concurrently with the render pipeline (normalise + render), adding ~90s to a 5-min render. Fix: fire proxy generation only after `pipeline-done`. Never start a WSL2 FFmpeg background task while a render is in progress.
- **WSL2 HEVC normalise: I/O-bound from NTFS, CPU-bound from tmpfs** ‚Äî the original warning (ProcessPoolExecutor slows things down) applied when normalise read full 4K source clips from NTFS. After the B-0 pre-trim step copies only the needed segment to WSL2 `/tmp` (tmpfs = RAM), normalise reads from RAM and becomes CPU-bound (HEVC software decode). At that point `ThreadPoolExecutor(max_workers=4)` IS effective: sequential 160s ‚Üí parallel 80s for 10 clips. Cap threads per worker with `-threads N` (global FFmpeg flag before `-i`) to avoid over-subscription: `threads_per_worker = max(1, os.cpu_count() // max_workers)`. NOTE: the global `-threads` flag in FFmpeg does not reliably cap libx264's internal thread pool ‚Äî libx264 manages its own threads independently. Measured effect on this machine (16 CPUs, 4 workers, `-threads 4`): no further improvement beyond the 2√ó already gained from the 4-worker pool.
- **B-0 pre-trim before normalise: biggest single render speedup** ‚Äî normalise.py processes whatever files it receives. If full 4K source clips (1‚Äì5 min) are passed, it wastes 4‚Äì10√ó time normalising footage the user trimmed away. Fix: before calling `normalise()`, FFmpeg copy-trim each clip to `[in_s - 2s, out_s + 0.5s]` using `-ss before -i -c copy` (fast, ~1-3s per clip). Adjust `in_ms`/`out_ms` in the pipeline_clips dict to be relative to the pre-trimmed file's timeline. Result: 10 min normalise ‚Üí 3 min total pipeline for a 1m26s DJI 4K film (10 clips). The 2s pre-roll ensures keyframe alignment so the copy trim doesn't cut mid-GOP.
- **Use `-preset ultrafast` for normalised intermediates** ‚Äî normalise.py produces temp files that are re-encoded again by the xfade/concat render step. Using `-preset fast` for these intermediates wastes 3‚Äì4√ó CPU on motion estimation for files that will be re-encoded anyway. `ultrafast` produces the same final output quality at ~3‚Äì4√ó lower normalise time (confirmed: ~3 min ‚Üí ~60‚Äì90s on 3√ó4K clips). Always use `ultrafast` for intermediate normalise; `fast`/`medium` is only meaningful for the final render output.
- **Always specify `-c:v libx264 -pix_fmt yuv420p -profile:v main`** ‚Äî omitting `-c:v` after `-filter_complex` can silently fall back to HEVC, which Windows Photos/Media Player rejects with error 0x80004005 **when the machine has no HEVC decoder installed**. Reframed 2026-07-11 (#85): this is not "WMP rejects HEVC" universally ‚Äî WMP/Media Foundation decode HEVC fine once the Microsoft HEVC Video Extension is present (confirmed via a real `hevc_amf`-encoded test file: opened AND played, position advanced, no artifacts). The extension is a separate MS Store package, not guaranteed on every end-user machine ‚Äî that's the real risk for any *deliberate* HEVC output (see `## FFmpeg ‚Äî hevc_amf viable but needs a runtime capability probe, not an install-time assumption` below), not a reason to avoid it outright.
- **Single-clip shortcut**: use simple `-vf "scale=-2:360"` without `-filter_complex` ‚Äî avoids needless complexity and the constraint that scale can't be in both `-vf` and `-filter_complex`.
- **`-map 0:a:0?` not `-map 0:a?`** ‚Äî DJI clips can contain multiple audio streams; `0:a?` maps all of them. Always use the indexed form when normalising DJI footage.
- **Force `-ar 48000` at every re-encode site** ‚Äî DJI Osmo Pocket 3 records at 96kHz; some players (including certain mobile decoders) reject non-48kHz AAC. Add `-ar 48000` to every FFmpeg call that re-encodes audio: normalise, inject_silence, single-clip render, multi-clip render, music mix, loudnorm. One missed site means the final output inherits 96kHz from the concat stream.
- **Multi-clip render path needs explicit audio codec args** ‚Äî the filter_complex concat output has no implicit codec; omitting `-c:a aac -b:a 128k -ar 48000` causes FFmpeg to auto-select, inheriting the source sample rate. Always include audio codec args on every FFmpeg output regardless of path taken.
- **Single-pass loudnorm LUFS accuracy: real bar is +-2.0 on acrossfade content** ‚Äî two-pass loudnorm (linear=true) hits -14 LUFS within +-0.3 LU on any content. Single-pass (dynamic mode, fused into the encode) hits -14 +-0.5 on single-clip and music-on paths, but lands ~-15.7 (1.7 LU low) on multi-clip acrossfade chains where the dynamic compressor undershoots on level transitions. This is inherent to single-pass; it is not distortion, just quieter. Accepted bar for consumer output: +-2.0 LUFS (i.e. -12 to -16). If precise loudness is required (broadcast or streaming platform delivery), revert to two-pass or add a volumedetect correction pass.

## FFmpeg ‚Äî hevc_amf viable but needs a runtime capability probe, not an install-time assumption

**Problem:** #85 investigated switching the final render encode from `h264_amf` to `hevc_amf` for a speed win. Standalone benchmark (ffmpeg 8.0.1, AMD AMF, 12s/4K test clip): `hevc_amf` at the *same* bitrate as the current `h264_amf` tier encoded ~2.3x faster; at *half* bitrate (the expected equivalent-quality target for H.265), still ~2.1x faster at ~50% file size, with no visible quality loss at any tier tested. **Solution:** Any real implementation (#110) must probe `hevc_amf` capability at runtime (mirror the existing `_detect_amf()` pattern in `encoder.py`) and gate it behind an explicit opt-in rather than a silent default, since there's no way for RushCut to know whether the *playback* machine (which may not be the render machine) has the extension. Always pass `-b:v` explicitly on any `hevc_amf` codec_args ‚Äî ffmpeg >=7.1 has an open AMD AMF SDK bug ([AMF#514](https://github.com/GPUOpen-LibrariesAndSDKs/AMF/issues/514)) where omitting `-b:v` silently defaults to 20M and ignores `-maxrate` entirely, invalidating any bitrate/file-size comparison.
**Context:** `pipeline/encoder.py` `video_encoder_args()` if/when a `hevc_amf` branch is added. Full benchmark data: issue #85 (closed GO) + issue #110 (shipped as a file-size win, speed win did not hold ‚Äî see next entry).

## FFmpeg ‚Äî hevc_amf's encode-side speed win does not survive the real U1g pipeline (#110, shipped 2026-07-11)

**Problem:** #85's standalone benchmark (single straight `ffmpeg` encode pass, no decode/filter overhead) showed hevc_amf ~2.1-2.3x faster than h264_amf. **Context:** `pipeline/encoder.py` `video_encoder_args()` hevc_amf branch, `pipeline/render.py` `_boundary_codec_args()`. ## FFmpeg ‚Äî `-progress pipe:1` field is `out_time_us` (microseconds), and `out_time_ms` is also microseconds despite the name (#119, shipped 2026-07-15)

**Problem:** `ffmpeg -progress pipe:1` emits both `out_time_ms=` and `out_time_us=` lines. `out_time_ms` is a long-standing FFmpeg misnomer ‚Äî it is ALSO in microseconds, not milliseconds, so parsing it as milliseconds silently computes progress fractions ~1000x too small.
**Solution:** Always parse `out_time_us=` (the unambiguous field), never `out_time_ms=`. Divide by `1_000_000.0` to get seconds. Guard for `out_time_us=N/A` (emitted before the first frame) and clamp the resulting fraction to `[0, 1]` before using it.
**Context:** `pipeline/utils.py` `ffmpeg_run_progress()` (#119, interior render progress ticks). Confirmed via a standalone spike (2026-07-15) that `-progress pipe:1` streams incrementally ‚Äî not bursted at process exit ‚Äî on BOTH native WSL ffmpeg AND Windows `ffmpeg.exe` invoked from WSL via the interop bridge (the same binary the AMF encode path uses), so this parsing is safe to wire unconditionally for both encoder families. Any future ad-hoc spike script that invokes Windows `ffmpeg.exe` directly from WSL must pass a `C:\...`-style output path, not a raw `/mnt/c/...` path ‚Äî Windows executables cannot parse WSL path syntax and fail with "No such file or directory" (distinct from the reverse-direction trap already documented for Python scripts writing Windows-path strings from WSL).

## SQLite ‚Äî ISO 8601 T-separator breaks datetime() comparisons

**Problem:** `created_at` timestamps stored via Rust `chrono::Utc::now().to_rfc3339()` are ISO 8601 format (`"2026-06-06T20:12:11Z"`, T separator and Z suffix). SQLite's `datetime('now', ...)` function returns space-separated format (`"2026-06-06 20:12:11"`, no Z). A raw `created_at < datetime('now', '-900 seconds')` string comparison always fails for ISO 8601 values because ASCII `T` (84) > ` ` (32) ‚Äî so every stored timestamp appears "greater than" the threshold regardless of age.
**Solution:** Wrap the column in `datetime()` to normalise before comparison: `datetime(created_at) < datetime('now', '-900 seconds')`. SQLite's `datetime()` accepts ISO 8601 input (formats 5‚Äì7 in the spec, including T separator and Z/UTC suffix) and normalises to space-separated output, making both sides comparable. This applies to every SQL query that compares a chrono-written timestamp against a SQLite `datetime()` expression ‚Äî audit any `WHERE created_at <` / `>` clause that was written before this was discovered.
**Context:** `src-tauri/src/db.rs` ‚Äî confirmed in `get_stuck_processing_jobs()` (U1c) and the 60-min backstop `UPDATE` in `list_projects()`. Both were silently non-functional; the self-heal never fired until the fix. Detectable only via live test ‚Äî `cargo check` and WDIO fast suite cannot catch this.

## Pipeline Python ‚Äî manifest numeric field falsiness trap

**Problem:** `float(d.get("clip_volume", 1.0) or 1.0)` coerces Python's falsy `0.0` (muted clip) to `1.0` (full volume) because `0.0 or 1.0 == 1.0`. ## React imperative style override silently persists when state hasn't changed

**Problem:** Imperatively setting `v.style.opacity = "0"` on a `<video>` element while React controls `opacity` via the JSX `style` prop. ## CSS animation ‚Äî `@keyframes` with `var()` custom properties blocks compositor acceleration

**Problem:** A CSS `@keyframes` rule that reads a custom property (e.g. `transform: scale(var(--kb-from))`) cannot be promoted to the GPU compositor in Chromium/WebView2. The compositor evaluates keyframe values at rasterisation time; custom properties are resolved on the main thread per-frame, so the animation runs on the main thread (~60Hz JS timer) rather than the compositor (~120Hz rAF). Result: visibly choppy CSS scale animations, even with `will-change: transform` on the element.
**Solution:** Use the Web Animations API (`element.animate([{transform:"scale(1)"},{transform:"scale(1.5)"}], {duration, easing, fill, iterations})`) with literal values ‚Äî WAAPI `transform` animations ARE compositor-accelerated. `anim.currentTime = elapsedMs` provides precise seek without the `animation:none` / `offsetHeight` reflow-restart workaround. SHIPPED U3d in `Arrange.tsx` (`kbAnimRef`); measured 60fps compositor draw, 0‚Äì1 dropped frames.
**Context:** `src/pages/Arrange.tsx` `videoWrapRef` gradual Ken Burns zoom. Applies to any CSS keyframe that reads a custom property in a `transform`/`opacity` value. NOTE: WAAPI fixed the *mechanism* (compositor 60fps confirmed) but did NOT remove the *perceived* judder during zoom ‚Äî see "WebView2 ‚Äî gradual zoom preview judder is 30fps source content, not a rendering defect".
**WAAPI gotcha:** calling `anim.play()` when `currentTime >= duration` resets it to 0 (spec). Guard with `if (playing && elapsedMs < durMs) anim.play(); else anim.pause()` or resuming after the zoom completes snaps the scale back to start. Also write `transition:"none"` (not `undefined`) and order it BEFORE `transform` in the JSX style object, so switching Fixed‚ÜíGradual doesn't briefly run the Fixed-mode `transition: transform 0.3s` against the cleared inline transform.

## FFmpeg ‚Äî music looping

- **`asetpts=PTS-STARTPTS` must come before `atrim` when using `-stream_loop -1`** ‚Äî with infinite loop, FFmpeg assigns continuously rising PTS values across loop boundaries. If `atrim=0:{duration}` runs first it sees inflated timestamps and may cut too early. Always reset timestamps first: `[1:a]asetpts=PTS-STARTPTS,atrim=0:{duration:.4f},...`. Wrong order produces silent music gaps at the expected trim point.
- **Do NOT use `-af apad` in normalise.py** ‚Äî `apad` without an explicit `whole_dur` in a DJI HEVC normalise command causes normalise to hang indefinitely (10+ min, never completes). Root cause: DJI HEVC containers have unreliable or N/A duration headers; `-fps_mode cfr` + `apad` together cause FFmpeg to keep encoding frames past the real EOF waiting for the audio to close, which never happens. The correct fix for the audio/video duration mismatch (64‚Äì120ms per clip) is to apply `apad=whole_dur={video_dur}` per clip inside the filter_complex at the render/concat step ‚Äî where the exact video duration is already known from `get_duration()`.
- **`aresample=async=N` worsens monotonic drift from CFR resampling** ‚Äî `async` mode compensates drift by inserting/dropping audio samples. DJI HEVC decoded to 25fps CFR produces *monotonic* drift (same direction every clip); `async` fights it by inserting samples, creating audible pops/jumps rather than smooth drift. Do not use `aresample=async` for this class of drift. The correct fix requires reading `[sync-check]` log output to identify where drift enters (normalise ‚Üí concat ‚Üí music), then correcting timestamps at that specific step.
- **`volumedetect` mean_volume is unreliable on DJI wind-noise footage** ‚Äî DJI clips with wind noise register mean_volume of -14 to -16 dBFS. Using this as a relative anchor (e.g. `clip_mean + offset_db`) pushes music to -26 to -28 dBFS at "balanced" preset ‚Äî effectively inaudible. Use `loudnorm` integrated LUFS instead, or clamp the measured mean to a floor (e.g. -20 dBFS) before computing the offset.
- **`crop` filter has NO `eval` option in FFmpeg 6.1.1** ‚Äî adding `:eval=frame` to a `crop` filter raises `Error applying option 'eval' to filter 'crop': Option not found`. Only the `scale` filter requires `eval=frame` for time-varying dimensions. The `crop` filter's `x` and `y` expressions re-evaluate every frame natively without any flag ‚Äî pass the time-varying expressions as the x/y arguments directly.

## Workflow-WSL ‚Äî WSL inline commands: use script files not nested quoting

**Problem:** Passing complex bash commands through PowerShell ‚Üí WSL ‚Üí bash requires three layers of quoting. Variables, semicolons, and tee/pipe operators all have characters that get swallowed or mis-escaped by each shell layer, causing silent failures or syntax errors that are very hard to diagnose. This also applies specifically to `$var` references: `wsl -- bash -lc "for f in a.mp4 b.mp4; do echo \"$f\"; done"` run via the PowerShell tool fails with a PowerShell parser error (`Variable reference is not valid`) ‚Äî PowerShell tries to interpret `$f` itself before the string ever reaches `bash -lc`, even though it's meant as a bash loop variable. Confirmed 2026-07-11 probing candidate clip durations for the #104 diagnostic.
**Solution:** Write the command to a `.sh` file in a Windows temp path (e.g. `/mnt/c/Users/<you>/AppData/Local/Temp/`), then invoke `wsl -d Ubuntu-24.04 -u root -- bash /mnt/c/.../script.sh`. The script file avoids all quoting layers entirely ‚Äî including the `$var`-collision case above. Use this pattern for any WSL command beyond a simple one-liner, or any command containing a `$variable` or spanning multiple lines.
**Context:** Any session running multi-step FFmpeg pipelines or Python scripts via WSL from PowerShell. Same underlying principle as the "multi-line diagnostic/repair scripts must be written to a temp .py file" rule in `.claude/rules/pipeline.md` ‚Äî this extends it to any multi-line or `$`-containing bash invocation, not just Python `-c` strings.

## React ‚Äî Library jobsMap staleness when render starts mid-session

**Problem:** `Library.tsx` builds its `jobsMap` (projectId ‚Üí Job) from `list_projects_cmd` on mount; subsequent `pipeline-progress` events are resolved via `findProject(jobId)`. If a new render starts AFTER Library has already done its initial prefetch, the new job's ID isn't in `jobsMap` and all its progress events are silently ignored.

## Workflow ‚Äî Diagnosing pipeline failures: check WSL /tmp state first

**Problem:** A render returns "Pipeline exited with status: exit code: 15" or similar non-zero exit. Reading `pipeline-latest.log` shows a half-complete run that ends mid-stage with no ERROR line.
**Solution:** Exit code 15 = SIGTERM ‚Äî WSL2 killed the process (restart, memory pressure, or a concurrent heavy render that stressed WSL). **Context:** Any render failure with no ERROR line in the log and a numeric signal exit code (15 = SIGTERM, 9 = SIGKILL). The background perftest task running concurrently is a common trigger.

## Tauri ‚Äî check-then-insert command guards need DB-level atomicity, not an in-memory Mutex

**Problem:** `start_job` (`src-tauri/src/lib.rs:1218`) guards against duplicate renders with `get_active_job()` (a `SELECT`) followed much later by `insert_job()` (a separate `INSERT`) ‚Äî no transaction, no unique constraint between them. Two or three calls can each pass the `SELECT` before any commits the `INSERT`, so all proceed. Confirmed live (#89, 2026-07-07): three duplicate renders fired at the same instant, spawned three concurrent 4K WSL pipelines, and all three died together from the resulting WSL memory pressure ‚Äî the exact scenario the guard's own comment says it exists to prevent.
**Solution:** An in-memory `Mutex`/`HashSet` guard (the pattern already used to fix the same bug shape in `generate_proxies_cmd`'s boost guard) only closes the race for in-process callers with no `await` between check and act. It does NOT close a race where the check and the write both hit SQLite with async work in between ‚Äî that needs a DB-level constraint (e.g. a unique index/constraint on `project_id` filtered to `status IN ('pending','processing')`), or the check+insert wrapped in one transaction. Diagnostic signature for this bug class: multiple `jobs` rows with **identical `created_at`**, pipeline logs of **identical byte size**, all dying at the **identical point** ‚Äî that combination means duplicate concurrent submission, not a random crash.
**Context:** Any Tauri command with a "don't double-submit" guard backed by SQLite (`start_job`, `add_clip_cut` ‚Äî see #70 ‚Äî and similar). Before trusting a guard, check whether the check-then-act gap spans an `await`.
**Fixed (#89, 2026-07-08):** `db::init()` now self-heals any pre-existing duplicate active jobs per project on every startup, then creates `CREATE UNIQUE INDEX idx_jobs_active_per_project ON jobs(project_id) WHERE status IN ('pending','processing')`. `start_job` catches the resulting `ConstraintViolation` on a rejected duplicate insert and re-attaches to the winning job's id instead of erroring (falls back to a normal retryable error only in the narrow case where the winner already finished by the time the loser re-queries). Verified live: fired `start_job` twice concurrently via `Promise.allSettled` against the running app ‚Äî both calls returned the same job id, exactly one `jobs` row was created, exactly one WSL pipeline ran.

## Python / tooling

- **`FFMPEG_BIN`/`FFPROBE_BIN` env vars** ‚Äî hardcoding `/usr/local/bin/ffmpeg` blocks local testing without Docker. Read from env vars with Lambda-path as default; also makes CI flexible.
- **Windows console encoding** ‚Äî `print()` on cp1252 chokes on `‚Üí`, `‚úÖ`, `‚ùå`. Use `->`, `[PASS]`, `[FAIL]`.
- **WSL path mangling in Git Bash** ‚Äî Git Bash rewrites paths starting with `/mnt/c/` to Windows paths when passed to `wsl`. Always invoke `wsl` commands from PowerShell; in Git Bash use `//mnt/c/` prefix as a workaround. **Claude Code's Bash tool runs in Git Bash** ‚Äî every WSL call must be wrapped as `powershell.exe -Command "wsl -d Ubuntu-24.04 -u root -- ..."`. Glob wildcard patterns in PowerShell args also get expanded by Git Bash; use `cmd.exe /c tasklist ...` for process lookups. **Prefer the dedicated PowerShell tool** (`PowerShell` tool, not `Bash` + `powershell.exe -Command`) for any PowerShell code containing `$variables`, `|` pipes, or multi-line logic ‚Äî Git Bash mangles both `$` (treats as shell variable) and bare `|` characters inside the `-Command` string, causing silent failures that require retries.
- **PowerShell `Out-File` writes UTF-8 BOM** ‚Äî Python's `json.loads()` raises `JSONDecodeError: Unexpected UTF-8 BOM` on any file written by PowerShell's `Out-File`. Write JSON files destined for Python via WSL (`cat > file` or `python3 -c "... write_text(...)"`) or use `[System.IO.File]::WriteAllText(path, content, (New-Object System.Text.UTF8Encoding $false))`. For test/debug manifests the cleanest pattern is: write via Python in WSL using `pathlib.Path(...).write_text(json.dumps(data))` ‚Äî no escaping issues, no BOM, no Unicode escape errors with Windows paths (use raw strings: `r'C:\clips\...'`).
- **`bash -c 'pattern|pipe'` via PowerShell `-Command` mangles pipes** ‚Äî `powershell.exe -Command "wsl -- bash -c 'grep -E \"foo|bar\" file'"` causes PowerShell to parse `|bar` as a PowerShell pipeline before the string reaches bash. Result: grep sees only `foo` and then tries to run `bar` as a command. Fix: use `python3 -c "import subprocess; r = subprocess.run(['grep', '-E', 'foo|bar', ...])"` inside WSL, or pass the grep pattern as a separate quoted arg without pipes if possible.
- **Concurrent `run.py` processes corrupt a shared `pipeline-latest.log`** ‚Äî opening the same log file with `mode="w"` in two processes: the second process truncates the file but retains the old file offset, producing a sparse file with null-byte gaps and interleaved output from both jobs. Fix: each job writes to a per-job file (`pipeline-{job_id}.log`); `pipeline-latest.log` is a symlink (`Path.unlink(missing_ok=True)` + `Path.symlink_to(f"pipeline-{job_id}.log")`) updated atomically to the most recently started job.
- **`run.py` config dict must include every field that `render.py` reads** ‚Äî `run.py` builds the config dict from manifest settings and passes it to `render.py`. Any field not explicitly added here silently defaults to `""` / `False` / `0` when `render.py` calls `config.get("field", default)`. Pattern: when adding a new `JobConfig` field, check all three of: `buildJobConfig.ts` (reads sessionStorage), `run.py` (reads manifest, builds config dict), and `render.py` (reads config). Missing `run.py` is the most common silent failure ‚Äî the UI sends the value in the manifest but the pipeline never sees it.
- **Pipeline package relative imports** ‚Äî If `pipeline/*.py` modules use `from .module import ...`, the entry script (`run.py`) must add the *parent* directory of `pipeline/` to `sys.path`, then import as `from pipeline.render import run_pipeline`. Inserting `pipeline/` itself breaks all relative imports: Python treats `render` as a top-level module without a parent package.
- **pipeline/ scripts invoked directly must use absolute imports throughout** ‚Äî Any `.py` file inside `pipeline/` that is executed directly as a script (`python3 .../warm_zoom.py`) has `__package__ = None`, so `from .render import x` raises `ImportError: attempted relative import with no known parent package` ‚Äî even inside a helper function that runs AFTER `main()` has inserted the parent dir into `sys.path`. The `sys.path` manipulation fixes name resolution but not `__package__`. Fix: use `from pipeline.render import x` (absolute) everywhere in the file. This applies to `warm_zoom.py` and any future entry-point scripts added to the package.
- **`subprocess.run(cmd, check=True)` with list args** handles paths with spaces correctly; no `shell=True` needed.
- **WSL2 `/tmp/<job_id>/` accumulates 1-3 GB per render** ‚Äî `render.py` creates `TMP_BASE / job_id` for normalised intermediates; they persist until WSL2 shuts down if not explicitly deleted. Fix: `shutil.rmtree(f"/tmp/{job_id}", ignore_errors=True)` in `run.py` immediately after `shutil.copy2` succeeds. Wrapup cleans crash orphans via `wsl -- sh -c 'rm -rf /tmp/*/'`. Proxies in `%APPDATA%\rushcut\proxies\` are a persistent cache ‚Äî do NOT clean those.

## Dual-buffer port to a new mode ‚Äî `isValid` guard + `pendingStartMsRef` pattern

**Problem:** Porting `gateFrameRevealThen` to a second mode (e.g. clip mode alongside film mode) requires decoupling the gate's staleness guard from a hard-coded mode ref. **Solution:** (1) Add an `isValid: () => boolean` param to `gateFrameRevealThen`; each caller passes its own staleness closure (`() => !filmModeRef.current && clipSlotGenRef.current[slot] === thisGen`). ## Dual-buffer onError double-fire guard needs a gen-bump at EVERY src-set site, not just the ones with a reveal gate

**Problem:** Porting a proxy-fallback `onError` handler (e.g. Sound.tsx's #51 pattern) to a second dual-buffer engine and guarding it against React's double-fire with `slotGenRef` only works if `slotGenRef` is bumped at every function that assigns `v.src` for that slot ‚Äî including "dumb" preload functions with no `gateFrameRevealThen`/rVFC gate of their own. It's easy to assume the gen counter only needs bumping where a reveal gate reads it (`loadIntoSlot`, `crossSeekToClip`) and skip the plain preload function, since nothing there currently consumes `thisGen`. Treat the gen counter as "identity of the current load," not just "staleness guard for pending async callbacks" ‚Äî the two uses have different completeness requirements.
**Context:** `src/pages/Trimmer.tsx` Film-mode dual-buffer, #90. Store the double-fire "have I already reacted to this failure" marker as a `dataset` attribute on the `<video>` element itself (e.g. `dataset.advancedGen = String(gen)`), not a second ref ‚Äî avoids a ref that duplicates `slotGenRef`'s data and stays colocated with the existing `dataset.clipId`/`dataset.usingSource` idiom.

## Sound.tsx ‚Äî intentionally naive timeline; filmstrip seek requires a conversion layer

**Problem:** Sound's master preview plays proxies sequentially with no xfade blending, so its internal timeline (`totalMs`, `clipStartMsRef`, `seekToFilmMs`) accumulates raw clip durations without xfade subtraction. The StickyFilmStrip uses telescoped (card-inclusive, xfade-subtracted) time. Wiring strip click directly to `seekToFilmMs` gives seeks that are off by `xfadeMs √ó (n‚àí1)` for multi-clip projects with transitions, plus a 3s card lead when an open card is enabled.
**Solution:** Introduce a call-site conversion function (`handleStripSeek`) that maps strip telescoped-ms ‚Üí clip index + within-clip offset via `filmTimeAtClipStart` iteration, then calls `seekToFilmMs` with the naive accumulated position. Card regions (open/close) snap to the adjacent clip boundary. The scrub bar is internally consistent (both input and `seekToFilmMs` speak naive) so it does NOT need the conversion ‚Äî leave it alone to avoid double-conversion.
**Context:** `src/pages/Sound.tsx` `handleStripSeek` (added #74 follow-up). The scrub bar and filmstrip playhead will show slightly different positions for the same playback moment when transitions are active ‚Äî this is a known acceptable caveat until Sound gets a full telescoped-time rewrite. Also: `playheadMs` state for the strip needle is throttled to ~10fps (100ms) inside `handleFilmTimeUpdate` to avoid the full 4-66Hz re-render load flagged in the existing imperative-DOM comment.

## WebView2 ‚Äî GPU compositor presents frame 0 first; rVFC `metadata.mediaTime` is the only reliable gate

**Problem:** After a video `src` change + `load()` + seek, WebView2's GPU compositor presents frame 0 as the first composited frame before the seek-target keyframe is decoded. Neither `seeked` nor `v.play().then(...)` (Option F) nor a bare rVFC callback (Option C) reliably guards against this ‚Äî they all fire before the seek-target frame is compositor-committed. Option F specifically: `.play().then()` resolves when audio starts, not when video frame is presented; `.pause()` then freezes at frame 0; reveal shows frame 0.
**Solution (FIXED ‚Äî Option H):** Use `requestVideoFrameCallback` with `metadata.mediaTime` inspection: only reveal when `metadata.mediaTime >= targetSec - 0.05`. Frame-0 leaks have `mediaTime ‚âà 0` and are silently skipped; re-register rVFC for the next frame. Cap at `MAX_WAITS=30` (~500ms) to prevent infinite loops. For cross-clip seeks, load the new clip into the **opposite** (non-active) slot ‚Äî the active slot keeps showing the outgoing frame while the new clip loads; swap visibility only when `metadata.mediaTime` passes the gate. This mirrors the proven `advanceFilmClip` pattern. See `gateFrameRevealThen()` + `crossSeekToClip()` in `src/pages/Trimmer.tsx`.
**Context:** `src/pages/Trimmer.tsx` film mode. Affects cross-clip seeks triggered by `seekFilmTo`. Same-clip seeks and initial film load (`loadIntoSlot`) use the same `gateFrameRevealThen` helper. `rVFC` confirmed supported in WebView2 (Edg/148): `typeof v.requestVideoFrameCallback === 'function'` ‚Üí true.

## WebView2 ‚Äî a "natural advance" playback path is easy to miss when porting the rVFC gate, and the gate itself must survive pause

**Problem:** When `gateFrameRevealThen` is ported into a screen with an "advance to next clip during continuous playback" path (as distinct from manual/seek-triggered clip switches), it's easy to gate the manual-seek paths correctly (`loadIntoSlot`, `crossSeekToClip`) but leave the natural end-of-clip advance (`onEnded`/`onTimeUpdate >= outSec`) ungated ‚Äî it looks like a simpler "just swap and play" case, but WebView2's frame-0-first behavior applies there identically. Investigated on #91 (Sound.tsx `advanceFilmClipRough`): the progress bar/active-slot flip was happening synchronously, before the preloaded next clip's frame was confirmed rendered, producing "seeker races ahead, then freezes" ‚Äî the exact same failure class this file already documents above, just in a spot that wasn't audited when the gate was first built. Additionally, once gated, `gateFrameRevealThen`'s `check()` must not do a bare `return` on `!filmPlayingRef.current` ‚Äî `requestVideoFrameCallback` only fires on a newly presented frame, so an early-return-and-forget on pause permanently kills the poll chain; resuming playback plays the video again but nothing is left listening, stranding the pending reveal forever.
**Solution:** (1) Audit every place a slot's active-clip index/visibility/progress state is written, not just the ones with an obvious "seek" name ‚Äî search for every synchronous `setSlotVisible`/`activeFilmSlotRef =`/progress-bar write and confirm each is inside a `gateFrameRevealThen` (or equivalent) `onReady`, never before it. (2) In the gate's poll loop, only abandon on a genuine invalidation (gen/slot mismatch); on `!filmPlayingRef.current` alone, re-arm (`rVFC?.call(v, check); return;`) instead of dropping the chain ‚Äî this lets `waits` stay unspent during an arbitrarily long pause and lets resume revive the same pending gate. (3) Track a `pendingGateSlotRef` (which slot is mid-gate, separate from `activeFilmSlotRef`) so `pauseFilmPlayback`/`resumeFilmPlayback` can pause/resume BOTH the active slot and any slot mid-transition ‚Äî pausing only the active slot leaves an invisible mid-gate video still playing (and audible) through the pause.
**Context:** `src/pages/Sound.tsx` dual-buffer engine (#91). Same audit is worth doing on any other screen with a ported natural-advance path off this pattern (Trimmer.tsx film mode uses the same origin ‚Äî not yet re-audited for this specific gap, flagged as a non-blocking follow-up).

## WebView2 ‚Äî Web Audio API is CORS-blocked for `asset.localhost` files

**Problem:** `AudioContext.createMediaElementSource(videoEl)` and `videoEl.captureStream()` both throw a SecurityError when the video element's `src` is an `asset.localhost` URL. The asset protocol returns no `Access-Control-Allow-Origin` header; `localhost:1420` (Vite) and `asset.localhost` are treated as different origins by WebView2/Chromium, and both Web Audio APIs enforce CORS for cross-origin media. Setting `video.crossOrigin = "anonymous"` makes it worse (enforces CORS where before the request was just unauthenticated). `video.volume` is capped at 1.0 by the HTML5 spec ‚Äî there is no pure-JS way to get > 100% gain for `asset.localhost` media.
**Solution:** For gain < 1.0, use `video.volume = Math.min(1.0, gain)` directly. For >100% gain preview, the only correct fix is a custom Tauri URI scheme handler in Rust that serves files with `Access-Control-Allow-Origin: *` ‚Äî enabling `createMediaElementSource` to work. Until that Rust work is done, cap preview at 100% and apply real gain via FFmpeg `volume` filter on render. Do NOT ship 150%/200% chips without the Rust protocol fix ‚Äî users will expect audible difference.
**Context:** Arrange Sound tab per-clip volume preview. Any future feature requiring Web Audio processing of local video files via `asset.localhost`.

## SQLite ‚Äî schema constraints

- **No `ON DELETE CASCADE` without `PRAGMA foreign_keys = ON`** ‚Äî rusqlite opens each connection without enabling FK enforcement. Even if FKs are declared in the schema, `DELETE FROM projects WHERE id = ?` won't cascade. Always delete in child-first order manually: `clips` ‚Üí `jobs` ‚Üí `projects`. Confirm by reading the schema DDL before writing any delete logic.
- **No `ADD COLUMN IF NOT EXISTS` in SQLite** ‚Äî running `ALTER TABLE t ADD COLUMN c TEXT` a second time raises "duplicate column name: c" and crashes. Use a migration guard: `SELECT COUNT(*) FROM pragma_table_info('jobs') WHERE name='column_name'` ‚Üí only run ALTER if count is 0. Do this in Rust `db.rs` `init()` immediately after the initial `execute_batch` schema creation.

## UI-to-pipeline value mapping

- **Slider sends 0‚Äì100, pipeline expects 0.0‚Äì1.0** ‚Äî React range inputs return integers (0‚Äì100); pipeline functions expect float factors (0.0‚Äì1.0). Scale in `run.py` at the settings boundary: `settings.get("music_volume", 40) / 100.0`. If the default in `run.py` is set as a raw float (e.g. `0.4`), it will be silently wrong the first time the UI sends an integer `40`. Always align the default unit with the source: use `40` (integer, UI scale) as the default in `run.py` and always divide. NOTE: as of Batch 12b `music_volume` is now a string union `"subtle"|"balanced"|"prominent"` ‚Äî the 0‚Äì100 integer pattern no longer applies to this field specifically, but the principle remains valid for any new numeric config field.
- **Per-clip pre-processing cost is multiplicative** ‚Äî adding an FFmpeg pass per clip before the encode phase (e.g. scene detection scoring) multiplies processing time by N clips. On 30 DJI clips at ~20s each, that is 10+ extra minutes before a single frame is rendered. Profile realistically on target footage size before shipping any per-clip analysis step; prefer single-pass designs where the analysis output is reused for multiple purposes (trim + score from the same FFmpeg call).

## Native Win32 splash + startup performance

- **`windows` crate 0.58: `HWND` wraps `*mut core::ffi::c_void`, not `isize`** ‚Äî code that constructs `HWND(val as isize)` fails to compile. Store the raw pointer as `usize` in an `AtomicUsize`, then reconstruct as `HWND(val as *mut core::ffi::c_void)`. The `isize` constructor existed in 0.52 and earlier; 0.58 changed the type.
- **`UpdateWindow` is absent from `windows` 0.58 `Win32_UI_WindowsAndMessaging`** ‚Äî removed without replacement. Use `ShowWindow` to make the window visible; the first `WM_TIMER` fires `InvalidateRect` which triggers an immediate `WM_PAINT`. No `UpdateWindow` needed.
- **`spawn_blocking` required for `std::process::Command` inside `async` context** ‚Äî `tauri::async_runtime::spawn(async move { Command::new("wsl")... })` blocks an async thread-pool thread for the duration of the process (confirmed 6-8s for `wsl --status`). Wrap in `tokio::task::spawn_blocking(|| { Command::... })`.await` to run on a dedicated blocking thread without stalling the async pool.
- **`Manager` trait must be imported to call `app.get_webview_window()`** ‚Äî `app.get_webview_window("main")` fails to compile with "method not found" unless `use tauri::Manager;` is in scope. It is not re-exported via `tauri::*`. Add it explicitly alongside `Emitter`.
- **`visible: false` in tauri.conf.json breaks E2E** ‚Äî WebDriver (msedgedriver) cannot attach to a window that has never been shown: "Failed to create a session". Fix: set `visible: false` in config so the window starts hidden (covered by the native splash), then call `win.show()` in `setup()` immediately after db::init. The native splash (`WS_EX_TOPMOST`) covers the briefly-shown window from the user's perspective while E2E always has an accessible DOM.
- **`dataDirectory` in tauri.conf.json does NOT redirect WebView2 user data** ‚Äî adding `"dataDirectory": "webview-data"` to the window config has no effect; the directory is never created. WebView2 already persists its user data (compiled shaders, code cache) to `%LOCALAPPDATA%\<identifier>\EBWebView` by default, which provides cache persistence across launches. Do not add `dataDirectory` to tauri.conf.json to "fix" cold starts ‚Äî it's a no-op.
- **`app-ready` fires before React's `listen()` when WSL check is async** ‚Äî with sync WSL check, `app-ready` fires ~6-8s after binary starts (after React has loaded Vite). With async check, `app-ready` fires ~50ms after binary starts ‚Äî before WebView2 has even loaded `index.html` (~4-6s later). The `listen("app-ready", ...)` call never sees the event. Fix: add a short fallback timeout (500ms is sufficient ‚Äî the window shows the `#rc-splash` dark overlay until then). Alternative: use `confirm_app_loaded` (React mount invoke) as the close signal instead of relying on `app-ready` timing.
- **`pnpm dev` is the wrong test vehicle for native splash** ‚Äî `pnpm dev` runs `tauri dev` which compiles Rust (~15-25s) before launching any binary. No native splash can appear during compilation. Correct test workflow: `pnpm dev:vite` once (stays running), then double-click `src-tauri\target\debug\rushcut.exe` directly. Rust only needs recompiling when `.rs` files change ‚Äî keep Vite running and re-launch the binary.

## Tauri / Windows dev

- **Rustup PATH only applies to new terminals** ‚Äî after `winget install Rustlang.Rustup`, `cargo` is available in newly opened terminals only. Existing CMD/PowerShell windows don't inherit the updated PATH. Fix for the current session: `$env:PATH += ";$env:USERPROFILE\.cargo\bin"`. Fix permanently: `[System.Environment]::SetEnvironmentVariable("PATH", $env:PATH + ";$env:USERPROFILE\.cargo\bin", "Machine")` then reopen terminal.
- **`pnpm dev` = `tauri dev`** ‚Äî this starts Vite (port 1420) then compiles Rust and opens the Tauri window. `pnpm dev:vite` alone starts only the React frontend; all `invoke()` calls throw "Cannot read properties of undefined (reading 'invoke')". The Preview MCP can connect to `:1420` for UI-layer testing, but no Tauri backend commands will work ‚Äî use it only for layout/navigation/React state checks. Startup shows `[wsl_check] ok` in the terminal if WSL2 is available. A blank black window on first launch is expected until React routes are wired.
- **Tauri 2.x plugin permissions are runtime-only** ‚Äî missing capability entries throw `not allowed` at runtime, not at compile time. Declare all needed permissions in `src-tauri/capabilities/default.json` (e.g. `"dialog:allow-open"` for the folder picker). `cargo check` passes silently even when permissions are missing.
- **Tauri plugin config: `null` not `{}`** ‚Äî plugins with no options must be `"plugin-name": null` in `tauri.conf.json`. Using `{}` causes a deserialization panic at startup: `invalid type: map, expected unit`.
- **All Tauri commands must be in a single `generate_handler![]`** ‚Äî only the last `invoke_handler()` call is registered. If you add a second `invoke_handler`, the first is silently dropped. Collect all commands in one list.
- **Tauri 2.x `invoke` command names must match exactly** ‚Äî the JS `invoke("get_job_cmd")` string must match the Rust `#[tauri::command] fn get_job_cmd` name. Mismatches give a runtime "command not found" error, not a build error.
- **`convertFileSrc` is the only correct asset URL API on Windows** ‚Äî constructing `asset://localhost/C:/clips/foo.mp4` manually produces URLs that Tauri 2.x rejects silently; the `<video>` element renders nothing. Import `convertFileSrc` from `@tauri-apps/api/core`; it outputs `https://asset.localhost/C:/clips/foo.mp4`. Always use it for any local file served to the WebView.
- **`assetProtocol.scope` must include all directories the WebView will read from** ‚Äî a missing scope entry causes a silent `403 Forbidden` on the video element with no warning at build or compile time. The scope in `tauri.conf.json` must cover: processed output (`C:\\clips\\processed\\**`), source clips (any drive the user picks from ‚Äî use `C:\\**`, `D:\\**`, `E:\\**` or similar), and `$APPDATA\\rushcut\\**` for proxy files. Thumbnails served as base64 data URIs bypass this entirely ‚Äî only file-path assets are affected.
- **`run.py` must explicitly forward all `JobConfig` fields** ‚Äî if a new field is added to the TypeScript `JobConfig` type but not added to the settings dict in `run.py`, the pipeline silently uses its own default (which may be wrong). Convention: every `JobConfig` field maps to one `settings.get(key, safe_default)` line in `run.py`.
- **`tauri::State<'_>` is not `Send` ‚Äî cannot be moved into `spawn()`** ‚Äî `tauri::async_runtime::spawn(async move { ... })` requires all captured values to implement `Send`. `tauri::State<'_, T>` does not. Fix: before the spawn, extract the inner `Arc` via `Arc::clone(&*state)` and move the clone into the closure instead. If `T` is already an `Arc<Mutex<...>>`, this is a single `Arc::clone` call.
- **`window.confirm()` is silently swallowed by Tauri WebView2** ‚Äî `window.confirm()` returns `true` immediately without showing any dialog. Never use it for destructive-action confirmation in a Tauri app. Replace with an in-app React state `pendingConfirm` that renders an inline confirmation panel inside the component tree.

- **DJI HEVC + Chrome thumbnail generation:** Chrome reads the embedded MJPEG stream (stream 1) from DJI containers when generating thumbnails via `<video>` + canvas, so `generateThumbnail()` often succeeds locally. ## Pipeline ‚Äî h264_amf GPU encoder (AMD, Windows-native ffmpeg, Batch Q)

**AMF final path uses VBR, not CQP (U4e) ‚Äî CQP's uncapped peaks crash WebView2.** AMF `-rc cqp` produces a *constant quality* stream with an uncapped bitrate: on dense 4K DJI footage the instantaneous bitrate can spike to 60+ Mbps, and the resulting file crashes WebView2's `<video>` decoder on the Render screen (decode pressure, not a code bug). The U4e fix: the final encode uses peak-constrained VBR ‚Äî `-rc vbr_peak -b:v 20M -maxrate 24M -bufsize 24M -quality quality`. `-maxrate` MUST be set explicitly or AMF picks its own (high) ceiling. Draft mode keeps CQP (`-rc cqp -qp_i 30 -quality speed`) since drafts aren't played in-app. Both bitrates live in the single `FINAL_BITRATE`/`AMF_MAXRATE` constants in `encoder.py`. (Historical: pre-U4e the shipped default was `-rc cqp -qp_i 23`, ~6% larger than libx264 crf22; that's gone.)

**AMF auto-enables for 4K, silently (U4e):** `video_encoder_args()` gates AMF on `use_amf or RUSHCUT_USE_AMF or output_resolution == "4k"` ‚Äî there is NO UI toggle (the old "Fast render" toggle was removed; it distorted output and needed user education). VBR's consistent high bitrate compensates for AMD's missing B-frames. `RUSHCUT_FORCE_LIBX264=1` still wins (dev escape hatch). 1080p stays libx264 unless `RUSHCUT_USE_AMF` is set.

**B-frames not supported on AMD AMF (hardware-level, not a config issue):** h264_amf with `-rc cqp` produces `has_b_frames=0` regardless of any `-bf N` flag ‚Äî the AMD driver silently ignores it. This was confirmed at both CQP and VBR rate control modes on this hardware (AMD GPU, WinGet ffmpeg 8.0.1). libx264 `-preset fast` produces `has_b_frames=2`, which makes pans look noticeably smoother. There is no AMF flag combination that restores B-frames on this machine. Workaround if motion quality matters: use libx264 via `RUSHCUT_FORCE_LIBX264=1` or add a quality-mode UI selector. NVENC (Nvidia) does support B-frames and would not have this limitation.

**Encode speed:** 1.7√ó faster than libx264 on AMD GPU (30s 4K clip: 27.8s AMF vs 46.6s libx264). AMD AMF is structurally slower than Nvidia NVENC ‚Äî do not expect 5‚Äì10√ó speedup on AMD hardware. On a real 8-clip 4K render, Step 5 drops by roughly 70s.

**Path translation (WSL Python ‚Üí Windows ffmpeg.exe):** Windows ffmpeg.exe cannot be invoked via a `C:\...` string from WSL `subprocess.run` ‚Äî it needs the `/mnt/c/...` WSL-accessible form. Conversely, file path *arguments* to Windows ffmpeg must be Windows-form (`C:\...` or `\\wsl.localhost\Ubuntu-24.04\...`). Two separate helpers handle this: `_win_to_wsl()` (binary path) and `to_win_path()` (file args) in `pipeline/encoder.py`.

**WSL tmpfs is invisible to Windows via wsl.localhost if WSL was restarted:** `/tmp` files written before a `wsl --shutdown` or OOM restart are gone; and immediately after restart, the `\\wsl.localhost\Ubuntu-24.04\tmp\` share may appear empty. The pipeline is safe because WSL stays alive for the entire render job ‚Äî but benchmark scripts that regenerate test sources after a WSL restart must write to an NTFS path (`/mnt/c/...`) to be accessible from Windows ffmpeg.

**RUSHCUT_FORCE_LIBX264=1 env var bypasses AMF detection** ‚Äî use this to test the libx264 fallback path without touching hardware.

**`amf_fallback` pattern for silent encoder fallback detection:** When `use_amf=True` and the initial AMF encode raises `RuntimeError`, `render.py` retries with libx264 via `_run_with_amf_fallback()`. A list-closure `amf_fallback_flag = [False]` tracks whether fallback occurred (list used for mutability inside a nested function). At pipeline end, `amf_fallback=0/1` is appended to the ANALYSIS stdout line. `src-tauri/src/lib.rs` captures `last_analysis` per job and emits it in the `pipeline-done` event's `"analysis"` JSON field. React reads `/(^|,)amf_fallback=1(,|$)/.test(analysis)` to surface a toast. This pattern is reusable for any silent fallback that must be surfaced to the user after the fact.

**`pipeline-done` event carries `analysis` field (Batch R Part C):** The `pipeline-done` Tauri event payload now includes `{ jobId, stage, progress, message, outputPath, analysis }` where `analysis` is the full ANALYSIS stdout string (or null if not emitted). React listeners must be typed `PipelineProgressEvent & { analysis?: string | null }` to access it. The field allows post-render decision surfacing (encoder used, fallback state, proxy stats) without requiring a DB query.

**`generate_proxy_file_low_priority` must use the same AMF/libx264 arg branching as the normal-priority variant:** `generate_proxy_file_normal_priority` was fixed in Batch S to branch on `is_amf = encoder == "h264_amf"` (AMF needs `-rc cqp -qp_i 30 -qp_p 30 -quality speed`; libx264 needs `-preset ultrafast -crf 23`). The low-priority function was NOT updated at the same time. It kept the libx264-style args while still calling `detect_best_encoder()` ‚Äî so on AMF hardware, every low-priority encode failed silently with `elapsed=0.1s`. The fix (Batch S4) mirrors the full arg branching from the normal-priority function. **Rule:** whenever touching either proxy encode function, always check that BOTH functions stay in sync on encoder args. **Critical addition (#64):** the AMF args block in BOTH functions must include `-colorspace bt709 -color_primaries bt709 -color_trc bt709` ‚Äî without these, proxies carry `prim:reserved` which causes downstream AMF segment encodes to fail with swscaler -129 (see AMF yuv444p negotiation entry above).

**AMF yuv444p negotiation ‚Äî definitive fix is `setparams` in the per-input canvas chain (#64):** Three levers; levers 1‚Äì2 pin pixel format, lever 3 normalizes color metadata in-graph.

- *Lever 1 (terminal format node ‚Äî necessary but not sufficient):* `[vpix]format=yuv420p[vout]` at the filtergraph tail locks the buffersink but does NOT prevent AMF requesting yuv444p when upstream inputs carry mismatched color primaries. Segments with consistent color metadata succeed; segments mixing `prim:reserved` (AMF proxy) and `prim:unknown` (normalise) still trigger `swscaler -129`.
- *Lever 2 (per-input canvas format node ‚Äî necessary but not sufficient alone):* `,format=yuv420p` on each `[{i}:v]{canvas}[sv{i}]` in `build_batch_video_fc` and `build_filter_complex` (transitions.py) locks buffersrc to yuv420p. Eliminates format-driven yuv444p but fails because `format=` only sets pixel format ‚Äî it does NOT strip color metadata tags from frames. Swscaler still sees `prim:reserved` on the decoded input and tries to negotiate yuv444p.
- *Lever 3 (setparams in canvas chain ‚Äî RESOLVES IT):* Add `,setparams=color_primaries=bt709:color_trc=bt709:colorspace=bt709` after `format=yuv420p` in both canvas strings (transitions.py). `setparams` overrides frame-level color metadata without changing pixel values ‚Äî sets `prim:bt709` on every `[sv{i}]` output regardless of what the source proxy carried. Downstream AMF sees consistent bt709 and accepts yuv420p. **h264_amf ignores `-color_primaries bt709` on the ENCODER command line** (it always writes `prim:reserved` in the bitstream regardless of the flag), so fixing it at the source (proxy gen) does not help ‚Äî fix must be in-graph.
- **Source identification rule:** `prim:reserved` always originates from an AMF-encoded file (h264_amf cannot be told to write a different primaries value). Trace failing segment's inputs via `trim_N.mp4` / `render_in_N.mp4` ‚Üí if the proxy or zoom cache was AMF-encoded, `prim:reserved` will be present. The canvas `setparams` node is the authoritative fix; do not attempt to fix it at the encode site.
- **`segmented render complete (no fallback)` is unreliable ‚Äî see tracking bug note below.**

**`[U1g] segmented render complete (no fallback)` message is unreliable ‚Äî tracking bug:** This success marker checks a variable that is NOT set by `_run_with_amf_fallback()`. When a U1g segment batch falls back to libx264, `amf_fallback_flag[0]` (outer `run_pipeline` scope) is correctly set to True, but the U1g completion log reads "(no fallback)" anyway. Do not rely on this message to confirm AMF success ‚Äî grep `AMF_FALLBACK=1` from the fallback warning instead, or check that no `h264 (Constrained Baseline)` / libx264 encoder appears in the segment stream info lines.

**AMF fallback log line is now generic ‚Äî do not assume "#64" from the log alone:** `_run_with_amf_fallback()` in `render.py` catches ANY `RuntimeError` from an AMF encode and retries on libx264. Historically the warning read `*** AMF FALLBACK (#64) ***`, which only describes the yuv420p‚Üíyuv444p swscaler negotiation failure documented above. A second, unrelated failure mode (permission-denied writes, see #86 entry below) can trigger the same warning. The log line no longer says `(#64)` ‚Äî it's just `*** AMF FALLBACK ***`. **Always read the actual ffmpeg stderr above the warning** to tell which failure mode occurred before assuming it's #64.

**#86 RESOLVED (2026-07-06): AMF "Permission denied" on U1g segment output was a `USERPROFILE`-not-inherited-into-WSL bug, not a UNC-path design flaw.** Root cause: `_resolve_render_work_dir()` was supposed to return an NTFS `/mnt/c` path via `os.environ["USERPROFILE"]`, but `USERPROFILE` is **not inherited into the WSL environment** from Rust's bare `wsl -d Ubuntu-24.04 -u root -- python3 run.py` spawn (no `WSLENV`/env forwarding). The lookup silently missed and fell through to a `/tmp/rushcut-render/<job_id>` path, which `to_win_path()` translates to `\\wsl.localhost\Ubuntu-24.04\tmp\...` ‚Äî a 9p UNC share that Windows-native `ffmpeg.exe` (AMF) cannot open for writing (reads over the same UNC path work fine; only file *creation* fails). This affected ALL AMF output targets, not just U1g segments: single-clip render, the monolithic multi-clip path, and the U1g open/close post-pass were equally broken ‚Äî the reporter only noticed U1g because their repro project took the segmented path. Confirmed via job log + `ffprobe`: every 4K AMF render since the 2026-07-04 WSL reinstall was silently falling back to libx264, making the paused #49/#78 bitrate work unverifiable against a real baseline.
**Fix:** `run.py` derives `ntfs_tmp_base = str(manifest_path.parent)` ‚Äî the manifest is always written by Rust to `%TEMP%\rushcut\<job_id>.json`, so its parent is guaranteed to exist and requires no env lookup ‚Äî and passes it through `job["config"]["ntfs_tmp_base"]`. `_resolve_render_work_dir(job_id, base=None)` now takes this as an explicit, preferred `base` param (the old `USERPROFILE` lookup remains only as a legacy/test-env fallback when `base` is not supplied). All AMF write targets were repointed at the resulting `render_work` NTFS dir; tmpfs `tmp` still holds normalise/pre-trim/audio-treatment intermediates (AMF only *reads* those over UNC, which works fine ‚Äî see `_apply_audio_treatment` writing `with_music.mp4`/`loudnorm.mp4` to `tmp`, never `render_work`). Cleanup in `run.py` removes `render_work` via the *same* `_resolve_render_work_dir` call (not a hand-reconstructed path) alongside the existing `/tmp/<job_id>` rmtree, so NTFS scratch doesn't accumulate per render.
**Verification (real render, job `bf04b1bd-...`, 9-clip 4K xfade project):** log showed `render_work(NTFS/AMF-out)=/mnt/c/...` (not UNC); all 3 U1g segments encoded via `h264_amf` with zero `AMF FALLBACK` lines; `drift=0 frame(s)`; `ffprobe` on the final output confirmed `TAG:encoder=Lavc62.11.100 h264_amf` at ~20 Mbps (matching the VBR `-b:v 20M -maxrate 24M` config); render cache showed `[cache] wrote` (fresh encode, not a stale pre-fix cache hit); NTFS scratch dir confirmed deleted post-render. Independently confirmed by `rushcut-qa-reviewer`.
**Same `USERPROFILE` gap exists in `pipeline/render_cache.py` `cache_dir()`** (the #19 render-cache directory) ‚Äî **deliberately not fixed as part of #86** to keep that fix atomic; tracked as a follow-up note on #81/#82 (Rust manifest wiring for the render-cache dir). **Empirically confirmed manifesting during #110 (2026-07-11):** the real cache dir landed at `/tmp/rushcut-render-cache` (WSL tmpfs), not `%APPDATA%\rushcut\render-cache\` (NTFS) ‚Äî so despite the module's own docstring describing it as a "persistent render-cache dir on NTFS," it currently lives on ephemeral tmpfs and is wiped on every WSL restart, silently defeating the whole point of a cross-session cache.

## Proxy gate ‚Äî `ready === 0` bypass condition is stale after Batch N

**Problem:** `submitJob` in `Render.tsx` contained `if (status.ready >= status.total || status.ready === 0)` to skip the proxy-wait gate. The `ready === 0` branch was correct before Batch N (no background gen existed; 0 ready = proxies not applicable). After Batch N, bg gen starts on Trimmer exit ‚Äî cold 4K projects have 0 proxies ready *while encoding is in flight*. The bypass fires anyway, skips the gate, and launches the full 169s HEVC normalise immediately.
**Solution:** Change to `if (status.ready >= status.total || (status.ready === 0 && !has4K))`. Non-4K cold renders still bypass (normalise is fast at 1080p). 4K cold renders fall through to the gate and get boosted bg gen. Measured result: t_normalise_s dropped from 169s ‚Üí 1s.
**Context:** `src/pages/Render.tsx` `submitJob` function. Any future change to the proxy gate condition must handle the cold-start case explicitly ‚Äî `ready === 0` is NOT a safe bypass when 4K clips exist.

## Proxy pipeline ‚Äî `HashSet` concurrency guard silently drops priority-upgrade calls

**Problem:** `generate_proxies_cmd` uses `Arc<Mutex<HashSet<project_id>>>` to prevent duplicate bg gen batches. When the render gate calls `generate_proxies_cmd(lowPriority: false)` to boost an in-flight low-priority batch, the guard detects the project is already running and returns `Ok(())` silently ‚Äî the boost never reaches FFmpeg. Clips continue encoding at `-threads 1 BELOW_NORMAL_PRIORITY_CLASS` (60‚Äì110s each) instead of `-threads 0 normal` (~10‚Äì20s each).
**Solution:** Allow the normal-priority boost to bypass the guard: `if set.contains(&project_id) && low_priority { return Ok(()); }`. Both batches run concurrently; a DB atomic claim (`UPDATE clips SET proxy_status='encoding' WHERE proxy_status NOT IN ('encoding','done')` ‚Üí `rows_affected == 1`) prevents two FFmpeg processes from writing the same proxy file. First batch to claim a clip owns the encode; second batch skips claimed clips via `continue`. On encode failure, reset status to `'queued'` (not `'encoding'`) so the other batch retries.
**Context:** `src-tauri/src/lib.rs` `generate_proxies_cmd` + `run_bg_proxy_batch`. `src-tauri/src/db.rs` `claim_clip_for_encoding` + `reset_stale_encoding_claims`. The `HashSet` pattern protects against duplicate runs but must NOT block priority escalation ‚Äî use the DB claim for per-clip mutual exclusion instead.

## Proxy pipeline ‚Äî background gen and render gate must both be resolution-aware

**Problem:** Background proxy gen produced 1080p proxies. `render.py` accepted any `height >= 1080` proxy for all render modes. A 4K render that reused a 1080p proxy upscaled from 1080p ‚Üí 2160p, degrading output quality ‚Äî AND still got ~2s normalise (wrong skip for wrong reason). Separately, `get_clips_needing_bg_proxy` filtered `proxy_status != 'done'` which prevented 1080p proxies from being upgraded to 2160p.
**Solution:** (1) Background gen always encodes at `scale=-2:2160` ‚Äî a 2160p proxy qualifies for both 1080p (`h >= 1080` ‚úì) and 4K (`h >= 2160` ‚úì) renders. (2) `render.py` gate: `required_proxy_h = 2160 if output_resolution == "4k" else 1080` ‚Äî rejects 1080p proxies for 4K renders and falls through to normalise from source. (3) `get_clips_needing_bg_proxy` returns ALL `include=1` clips; `run_bg_proxy_batch` calls `proxy_height_native()` to decide skip/upgrade/encode.
**Context:** Any change touching `generate_proxy_file_low_priority()` in `lib.rs`, `_proxy_height()` in `render.py`, or `get_clips_needing_bg_proxy()` in `db.rs`. Hardcoded height value `1080` anywhere in the proxy/normalise path is a latent 4K bug ‚Äî always parameterise by `output_resolution`.

## Proxy pipeline

- **Proxy encode timeout must be 600s, not 120s** ‚Äî `subprocess.run(..., timeout=120)` is too short for 4K HEVC source clips longer than ~60s at software decode speeds. A 90-second 4K DJI clip with software HEVC decode takes 90‚Äì180s to encode to 480p H.264 ultrafast, depending on CPU load. Set `timeout=600` (10 min ceiling) in `generate_proxy()`. The timeout killing FFmpeg mid-write produces a corrupt file ‚Äî see the moov-atom pattern below.
- **FFmpeg killed mid-write leaves a corrupt MP4 with missing moov atom** ‚Äî if the proxy encode process is killed (timeout, SIGTERM, crash), FFmpeg writes partial data but never finalises the container header. The resulting file passes `Path.exists()` but `ffprobe -v quiet -show_format` returns exit code 1. `proxy.py` must validate existing files with `ffprobe` before skipping re-encode ‚Äî a size check is insufficient (partial files can be several MB). Pattern: `if Path(proxy_wsl).exists() and not is_valid_proxy(proxy_wsl): Path(proxy_wsl).unlink(); generate_proxy(...)`.
- **4K HEVC software decode is ~1x realtime ‚Äî a 5-min source clip takes ~300s to proxy regardless of encoder** ‚Äî AMD AMF/NVENC GPU encoders are bottlenecked by the HEVC decode stage, not the encode stage. A 246 MB, 31s 4K DJI clip at high bitrate still took 78s to proxy via AMF (2.5x realtime) due to dense intra-frame complexity. Sessions with clips longer than ~2min will exceed the 120s gate target. Mitigation: partial-ready gate entry (gate advances when all SHORT clips are done and user clicks "continue"); long-clip segmented proxy gen. Treat long-clip sessions as a separate batch.
- **480p ultrafast proxies for scrubbing: ~4‚Äì8s per 30s clip** ‚Äî `-preset ultrafast -crf 28 -vf scale=-2:480` gives ~4√ó speedup over the FFmpeg default `medium` preset. For scrubbing use (not final output), ultrafast quality is indistinguishable. Do NOT use 720p for proxies ‚Äî pixel count difference (~2.25√ó) adds measurable encode time with no UX benefit at the small TrimBar display size.
- **Try source first; transcode only on `onError`** ‚Äî WebView2 cannot decode HEVC without the Windows HEVC Video Extension, but the extension is pre-installed on many Windows 11 machines. The correct architecture is: set `src = convertFileSrc(local_path)` always; only trigger proxy gen when WebView2 fires `onError`. For H.264/VP9/AV1 (always native in WebView2) and HEVC with extension (most Win11 users), playback is instant at source resolution ‚Äî no transcode, no wait. Never pre-generate proxies for all clips upfront; this is pure waste for the majority of users.
- **GPU encoder detection via OnceLock ‚Äî test once, reuse forever** ‚Äî `h264_nvenc` (Nvidia), `h264_qsv` (Intel QuickSync), and `h264_amf` (AMD) cut proxy encode time from 30‚Äì40s to 3‚Äì5s on a 1.5-min 4K HEVC clip. Detect the best available encoder once with a 1-frame lavfi test encode (`-f null -`), cache in `static BEST_ENCODER: OnceLock<String>`. Fallback: `libx264`. Pattern in `src-tauri/src/lib.rs` `detect_best_encoder()`.
- **[#92 ‚Äî CORRECTS a prior claim] Naming a GPU *output* encoder does NOT hardware-decode the input ‚Äî proxy gen needs an explicit `-hwaccel` flag.** An earlier note here said "GPU encoders handle their own hardware decode ‚Äî no separate `-hwaccel` flag needed." That was wrong. `-c:v h264_amf` only selects the *output* encoder; without `-hwaccel` FFmpeg software-decodes the 4K HEVC input at ~1x realtime, which is the real proxy bottleneck (decode, not encode). Adding `-hwaccel d3d11va` before `-i` measured **1.9x faster (92.1s ‚Üí 48.6s)** on a 52.85s 4K HEVC Main10 clip (AMD Radeon iGPU, driver 31.0.21925.1001). Applied to all three proxy functions in `lib.rs` (`generate_proxy_file_low_priority`, `_normal_priority`, `generate_proxy_file`).
  - **Decode-only, NOT `-hwaccel_output_format`.** Per [AMD's own FFmpeg-AMF docs](https://github.com/GPUOpen-LibrariesAndSDKs/AMF/wiki/FFmpeg-and-AMF-HW-Acceleration): when a `-vf` filter (we have `scale=-2:2160,format=yuv420p`) is present, `-hwaccel_output_format` cannot be used ‚Äî the filter runs on CPU regardless. So `-hwaccel d3d11va` alone (frames auto-download for the software scale) is the correct and only shape here. A full-GPU `scale_d3d11` path is not worth it.
  - **[TRAP] AMD driver bug #455:** `d3d11va`+AMF can emit a ~1KB file or hang on some Adrenalin versions ([AMF#455](https://github.com/GPUOpen-LibrariesAndSDKs/AMF/issues/455)). Fallback is `-hwaccel dxva2` (DX9), which works identically with software scale filters. Not needed on driver 31.0.21925.1001 (verified clean), but if a future proxy run produces a ~1KB output or hangs, switch to `dxva2` and check whether the driver changed.
- **[#92 ‚Äî lazy proxy path was silently broken on AMF] `generate_proxy_file` (the on-demand 1080p cold-seek proxy in `lib.rs`) passed `-preset ultrafast -crf 23` to whatever `detect_best_encoder()` returned, with no `is_amf` branch** ‚Äî unlike the two 2160p bg functions. On AMD machines that meant `h264_amf -preset ultrafast`, which ffmpeg 8.0.1 rejects with `Unable to parse option value "ultrafast"` (exit 234, NO output file). It also piped stderr to `Stdio::null()`, so the failure was invisible ‚Äî the cold-seek proxy never appeared, contributing to the #92 "cold seek never resolves" symptom (not just "slow"). Fix: branch on `is_amf` (AMF gets `-rc cqp -qp_i/-qp_p -quality speed` + bt709, libx264 gets `-preset/-crf`) and capture stderr to `proxy-bg.log`. Any new proxy/encode call site must mirror the same AMF-vs-libx264 arg split ‚Äî never pass libx264 preset/crf args to a GPU encoder.
- **`onError` double-fire guard for lazy proxy gen** ‚Äî `onError` on a `<video>` element fires every time a failed source is re-presented (e.g. React re-renders). Using `!clip.proxy_path` as the guard is insufficient: if proxy gen fails silently and proxy_path is never set, the trigger fires infinitely. Use a `Set<string>` ref (`generatingProxyRef`) keyed by clip ID; add on first trigger, check before re-triggering. Clear on clip nav so each clip gets one attempt.

## WebView2 ‚Äî `<video> onError` fires on decode failure too, not just missing files

**Problem:** `Render.tsx` showed "this render is no longer on disk" via `onError={() => setVideoMissing(true)}`. But `onError` fires for ANY media error ‚Äî including a WebView2 decode crash *during playback* of a file that is present and valid (e.g. a high-bitrate 4K clip mid-play). The user saw the missing-file warning appear seconds into a video that was sitting in the folder the whole time.
**Solution:** Track whether the video successfully loaded before treating an error as "missing": `const videoLoadedRef = useRef(false)`; set `true` in `onLoadedMetadata`; in `onError` only `setVideoMissing(true)` when `!videoLoadedRef.current`. Reset the ref to `false` everywhere a new src is assigned (`applyLatestRender`, `startNewVersion`, pipeline-done listener). A decode failure after a successful load is a playback problem, not a missing file ‚Äî don't mislabel it.
**Context:** `src/pages/Render.tsx`. Any `<video onError>` that distinguishes "file gone" from "couldn't play this frame" needs the loaded-ref gate. The underlying decode crash is the U4e bitrate problem (VBR cap fixes it); the ref just stops the wrong error message.

## E2E ‚Äî WDIO BiDi reports stale about:blank for WebView2 attach mode

**Problem:** WDIO v9 enables WebDriver BiDi by default; `browsingContext.getTree` returns `about:blank` even when CDP `/json/list` shows the correct URL ‚Äî the BiDi protocol has a known mismatch with WebView2's CDP attach implementation.

**Problem:** URL `waitUntil` strings and `toContain()` text checks become stale without any compile error when routing or UI copy changes. Examples: `gap-editor.spec.ts` waited for `/editor/` after "Open project" routes to `/trimmer/`; `trimmer.spec.ts` checked for `"In Film"` text that was removed in Batch 16b C3 (replaced by a green SVG dot badge with no text).

## E2E ‚Äî CDP /json/list URL vs WebDriver getUrl() mismatch

**Problem:** CDP REST `/json/list[].url` reflects the browser process's pending navigation target (e.g. `http://localhost:1420/`), while WebDriver `GET /url` reflects the renderer process (which may still show `about:blank` during navigation).
**Solution:** Use `/json/list` only to confirm the app has launched and navigated away from blank ‚Äî not as a proxy for what WebDriver will return. Always use `browser.waitUntil(getUrl())` after attaching msedgedriver.

## E2E ‚Äî Tailwind hover variants appear in class attribute (toContain matches inactive state)

**Problem:** `expect(el.getAttribute("class")).toContain("border-white/60")` matches even when the element is *inactive*, because the inactive class string contains `hover:border-white/60`. The Tailwind class attribute is a literal space-separated string of utility names including all variants.
**Solution:** When asserting active vs inactive state with `toContain`, pick a token that exists **only** in the active class ‚Äî not also present as a hover variant of the inactive class. Prefer background tokens (`bg-white/15`) over border tokens when the inactive class uses a matching hover border (`hover:border-white/60`). For inactive negation use `not.toContain("bg-white/15")`.

## UX / timing feedback

- **Rolling inactivity timeout beats wall-clock timeout for long pipelines** ‚Äî a hard `setTimeout(10min)` fires even when the pipeline is making steady forward progress, producing a false "timed out" error. Instead: start the timer on mount and reset it on each `pipeline-stage` event. Do NOT reset on every `pipeline-progress` tick ‚Äî a hung pipeline that emits noisy progress would never time out. The timer fires only when no stage change has arrived for the full timeout window.
- **ETA countdown timers are unreliable for non-linear pipelines** ‚Äî a remaining-time estimate based on `elapsed / progress * (100 - progress)` grows during slow pipeline stages (e.g. loudnorm), making it worse than nothing. Use a simple count-up elapsed timer instead (`useRef<number>(Date.now())` on component mount, tick every second). Users calibrate expectations from "it took 30s last time" not from a fluctuating estimate.
- **Start elapsed timer on mount, not on first progress event** ‚Äî initialise `startTimeRef = useRef<number>(Date.now())` at declaration time so the counter starts at 0 immediately; initialising lazily (e.g. on first `progress > 0`) causes a visible delay before counting starts.

## WebView2 ‚Äî `play()` called after `currentTime` write races the seek and silently stays paused

**Problem:** In WebView2 (and Chromium generally), calling `ma.play()` on the line immediately after `ma.currentTime = target` produces a resolved promise but leaves the element paused (`paused: true` after `.then()`). The seek starts asynchronously; `play()` queues before the seek settles, and the browser resolves the promise without actually starting playback. Confirmed via diagnostic logs: `play() RESOLVED { paused: true, muted: false }`.
**Solution:** Call `ma.play()` inside the mute-bridge `seeked` handler ‚Äî not after the `currentTime` assignment line. By the time `seeked` fires, the browser has confirmed data at the new position and `play()` reliably starts. Pattern used in `Sound.tsx` `trySync` (U6a): `ma.addEventListener("seeked", () => { ma.muted = false; if (shouldPlay) ma.play().catch(() => {}); }, { once: true }); ma.currentTime = target;`
**Context:** `Sound.tsx` `seekToFilmMs` `trySync()` ‚Äî wasIdle branch (post-film-end scrub). Applies to any `<audio>` or `<video>` where play must resume after a programmatic seek. The mute-bridge pattern is already in LEARNINGS (`Tauri ‚Äî audio dropout on currentTime write`) ‚Äî this is an extension of it covering the play-resume case.

## HTMLAudioElement ‚Äî use `ma.ended` not `ma.paused` to detect "track played to completion"

**Context:** `Sound.tsx` `seekToFilmMs` ‚Äî any branch that decides whether to call `ma.play()` after a seek. Applies broadly to any audio element where "reached the end" needs to be distinguished from other paused states.

## Pipeline ‚Äî Zoom warm/render race: throughput, not trigger timing, is the root bottleneck

**Problem:** `warm_zoom.py` and `render.py` `_zoom_worker` are independent cross-process producers. Even after bumping warm to NORMAL priority and adding a Sound-screen mount trigger, the render's zoom step still stalled 71s on a 4-clip cold project. Root cause: `warm_zoom.py` was fully **serial** ‚Äî one `(clip, resolution)` pair at a time. For 4 clips √ó 2 resolutions with 4 cold entries, serial total was 192s; the render started its zoom step ~38s into the warm, guaranteeing a race. Timing confirmed: `[zoom-coord]` logs show warm `encode_start` pairs separated by seconds, not milliseconds.
**Solution:** Three-part fix (#50): (A) bump warm spawn from `BELOW_NORMAL_PRIORITY_CLASS` to `NORMAL_PRIORITY_CLASS` in `lib.rs` `warm_zoom_cache_cmd`; (B) add a mount `useEffect` in `Sound.tsx` that calls `warm_zoom_cache_cmd` (the `{project_id}:zoom` Rust guard dedupes against the Arrange-unmount fire); (C) parallelize `warm_zoom.py` main() with `ThreadPoolExecutor(max_workers=2)` ‚Äî flatten `(clip_idx, clip, resolution)` into one work queue, dispatch all pairs, collect via `as_completed`. Per-key filesystem locks in `zoom_cache.py` (`acquire_or_wait`) handle concurrent access safely; temp filenames include `{clip_idx}_{resolution}` so workers never collide. Result: 2 cold clips √ó 2 resolutions encode as 2 parallel pairs (~51s ceiling vs 192s serial). Acceptance gate: `zoom_cache_hits == clip count` and `t_zoom_s < 5s` in `render-timing-log.jsonl`.
**Context:** `pipeline/warm_zoom.py` main() (parallel dispatch, max_workers=2), `pipeline/zoom_cache.py` (`acquire_or_wait` lock), `pipeline/render.py` `_zoom_worker`, `src-tauri/src/lib.rs` `warm_zoom_cache_cmd` (NORMAL priority), `src/pages/Sound.tsx` (mount trigger). **Do not raise to 4 workers** without profiling: 4 concurrent libx264 encodes at NORMAL contend heavily for CPU and can degrade a concurrent render. Worst-case (all clips cold + immediate render) still races at 2 workers; Option C (render preflight gate) remains deferred until logs show real-workflow misses.

## Pipeline ‚Äî ThreadPoolExecutor: use per-index bool array, not shared counter

**Problem:** Inside `ThreadPoolExecutor`, incrementing a shared counter with `counter[0] += 1` inside a worker is a read-modify-write race ‚Äî the GIL does not protect the compound `+= 1` operation on a list element when multiple threads execute it simultaneously.
**Solution:** Use a per-index bool array: `flags: list[bool] = [False] * n`. Each worker thread writes only to its own index (`flags[i] = True`), which is safe because thread `i` is the sole writer to index `i`. Sum after the pool joins: `count = sum(flags)`. This is the same pattern as the existing `zoom_status: list[str]` array in `render.py`.
**Context:** Any `ThreadPoolExecutor` use in `pipeline/render.py` where a boolean per-clip outcome (e.g. "did this clip use a proxy as zoom input?") needs to be counted after the pool. The mutable-list closure `flag = [False]` / `flag[0] = True` pattern is safe only when a single nested function (not a parallel pool) is the sole writer.

Opening 17+ such paths concurrently via the WSL 9P driver can flood the kernel page-cache allocator and cause WSL VM restart (exit 15).
**Solution:** `_localise_inputs()` in `render.py` (called immediately before the U1g render) copies any `/mnt/c/...` input to `render_in_{i}.mp4` in the job's WSL tmpfs dir. This is why `render_in_N.mp4` appears in ffmpeg commands even for clips that were not zoom-pre-encoded. If investigating a `render_in_N.mp4` reference in a log, check whether it came from Step 3 zoom encode or from `_localise_inputs()` ‚Äî look for the `[render] localise input N: PROXY_HASH.mp4` log line to distinguish.

## Workflow ‚Äî WSL fresh-distro install hangs non-interactively

**Problem:** `wsl --install -d Ubuntu-24.04` on a machine with WSL fully uninstalled (features disabled) needs two steps: enabling the Windows features (requires elevation ‚Äî Claude Code's shell cannot elevate, no UAC interaction possible) and then registering + first-launching the distro. The auto-launch after registration runs Ubuntu's interactive first-boot wizard (create UNIX username/password), which blocks forever waiting on stdin when invoked non-interactively (background process, no TTY). Symptom: `wsl -l -v` shows the distro stuck at "Provisioning... this might take a while" for 15+ minutes with near-zero cumulative CPU and no network activity ‚Äî this is a hang, not a slow download. Retrying `wsl --install` again on top of a stuck distro corrupts the appx registration (`Get-AppxPackage` returns nothing afterward).
**Solution:** (1) Elevating the Windows features requires the user to run one elevated PowerShell command themselves ‚Äî there is no workaround. (2) Once features are enabled, skip `wsl --install`'s auto-launch entirely: `wsl --unregister <name>` to clear any stuck/corrupted state, download the official rootfs tarball (`https://cloud-images.ubuntu.com/wsl/releases/24.04/current/ubuntu-noble-wsl-amd64-wsl.rootfs.tar.gz` for 24.04, confirm the exact filename via the directory listing since version numbers shift), then `wsl --import <name> <installDir> <rootfs.tar.gz> --version 2`. Import skips the OOBE wizard entirely and defaults to root ‚Äî which is exactly what this pipeline needs (`-u root` is already the invocation convention, see Invocation section above), so no separate user-creation step is needed at all.
**Context:** One-time dev-machine setup when WSL2 has been uninstalled. Applies to any fresh Ubuntu WSL distro install attempted non-interactively (background Bash/PowerShell tool, CI, remote session) ‚Äî not specific to this project's Python version, just to how Ubuntu's WSL first-boot works.

## Workflow-WSL ‚Äî One-off Python/sqlite checks across PowerShell -> WSL -> Python ‚Äî pipe via stdin, skip the temp file

**Problem:** `python3 -c "..."` invoked from PowerShell through `wsl -d Ubuntu-24.04 -u root` breaks ‚Äî quoting collapses across three shells (PowerShell -> bash -> python), producing `SyntaxError`/`command not found` errors even for a one-line script. The documented fix for *multi-line* scripts (write to a temp `.py` file, then invoke) works but is a two-step round trip (Write tool + PowerShell call) that's overkill for a single ad-hoc DB read.
**Solution:** For a one-off script (DB inspection, quick assertion check, args dump) that doesn't need to persist on disk, use a PowerShell here-string piped directly into `wsl ... python3` via stdin ‚Äî no temp file, no quoting collapse: `$py = @'<multi-line python>'@; $py | wsl -d Ubuntu-24.04 -u root python3`. Bash never re-parses the string (it arrives on stdin, not as a `-c` argument), so triple-quotes, f-strings, and nested quotes all pass through untouched.
**Context:** Any single-shot PowerShell->WSL->Python check (sqlite reads, encoder-args assertions, ffprobe post-processing) where the script's only purpose is this one invocation. Still use the existing temp-`.py`-file pattern (`pipeline.md`) for scripts worth re-running or debugging (diagnostic/repair scripts) ‚Äî the stdin shortcut is for genuinely throwaway one-liners.

## Workflow ‚Äî Don't strengthen acceptance checks beyond the approved plan's literal wording when briefing a cold-context reviewer

## Workflow ‚Äî Memory files are soft, `UserPromptSubmit` is a suggestion, `PreToolUse` is the actual enforcement boundary

**Problem:** On issue #75, the user's opening message contained the literal trigger phrase "dev plan" for the `rushcut-dev-plan` skill, but it was treated as generic instruction text and answered with an ad-hoc manual Plan Mode flow instead of calling the Skill tool ‚Äî silently skipping the auto-invoked `rushcut-qa-reviewer` step. A memory file was written to flag this, then a `UserPromptSubmit` hook that injects a mandatory instruction. Both are still soft: a memory entry is a recall bias the model can miss again, and `UserPromptSubmit` only injects context into the prompt ‚Äî the model can still ignore it at the moment it actually picks a tool, since nothing stops the tool call itself from succeeding.
**Solution (v1, superseded):** `UserPromptSubmit` wrote a small per-session state file that `PreToolUse` read on every subsequent tool call, denying anything except `Skill`/`AskUserQuestion` until the matching `Skill` call cleared it. Verified end-to-end against the live harness (armed this session's real `session_id`, confirmed subsequent `Bash` calls were genuinely denied, confirmed the attempt-based fail-open auto-cleared correctly) ‚Äî but one anomaly surfaced: the very first live tool call issued right after arming was not blocked (every call after that was, reliably). Root cause was never pinned down, but the failure mode is structural regardless of cause: two independent hook processes (`UserPromptSubmit` writing, `PreToolUse` reading) coordinating through a file is inherently a cross-process write/read dependency, and a state file that must exist by the time of an unrelated process's read is the wrong shape for "must never be skipped."
**Solution (v2, current):** Removed the shared state file entirely. `PreToolUse` (`enforce-skill-gate.js`, matcher `".*"`) is now fully self-contained ‚Äî it reads `transcript_path` (present in every hook payload), finds the most recent genuine human-typed message (`lib/transcript.js` `findLastHumanMessage()`), re-runs the same trigger-phrase match `UserPromptSubmit` uses (`lib/trigger-match.js`, now shared by both hooks), and only denies if no matching `Skill` tool_use appears later in the transcript. Zero cross-hook coordination, so the write/read race category is gone by construction, not mitigated. Non-obvious gotcha found while building this: a `Skill` invocation's own body text is injected into the transcript as a synthetic `user`-role entry too (marked `isMeta: true`, with a `sourceToolUseID`) ‚Äî naively scanning all `user`-role text for trigger phrases would false-positive on a skill's own markdown containing a phrase like "wrap up". `isGenuineHumanMessage()` filters these out (`isMeta` check) along with `tool_result` entries (also `user`-role, structurally distinguishable by content block `type`). Verified via 6 synthetic-transcript scenarios (deny / already-satisfied-allow / skill-body-false-positive-avoided / no-trigger-allow / exempt-tool-allow / unreadable-transcript-fail-open) rather than re-testing live against this session, since arming the old design's gate live had already caused a real self-inflicted lockup requiring 3 burned attempts to clear ‚Äî the new design has no file to manually arm, and fabricating a fake trigger message in the live transcript to test it would mean tampering with real session history.
**Context:** Any "when the user says X, action Y must always happen first" rule where the cost of a miss is high (e.g. skipping an automated QA step). `UserPromptSubmit`-only is fine for a nice-to-have nudge; use the `PreToolUse` gate pattern when the rule must actually hold. Always build in a fail-open expiry (time and/or attempt count) ‚Äî a `PreToolUse` deny with no escape hatch can wedge a whole session if the state ever gets stuck armed.

## Pipeline ‚Äî seed randomized-but-cacheable features from the cache signature, not job/request ID

**Problem:** A per-render random choice (e.g. shuffle transition order) seeded from `job.id` breaks any cache keyed on inputs ‚Äî identical clip-set+settings can never reproduce the same random choice, so a cache built for that feature is permanently a MISS even though every other cache-relevant input matches. This was `render_cache`'s v1 limitation for `shuffle_between` (#19/#83): the RNG seed and the cache key were computed from two different things (job.id vs. clip+settings hash), so they could never agree.
**Solution:** Seed the RNG from the same stable signature already used for cache lookup (`render_cache.signature()`'s `cache_sig`), not from the per-invocation job/request ID. Compute the signature once, early, before any random-choice code path runs, and pass that single value everywhere the old per-render ID was used as a seed. Verified end-to-end: two renders with identical clips+settings now produce the identical `cache_sig`, identical shuffle order, and a cache HIT; changing any signature-relevant setting changes both the shuffle order and the cache key together (correct invalidation).
**Context:** `pipeline/render.py` (`cache_sig` at the top of `run_pipeline`) + `pipeline/transitions.py` (`resolve_cut_names`, `build_filter_complex` `seed` param). Applies to any future "randomize X per render" pipeline feature that should also participate in the render cache ‚Äî the seed and the cache key must always be derived from the same inputs.

## Workflow ‚Äî git-filter-repo history rewrite ‚Äî use WSL when Windows Python/pip is unavailable, and `git config --global --add safe.directory` is required for the WSL-mounted NTFS repo

**Problem:** `git-filter-repo` needs a working `pip` to install. When Windows pip is broken (see above) or git-filter-repo isn't installed at all, the pipeline's WSL2 Python3/pip (already used for the render pipeline) is a ready fallback ‚Äî but running git commands against the NTFS-mounted repo path (`/mnt/c/apps/rushcut`) from WSL immediately fails with `fatal: detected dubious ownership in repository`.
**Solution:** `wsl -d Ubuntu-24.04 -u root -- pip3 install --break-system-packages git-filter-repo`, then `git config --global --add safe.directory /mnt/c/apps/rushcut` once (persists across WSL sessions) before running `git filter-repo --path <path> --invert-paths --force` from `/mnt/c/apps/rushcut`. `filter-repo` strips the `origin` remote as a safety measure ‚Äî re-add it with `git remote add origin <url>` afterward before pushing. If the rewritten branch's HEAD hash is unchanged after the rewrite, that means the path being removed was never actually in that branch's ancestry (only reachable via a different local branch) ‚Äî `git push --force` will correctly report "Everything up-to-date" in that case; it's not a sign the rewrite failed. ## Workflow ‚Äî a commit message describing a secret/PII purge can silently reintroduce the exact string it removed

**Problem:** After scrubbing a leaked token or personal username from file contents, writing a commit message that explains the fix by quoting the old value verbatim (`"X" -> "Y"`, or `git grep -n <the-secret>` as an example command) re-embeds that same string into git history via the message text ‚Äî even though every file diff in that same commit is clean. `git grep`/`git log -p` searches over file content won't catch this; only a message-inclusive search (`git log --all --format="%B" | grep ...`) will.

## Pipeline ‚Äî parallelising short FFmpeg re-encode jobs hits a fixed floor that is encode-specific, not a generic per-invocation cost

**Problem:** #99 parallelised `trim.py`'s Step 2 per-clip trim with a `ThreadPoolExecutor` mirroring `normalise.py`'s proven pattern (`MAX_PARALLEL_TRIM = min(4, os.cpu_count())`, per-worker `-threads` cap). Expected ~3-4x speedup (matching `normalise.py`'s precedent); measured only ~1.1-1.3x across every configuration tried (4√ó2, 2√ó4, 3√ó2 workers√óthreads; tested on both 8-core and 16-core WSL allocations; tested with inputs on `/mnt/c` NTFS and pre-localised to tmpfs ‚Äî none of these variables moved the result). Per-job wall time was consistent across all configs: roughly a **7-8s fixed floor** regardless of thread count or concurrency, plus a smaller component proportional to output duration (~1.4x realtime). Two hypotheses were tested and disconfirmed at the time: (1) CPU thread-starvation from capping `-threads` too low; (2) WSL 9P/NTFS I/O contention from concurrent `/mnt/c` reads. #99 provisionally attributed the floor to generic FFmpeg process-spawn/probe/init overhead.
**Correction (2026-07-11, #104 scoping pass):** that attribution was wrong. Reproduced #99's exact 4-way-contention conditions and additionally timed a pure `-c copy` invocation (no encode) under the same contention: it took ~0.28-0.29s/job ‚Äî no floor at all ‚Äî while the full-reencode jobs reproduced #99's 7.88-7.90s/job exactly. **The floor is libx264 encode work competing for CPU under thread-capped contention, not FFmpeg process-spawn/probe/init cost.** A stream-copy job does no encode work, so it doesn't compete for that resource.
**Solution:** The parallelism from #99 is harmless and ships (correct output, ~1.2x average win), but a `ThreadPoolExecutor` wrapper alone cannot reduce the encode-floor itself ‚Äî it can only add more workers each paying their own share of real encode work. **#104 and #107 both then tried to reduce the encode work itself and both closed NO-GO on real measurement ‚Äî see the two entries directly below.** The scoping-pass GO this entry originally described did not survive contact with a real end-to-end build.
**Context:** Any future "parallelise this per-clip FFmpeg loop" change, or any attempt to explain away a stubborn per-job floor as "process overhead." Before accepting that framing, isolate whether the job actually does encode work ‚Äî time a `-c copy`/remux equivalent under the same contention conditions first. If copy is much cheaper than encode under contention, the floor is encode-specific ‚Äî but that alone does NOT guarantee a composed fix (fewer-encoded-seconds + extra copy/concat/probe calls) nets a real win; see the partial-GOP entry below for why.

## Pipeline ‚Äî partial-GOP trim (#104): a cheaper head-reencode does NOT survive composing it with copy+concat+probe overhead

**Problem:** The #104 scoping pass (entry above) correctly showed the per-job floor is encode-specific, and a follow-up isolated measurement confirmed a shortened re-encode really is cheaper under contention (4.16s avg for a ~7.3s head segment vs 5.52s avg for the full ~12s window ‚Äî a real 1.36s/job saving). The natural conclusion was that re-encoding only the boundary GOP and stream-copying the rest should be a net win. It wasn't. Built `trim_smart()` (head re-encode via existing `trim()` + `-c copy` tail + concat demuxer), ran it head-to-head against the current `trim()` on the real 6-clip DJI test set under identical 4-way `ThreadPoolExecutor` contention, and it was **19-31% SLOWER** across two separate real runs, not faster. Frame-accuracy was fine (every output matched within tolerance) ‚Äî this was a pure performance regression. All individual pieces tested cheap in isolation (concat-only ~0.4s/job, copy near-zero, the shortened re-encode itself 1.36s/job cheaper) but did not compose into a net win ‚Äî the extra ffprobe keyframe-probe call is the leading unmeasured suspect for where the composed overhead actually went, not conclusively isolated.
**Solution:** Closed NO-GO (same pattern as #107 below ‚Äî scoping/isolated measurement said GO, real end-to-end measurement said NO). `trim_smart()` is left in `pipeline/trim.py`, correct and gate-passing, but NOT wired into `render.py` ‚Äî a code comment on the function documents the regression numbers so it isn't silently re-attempted. Do not re-wire without new evidence the composed overhead (specifically: isolate the keyframe-probe `ffprobe -read_intervals` cost under contention, which was never separated out) has actually changed.
**Context:** `pipeline/trim.py` `trim_smart()` (unused). Any future "shorten the encode, copy/concat the rest" idea for this pipeline. **The general lesson: an isolated single-invocation A/B measurement (this piece is cheaper than that piece) does not predict a composed multi-invocation pipeline's real cost ‚Äî always run the actual end-to-end comparison (old path vs new path, same inputs, same contention) before declaring a win, even after every component-level gate has passed.** Full numbers: issue #104 (closed 2026-07-11).

## Workflow ‚Äî ffprobe CSV output ignores the field order requested in `-show_entries`

**Problem:** `ffprobe -show_entries frame=pkt_pts_time,pict_type,key_frame -of csv=p=0` was assumed to emit columns in the exact order listed (`pkt_pts_time,pict_type,key_frame`). It doesn't ‚Äî ffprobe uses its own internal canonical field order regardless of the order requested (observed: `key_frame,pict_type,pts_time`). A script that indexes CSV columns positionally based on the requested order (e.g. `parts[2] == "1"` expecting `key_frame` there) silently reads the wrong field and can report a plausible-but-wrong result (e.g. "0 keyframes found" on a file that has several) with no error.
**Solution:** Never trust requested `-show_entries` order for positional CSV parsing. Either use `-of csv=p=0` and verify actual column order on a known sample first, or use `-of json`/`-of default` (key=value pairs, order-independent) and parse by field name instead of position. For any diagnostic script that probes structured FFmpeg/ffprobe output and could silently produce a plausible-but-wrong zero/empty result, assert the result is non-empty before trusting it (e.g. `assert len(keyframes) > 0, "parsing bug or genuinely no keyframes -- verify before proceeding"`) ‚Äî a silent zero is a parsing bug until proven otherwise.
**Context:** Any diagnostic or pipeline script using `ffprobe ... -show_entries ... -of csv`. Confirmed 2026-07-11 during the #104 GOP-structure diagnostic ‚Äî an initial "0 keyframes" result was actually 4 real keyframes misread due to this column-order assumption.

## Pipeline ‚Äî `TIMING:normalise=` / `t_normalise_s` brackets B-0 pre-trim, not just `normalise()` work

**Problem:** #100 reported `t_normalise_s=25.0s` on a full proxy-skip render (19 clips, all clips reused a cached 2160p proxy) where `normalise()` is expected to be skipped entirely ‚Äî 10x higher than the expected ~2-5s. On the exact all-proxy scenario this entry originally diagnosed, `TIMING:pretrim` now reads `0.0s (jobs=0)` ‚Äî the wasted work no longer happens, so the misleading-label problem no longer has anything misleading to measure. WSL2 per-process spawn/init overhead (~0.6-0.7s/clip based on #135's live measurement, 1.4s for 2 clips) remains the standing explanation for B-0's per-clip floor when it does run ‚Äî consistent with #104's finding that CPU-starvation and I/O-contention were ruled out as the cause of short-FFmpeg-job floors generally.
**Context:** Any future timing-log analysis or "why is stage X slow" investigation on a proxy-heavy render. `pipeline/render.py`'s Step 1 block (~line 636 onward). Confirmed #100 (2026-07-11), root-caused and fixed #135 (2026-07-20).

## Pipeline ‚Äî progress-bar stage weights are hardcoded fixed percentages, mis-weighted against real per-stage cost (#12, fixed 2026-07-12)

**Problem:** `run_pipeline()`'s `report(N)` calls used hardcoded percentages tied to pipeline steps (e.g. `report(60)` "Rendering" ‚Üí `report(80)` "Mixing music") regardless of whether that step ran or how long it actually took. Two compounding effects: (1) a skipped stage (no music/cards/zoom) still consumed its fixed checkpoint tick, wasting bar range on work that took ~0s; (2) the "Rendering" step ‚Äî unconditional and, per `render-timing-log.jsonl`, typically 60-70% of total wall time ‚Äî only ever owned a fixed 20-point window with zero interior ticks, so the bar sat frozen through the single longest step, then jumped rapidly through several trailing checkpoints once it finished. Net effect: "stuck then jumps to 100%."
**Solution:** Compute stage weight buckets as named constants (`STAGE_WEIGHTS`, empirical stable buckets from real timing-log ratios, NOT live-derived per-render ‚Äî a single render's ratios overfit to its own clip count/resolution) and redistribute dynamically at pipeline start via `_compute_checkpoints(normalise_active, trim_active, zoom_active, cards_active)`: excluded (inactive) stages contribute zero weight, and the remaining active stages' weights are renormalized to fill 0->98. `render` and `audio` are never zero-weighted (they always run, even as a fast passthrough) so they always absorb whatever weight a skipped stage would have used ‚Äî this is what fixes both the "stuck" (skipped stage no longer parks the bar) and "jump" (render's window now honestly reflects it's the dominant cost) symptoms simultaneously. Verified live: two real renders on a fully-proxied 4-clip project produced exact-match `PROGRESS:` sequences against `_compute_checkpoints()`'s output (e.g. `0,22,25,27,93,98`), including three consecutive same-value ticks firing within 2ms when zoom+cards were both skipped (zero visible stall).
**Context:** Any future change to `run_pipeline()`'s stage structure (new stage, stage removed, or reordered) must update `STAGE_WEIGHTS`/`_compute_checkpoints()`/`STAGE_ORDER` together ‚Äî the checkpoint table assumes a fixed stage order (`normalise, trim, zoom, cards, render, audio`). Real interior progress ticks *during* the FFmpeg encode itself (frame-based or per-U1g-batch) is explicitly NOT part of this fix ‚Äî the "Rendering" window is now correctly *sized* but still un-ticked internally; that's a separate, bigger follow-up if ever wanted. Interior ticks were later added for Rendering (#119, `-progress pipe:1` frame parsing) and for Trim (#129, per-clip `as_completed` ticks ‚Äî see the dedicated entry below). Normalise already had per-clip ticks from the start via `on_clip_done`.

## Pipeline ‚Äî a GitHub issue's own root-cause diagnosis can be incomplete when the bug is "variable only bound in one branch, read unconditionally later" (#121, fixed 2026-07-12)

**Problem:** #121 reported single-clip renders throwing `UnboundLocalError: has_open` in the ANALYSIS emit (`render.py`'s multi-line f-string at the end of `run_pipeline()`), with a stated fix of initializing just `has_open`/`has_close`. That fix alone would NOT have resolved the crash: `transition` (assigned only in the same multi-clip `else`-branch) has the identical bug and is referenced *earlier* in the same concatenated f-string ‚Äî Python raises on the first unbound-local name it hits in evaluation order, so the real crash is on `transition`, not `has_open`. A fourth variable, `boundary_reencode_s`, shared the same bug and would have surfaced next once `transition` and `has_open`/`has_close` were fixed.
**Solution:** When a bug report names one variable in a class of "assigned only inside an `if`/`else` branch, referenced unconditionally afterward," grep **every** name referenced in that same block/f-string against its own assignment site ‚Äî don't just patch the one name the report calls out. Fixed by pre-initializing all four (`transition`, `has_open`, `has_close`, `boundary_reencode_s`) before the branch, matching how other pre-branch state (`proxy_clip_indices`, `zoom_proxy_input`, `amf_fallback_flag`) is already scoped above it.
**Context:** Any Python `UnboundLocalError` fix in this codebase ‚Äî `report_analysis`/ANALYSIS-emit style functions in `render.py` are especially prone to this pattern since they read many locals unconditionally at the end of a long function with multiple conditional branches above. Verify empirically which name fails first if in doubt: a small standalone repro (`if branch: a=1; b=2` then read both unconditionally) confirms Python raises on the first name in *evaluation order*, not declaration order.

## FFmpeg ‚Äî batching N per-clip encodes into one `filter_complex` call is SLOWER and uses ~2x the memory of N separate throttled processes

**Problem:** The intuitive fix for "N separate FFmpeg processes pay a per-process encode-floor under `ThreadPoolExecutor` contention" is to fold all N encodes into one ffmpeg invocation (N inputs, N independent filter chains, N mapped outputs) so there's only one process to schedule. Measured on 6 real DJI clips under RushCut's exact production contention shape (`ThreadPoolExecutor(max_workers=4)`, mirroring `render.py`'s `MAX_PARALLEL_TRIM`): the batched single-process version was ~25-35% **slower** (avg 11.82s vs 8.83s) than the current N-process approach, not faster ‚Äî a single ffmpeg process running N parallel filter-chain+encode branches does not out-schedule N independently-dispatched OS processes under the OS's own scheduler. It also used **~2x the peak memory** (~2.75GB vs ~1.4GB above baseline for N=6) because the one-process design has no analog to `MAX_PARALLEL_TRIM`'s throttle ‚Äî all N encoders run concurrently by default, unlike the current per-clip loop which is capped at 4 concurrent processes.
**Solution:** Don't assume single-process-multiple-output beats multiple-processes for FFmpeg CPU/memory behavior ‚Äî it's workload- and scheduler-dependent (a documented superuser benchmark found the same reversal). If a similar "amortize the per-job floor" idea comes up again for any pipeline step (Step 2 trim, B-0 pre-trim, normalise), measure both directions on real footage before designing ‚Äî do not build the batched version on the strength of the amortization intuition alone. Also tested libx264's `-rc-lookahead 0` as a memory lever (Reddit-documented mechanism: N encoders each buffering `rc-lookahead` frames) ‚Äî it made no measurable difference at this N/resolution (peak mem 3341MB vs 3339MB), so lookahead buffers are not the dominant memory cost here; something else (likely N concurrent decode contexts + filter-graph buffering) is.

## Pipeline ‚Äî Step 2 trim had zero interior progress ticks; `as_completed` + checkpoint interpolation is the reusable fix shape (#129, fixed 2026-07-16)

**Problem:** `render.py`'s Step 2 "Trimming clips" loop submitted all clip trims to a `ThreadPoolExecutor` then iterated `for f in futures: f.result()` (blocks in submission order) and called `report(checkpoints["trim"])` exactly once, after every clip finished ‚Äî zero interior ticks. On a real 4K project this stage measured `t_trim_s=62.2s` (4 clips, `trim()` re-encodes per #96, not a stream copy) with the bar sitting dead at the prior checkpoint the entire time ‚Äî the same "stuck then jumps" symptom #12 fixed for the whole-pipeline stage weighting, just recurring one level deeper inside a single stage that has real per-clip parallel work.
**Solution:** Swap `for f in futures` (submission order) for `for f in as_completed(futures)` (completion order ‚Äî ticks fire as soon as ANY clip finishes, not in a fixed order) with a plain `done` counter incremented only inside that loop on the main thread (safe without a lock ‚Äî `as_completed` yields sequentially, never concurrently with the worker threads that only touch their own `trimmed[i]` slot). Interpolate progress between the two adjacent checkpoint values: `report(int(checkpoints["normalise"] + done/total_jobs * (checkpoints["trim"] - checkpoints["normalise"])))` ‚Äî mirrors the exact pattern Step 1's `_normalise_progress`/`on_clip_done` already used, so no new pattern was invented, just applied to the one stage that had been skipped.
**Context:** `pipeline/render.py` ~lines 749-770. Verified live via direct `pipeline/run.py` invocation (not just `ast.parse`) on a real 4-clip 4K project with genuine user trims, all proxy-skip (the exact scenario from the original bug report): log showed `PROGRESS:5 ‚Üí 11 ‚Üí 17 ‚Üí 23` across the 4 clips instead of one jump at the end. Known residual gap: per-clip tick granularity means the gap between ticks is bounded by single-clip trim time, not sub-3-second ‚Äî closing that further would need frame-level `-progress pipe:1` parsing inside `trim()` itself (the #119 pattern), which is a materially bigger change and was deliberately not in scope for this fix. If a future stage (music mix, U1g per-batch) needs the same treatment, this `as_completed`-with-checkpoint-interpolation shape is the one to copy, not `_normalise_progress` directly (that helper is normalise-specific).
**Update (#135 session, 2026-07-20 ‚Äî residual gap measured concretely, not yet fixed):** The "bounded by single-clip trim time" gap can be much larger than expected when Step 2's trim source is a proxy file rather than a tmpfs normalise intermediate. Two real 4K renders: all-proxy (4 clips, all trim inputs read from `/mnt/c` proxy files) showed a ~40.2s flat window before the first tick; a mixed run (2 of 4 trim inputs from `/mnt/c` proxies, 2 from tmpfs) showed only ~10.4s. Working hypothesis (2 data points, not yet isolated per the investigation-rigor standard): proxy-sourced Step 2 trim reads directly from slow WSL `/mnt/c` (9p protocol) instead of fast tmpfs, unlike normalise-sourced trim whose input already lives in tmpfs. Filed as a separate follow-up issue rather than folded into #135 (different root cause, needs its own A/B isolation) ‚Äî see that issue for status.

This directly matches what the user's own request floated as a fallback. Two things shipped: (1) a per-job-**start** `report_stage()` call in Step 2's `_trim_worker` (previously only fired on completion) guarded by a `threading.Lock` shared with the pre-existing completion-side calls in the main thread's `as_completed` loop ‚Äî the old "no lock needed, done only mutated on the main thread" reasoning no longer held once a worker thread also calls `report_stage()`/`report()` for the same stage; (2) a frontend-only fast liveness watchdog in `Render.tsx` (independent `setInterval`, ~1.5s cadence, 8s threshold ‚Äî deliberately separate from the pre-existing 30s/360s `stalled`-banner mechanism, different purpose) that pulses the progress bar (`animate-pulse` on the existing `#22c55e` fill, no new color) and swaps in reassuring text whenever `lastProgressAtRef` goes stale, reverting the instant either listener fires again. **Verified live via direct `pipeline/run.py` invocation** (real 4-clip 4K project, all-proxy, `/mnt/c` proxy sources, genuine 5s user trims): the per-job-start ticks fired cleanly at t=3.67s with zero garbled/interleaved `STAGE:` lines under real 4-way concurrency (confirms the lock design), but a real flat window of ~15.9s still followed before the first completion tick at t=19.60s (all 4 jobs completed within ~1s of each other, confirming the equal-duration-parallel-jobs hypothesis) ‚Äî **this concretely proves that backend tick-granularity work alone cannot close this gap**, which is exactly why the frontend pulse (not dependent on any backend signal) is the layer that actually satisfies the JTBD. The concurrent percentage-tick/`on_tick`-aggregation design (mapping `-progress pipe:1` fractions from N parallel ffmpeg processes into one `report()` call) was deliberately NOT built for #140 ‚Äî real-world evidence plus competitor precedent both pointed away from it. The `/mnt/c`-vs-tmpfs 9p-latency root cause itself remains unconfirmed/unfixed, intentionally decoupled into its own non-blocking follow-up issue (performance, not perceived-responsiveness) per the real-Perplexity recommendation not to conflate the two problems.
**Live user verification (same session, immediately after shipping):** on a real warm-proxy render the bar flowed with changing status messages and the user did not lose patience ‚Äî JTBD confirmed. **Problem:** `build_open_close_post_fc()` (in `transitions.py`) is written and proven only against U1g's `_boundary_reencode()` call site (multi-clip segmented render). ## Pipeline ‚Äî B-0 pre-trim ran unconditionally for proxy-eligible clips, discarding 100% of the work (#135, fixed 2026-07-20)

**Problem:** `render.py`'s Step 1 ran B-0 pre-trim (`pretrim_one_clip`, a `-c copy` seek) for EVERY clip before computing the proxy-vs-normalise partition, even though `decide_clip_source()` (the partition decision) only reads `proxy_path_wsl` ‚Äî a field B-0 never touches (B-0 only rewrites `in_ms`/`out_ms`). ## FFmpeg ‚Äî `signalstats` needs `metadata=print` chained after it to actually print per-frame stats to stderr

**Problem:** `blend=all_mode=difference,signalstats` alone attaches per-frame stats (YAVG, etc.) as filtergraph-internal frame metadata but prints nothing visible ‚Äî piping to `grep YAVG` silently matches zero lines (looks like the filter didn't run, or the frames are byte-identical, when actually the values were just never surfaced).
**Solution:** Chain `metadata=print:key=lavfi.signalstats.YAVG` (or omit `key=` to print all signalstats keys) after `signalstats` to actually write the values to stderr. Note the filter can emit multiple stat lines per requested output frame (one per internally-decoded frame feeding into the filtergraph before `-vframes 1`'s cutoff is reached, e.g. for GOP-accurate `-ss` seeking) ‚Äî take the last printed value as the one corresponding to the requested output frame, not the first.
**Context:** Any real-render pixel-diff verification comparing two ffmpeg outputs at a timestamp (e.g. proving a Ken Burns/zoom effect actually changed pixels, not just that the command ran without error). Used to verify #123's single-clip zoom fix ‚Äî `blend=all_mode=difference,signalstats,metadata=print:key=lavfi.signalstats.YAVG` on two same-timestamp frames from a zoomed vs. non-zoomed render gave a real, monotonically-meaningful diff signal (~26.7 at early motion, ~31.7 near end of a Ken Burns ease-in, vs. a strong constant ~50 for an always-on static crop).
