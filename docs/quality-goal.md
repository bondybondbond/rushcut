# Quality Goal

Transparency doc — not a strategy doc. Tracks progress toward the visual-quality north star so we don't re-propose rejected ideas or lose sight of the real bottleneck. Companion to [`docs/speed-goal.md`](speed-goal.md) — the two north stars are in tension (higher quality tends to cost render time) and any quality fix proposed here must respect the speed doc's ceiling, not quietly blow past it.

## North star

- **Close the gap to DaVinci Resolve's visual output quality**, with zoom/pan (Ken Burns) as the current headline gap — non-zoomed footage is already close after the 2026-07-12 CQP switch (see Current state below).
- **Hard constraint: no severe render-time regression.** Any fix here must stay inside `docs/speed-goal.md`'s 5-min cold-render ceiling. A quality win that blows the speed budget is not a win — it needs to be scoped as a toggle/opt-in, not a default.
- **Make it testable, not vibes-based.** "Closing the gap" = passing a blind side-by-side TV-check against a DaVinci Resolve export of the *same project*, at 2160p, specifically on zoomed/panned footage. Same TV-check discipline already used for the CQP switch and the #85/#110 encoder work — just anchored against an external DaVinci reference instead of an internal before/after.

## Current state (as of 2026-07-14)

- **Non-zoomed 4K quality: materially improved.** The launch-plan #1.1/#1.2 switch from AMF VBR (40M/50M) to CQP (`qp_i=18/qp_p=20`) was TV-checked and confirmed sharper on non-zoomed pan-heavy footage.
- **Zoomed (Ken Burns) footage: now looks soft *by comparison*.** Filed as #116. Not yet established whether this is a regression the CQP change exposed (rest of the frame got sharper, so the zoomed region's pre-existing softness now stands out) or something that got worse. That's the first thing any #116 work should establish.
- **Hypothesis #4 (unsharp mask) tested and REVERTED, 2026-07-14 — killed on speed alone, no TV-check performed.** `unsharp=5:5:1.0:5:5:0.0` (FFmpeg's documented default: luma 5x5/amount 1.0, chroma disabled) appended post-crop in `_kenburns_vf()` only (static crop branch untouched). Real A/B on the exact same 2-clip 4K project/manifest as hypotheses #1/#2 (reconstructed byte-for-byte from the prior sessions' logs — clips, trim windows, zoom modes, focal points all matched, `[cache] MISS` confirmed, no AMF): `render_s` 181.5s -> 227.1s (+25%), well past the pre-committed 200s/10% ceiling. Per the pre-committed rule ("if any one criterion fails, revert immediately, skip TV-check"), reverted without ever visually judging sharpness or halo risk -- the speed criterion alone was sufficient to kill it. `pipeline/zoom.py` diff is empty (confirmed via `git diff --stat`), nothing shipped. **All three ranked hypotheses (#1 proxy-QP30, #2 lanczos, #4 unsharp) are now tested and rejected, all on the same real-cost-no-benefit-or-too-costly pattern.** Real Perplexity (GPT-5.6 Terra, consulted before this test) independently confirmed the correct disposition ahead of time: market research found no consumer/prosumer NLE auto-sharpens post-zoom by default (unsharp is user-invoked, not a default), so even a hypothetical pass here should have shipped as opt-in, not default -- moot now given the speed-only kill.
- **Hypothesis #2 REVISED and REVERTED, 2026-07-14 — no quality benefit, real speed cost, not worth it.** `flags=lanczos` was added to all 3 `scale=` call sites in `pipeline/zoom.py`, real-tested (2-clip 4K, bicubic vs lanczos, both cache-MISS-verified, no AMF), then reverted same session. See entry below — same disposition pattern as hypothesis #1/#124.
- **Hypothesis #1 REVISED and REVERTED, 2026-07-14 — no quality benefit, real speed cost, not worth it.** #124 was implemented and code-verified working exactly as designed (real render on the same 2-clip project: `[C-proxy] 0 proxy-skip / 2 normalise`, cache `MISS`, both clips logged the new "zoom bypass" reason string — this was a genuine fresh normalise-sourced render, not a stale cached proxy-based one). Despite that, a follow-up TV check on the real shipped code path found: *"no visible sharpness improvement observed... zoom still looks soft."* This directly contradicts #118 Run B's earlier verdict ("clearly sharper... a start") on the identical project, which turns out to have been based only on the debug `force_normalise` flag, not the actual shipped condition. Combined with the real measured cost (+93s/2 zoomed clips, ~46.5s/clip at 4K — extrapolating to ~880s on a Ken-Burns-heavy 19-clip project against a speed budget already 1.7-2.35x over target), **the fix was reverted the same day.** Speed is priority #1; a confirmed-zero-benefit change does not get to keep a confirmed-real cost. Treat #118 Run B's original verdict as unreliable/inconclusive going forward, and do not assume proxy-vs-normalise contributes measurably to #116. Hypotheses #2 (bicubic-vs-lanczos) and #4 (unsharp mask) are now the primary open suspects for #116, with no confirmed lead yet. See `docs/speed-goal.md` #124 entry and `feedback_confirm_benefit_before_speed_cost.md`.

## #116 verdict (2026-07-14) — closing as won't-fix-at-this-render-budget

All three ranked filter/source-layer hypotheses are now tested and rejected in a single day:
1. Proxy QP30 bypass (#124) — no visible benefit, +93s/2 zoomed clips.
2. Bicubic→lanczos scale (`flags=lanczos`) — no visible benefit, +43% `render_s`.
3. Post-scale unsharp mask (`unsharp=5:5:1.0:5:5:0.0`) — killed on speed alone (+25% `render_s`, never reached TV-check).

No remaining untested lever inside the zoom filtergraph. Per the recommended next-steps (2026-07-14 issue comment) and real Perplexity's independent Gate-1 verdict: close #116 as **"Ken Burns softness at aggressive zoom depths is an inherent limitation at this render budget — not filter-layer-fixable."** The one credible remaining lever is architectural, not a filter tweak: a zoom-depth cap in the UI (e.g. "max 130% zoom on 4K") to prevent the softness from occurring at all, rather than trying to correct it after the fact. Not scoped or filed as its own issue yet — a future session's call, not an automatic follow-up.

## Outstanding hypotheses / relevant open issues (ranked)

1. **[SHIPPED as #124 then REVERTED, 2026-07-14 — NO-GO, no quality benefit + unacceptable speed cost, see Current state above] Proxy QP30 as final 4K render source.** Background proxies (`generate_proxy_file_low_priority`/`_normal_priority` in `src-tauri/src/lib.rs`) encode 2160p at `-rc cqp -qp_i 30 -qp_p 30` — a speed-tuned setting meant for scrubbing preview, not delivery. The render.py proxy-reuse gate (`.claude/rules/pipeline.md`) accepts any 2160p+ proxy as a full substitute for normalise on **any** 4K render, zoomed or not — so whenever proxies are warm (the common case), the *entire film's* final-render source is already QP30-compressed before zoom or the final encode ever runs. Zoom then re-magnifies that compression on top, confirmed via #118 Run B as a real contributor to #116, not just a plausible one. **Architectural framing:** the general NLE convention (DaVinci, Premiere, and others) is proxy-for-preview / full-resolution-source-for-final-render — final delivery is never gated by a preview-tier proxy. RushCut's gate treating a scrubbing-tier proxy as valid final-render input is a structural divergence from that convention, not just a tuning knob. **Test performed (#118 Run B):** #120's `force_normalise` debug flag (required a `cargo build` first — the flag's code had never actually been compiled into the running binary) forced fresh `normalise()` instead of proxy reuse on a 2-clip 4K project with Ken Burns zoom on both clips, identical CQP encode settings otherwise. No-zoom baseline: inconclusive on a PC monitor. Zoomed, evaluated on a TV: forced-normalise output noticeably better but not dramatically so — user's calibrated verdict was "better, but not insanely much sharper... a start," not a full fix. **Likely needs to be paired with #2 (bicubic→lanczos) and/or #4 (unsharp) to close the remaining gap — don't expect this fix alone to fully resolve #116.** **Recommended follow-up (not yet built):** a quality-mode flag/behavior change so proxy reuse is bypassed at final render — at minimum for zoomed clips, possibly for all final 4K renders — mirroring DaVinci Resolve's "revert to camera originals on Deliver" default. The mechanism already works end-to-end (`force_normalise`); what's left is deciding and shipping the default policy, not new plumbing.
2. **[TESTED 2026-07-14 — NO-GO, reverted same session] #116 — bicubic→lanczos scale filter.** Verified directly against this machine's real FFmpeg build (`ffmpeg -h filter=scale`): the `scale` filter's own `flags` option defaults to empty string, which falls through to the global `-sws_flags` default — confirmed **bicubic**, not bilinear (`-sws_flags <flags> ... (default bicubic)`). No code in `pipeline/` ever set `-sws_flags` or `flags=` on any `scale=` call, so `zoom.py`'s Ken Burns per-frame scale (`zoom.py:147`) and static crop-then-upscale path (`zoom.py:193`/`264`) were genuinely running on FFmpeg's bicubic default. **Real A/B performed:** `flags=lanczos` added to all 3 `scale=` sites in `pipeline/zoom.py`, tested via direct `pipeline/run.py` invocation (bypassing Tauri/DB, valid for non-AMF code-path A/Bs) on a real 2-clip 4K Ken Burns project, bicubic vs lanczos, both runs confirmed `[cache] MISS` (no stale-cache contamination), no AMF (`use_amf=False`, isolates the filter variable cleanly). **Quality result: no visible improvement.** Matched-timestamp (t=20s) still frames extracted from both renders and compared at native pixel resolution on two regions — (a) the zoomed subject's face/shirt-logo (has some motion blur) and (b) a fully static background object (water bottles, zero motion blur — a clean interpolation-only test). Both regions showed bicubic and lanczos as visually indistinguishable; no perceptible sharpening on high-contrast detail (bottle-straw stripes, metal-bar highlights). **Speed result: real, confirmed regression — corrects the original "noise" cost caveat.** `render_s` 181.5s -> 259.8s (+43%), `total_s` 214s -> 294s (+37%) on the 2-clip test. The original "~20% overhead, but `t_zoom_s` is <1% of total so it's noise" assumption was **wrong on its own terms**: the lanczos cost does not show up in the isolated `zoom_s` metric (stayed ~0.4s both runs) — it's inside the main filter_complex encode (`render_s`), which is NOT a small fraction of total render time. **Verdict (rushcut-pp-consultant, wrap-readiness round):** same shape as hypothesis #1/#124 — confirmed real speed cost, zero confirmed quality benefit. Reverted same session; `pipeline/zoom.py` is back to bicubic. **Before testing hypothesis #4 (unsharp), one cheap sanity check is recommended first: does the softness scale with zoom depth, or is it present even near 1.0x zoom?** If present even at minimal zoom, the cause is more likely upstream of the zoom filter entirely (proxy source resolution, normalise scale filter) than anything fixable by tuning the zoom-stage interpolation/sharpening — not yet run, flagged as the next step.
3. **#102 (OPEN, correctness bug) — test-integrity blocker, fix before running #1/#2 A/Bs on any project with an intro card.** Intro card + per-clip zoom together silently misapply zoom to the wrong clip (off-by-one in `zoom_vfs` vs. `current_paths` after card insertion) and drop zoom from the last real clip entirely. This isn't just a feature bug — any zoom-quality A/B run on a card-using project would silently compare the wrong clips' zoom output, corrupting the result with no obvious symptom. Ranked ahead of the sharpen hypothesis below for exactly this reason: cheap to repro/fix, and removes a hidden confound from every A/B that follows it.
4. **[NEW, untested]** Post-zoom `unsharp` mask as a counter-measure/complement to #1/#2 — a common upscale-then-sharpen technique, never tried here. Needs its own A/B; risk of looking artificial (halo/ringing) if overdone.
5. **#103 (OPEN, tech debt)** — the parallel-array pattern (`current_paths`/`clip_volumes`/`zoom_vfs`) underlying #102. Worth fixing alongside #102 rather than patching the symptom again before mid-roll cards are ever built.
6. **#65 (OPEN, likely stale — verify before treating as live work)** — "raise bitrate within WebView2 limit + eliminate full-film double-encode for open/close." Lever 2 (double-encode) already shipped as boundary-segment-only re-encode (see `.claude/rules/pipeline.md` U1g section). Lever 1 (bitrate cap) is very likely superseded by the 2026-07-12 CQP switch, which replaced the bitrate-tier constants entirely. Action: confirm with a fresh render + log check, then probably close as superseded rather than picking it back up.
7. **#95 (OPEN, feature)** — Slow Pan: gradual directional pan across a clip's full duration, no zoom. A new motion type, not a fix to existing motion quality. Low priority per the issue itself.
8. **#72 (OPEN, feature)** — same-clip zoom state transition (intra-clip dual-keyframe crossfade). Another motion-quality feature, not yet scoped.
9. **#113 (OPEN, speed spike, not a quality fix itself)** — `-hwaccel d3d11va` GPU decode probe for U1g's Windows-native ffmpeg.exe inputs. Relevant here only as a *speed-budget enabler*: if it frees real CPU headroom, that headroom could pay for a slower/higher-quality scale filter (lanczos) or an unsharp pass without touching the 5-min ceiling. Sequence after the direct zoom-quality items above, not instead of them.
10. **#109 (OPEN)** — hevc_amf for proxy encoding. Not a quality lever on its own (same QP tier, smaller file), but if it's ever picked up, that's the natural moment to also revisit the QP30 constant — halved bitrate need at a fixed QP could allow lowering QP for a similar file size, directly helping hypothesis #1.

## Learning log (brief — tried, success/fail — don't re-propose without new evidence)

- **#49/#78 (closed, GO)** — 4K AMF bitrate cap raised in steps within the WebView2 crash ceiling (#5). Superseded 2026-07-12 by the VBR→CQP switch. Real, TV-checked win for non-zoomed footage — this is *why* zoomed footage now looks soft by comparison (#116).
- **#84 (closed, REJECTED)** — AMF `-quality "balanced"` at the (now-superseded) 40M/50M bitrate tier: zero speed gain, visible quality loss. Tested under a bitrate-starved-RDO constraint that may no longer hold now that the encode runs CQP instead of VBR. Don't re-propose blind, but a narrow retest under CQP is defensible if CQP-era quality still isn't sufficient.
- **#64/#86 (closed)** — AMF reliability fixes (yuv420p→yuv444p swscaler negotiation, WSL UNC-path permission denial). Not quality levers themselves, but prerequisite correctness fixes — without them, AMF renders silently fell back to libx264, making any quality A/B untrustworthy.
- **#65 Lever 2 / boundary-segment-only re-encode (shipped)** — replaced the full-film double-encode for open/close transitions. Real win, already live — don't re-propose the whole-film re-encode fix.
- **#66/#67/#79 (closed, shipped)** — embed-zoom-in-filter_complex, replacing the old pre-encode+cache model. Landed at ~zero render-time cost (`t_zoom_s` <1% of total). **Correction (2026-07-14, #116 lanczos test):** this established the *graph shape* (embed vs pre-encode) is cheap — it does NOT mean every filter tweak inside that graph is free. The `t_zoom_s` metric only measures a separate timing checkpoint; a filter change that alters the cost of the main filter_complex encode itself (like adding `flags=lanczos` to the scale) shows up in `render_s` instead and can be a real, non-trivial cost (+43% confirmed). Don't reuse "zoom graph tweaks are cheap" as a blanket assumption for future filter changes — measure `render_s`, not just `zoom_s`.
- **#116 lanczos scale filter (2026-07-14, tested and REVERTED)** — see hypothesis #2 above for the full write-up. No visible quality benefit, +43% `render_s` regression. Don't re-propose without a different methodology or new evidence.
- **#101 (closed, GO/keep-as-is)** — 2160p (not tiered) proxy resolution validated for speed/reuse amortization. Scope was speed only — it never evaluated the QP30-as-final-source-quality tradeoff, so it is **not** evidence against hypothesis #1 above.
- **#92 (closed)** — proxy-gen `-hwaccel d3d11va` investigation. About proxy-gen *speed*, not proxy *quality* — orthogonal to this doc.
- **#110 (closed, shipped)** — hevc_amf for final render. A file-size win at equivalent visual quality (not a quality lever itself); real pipeline A/B showed ~50% smaller output, same visual result.
