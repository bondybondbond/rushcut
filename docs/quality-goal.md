# Quality Goal

Transparency doc — not a strategy doc. Tracks progress toward the visual-quality north star so we don't re-propose rejected ideas or lose sight of the real bottleneck. Companion to [`docs/speed-goal.md`](speed-goal.md) — the two north stars are in tension (higher quality tends to cost render time) and any quality fix proposed here must respect the speed doc's ceiling, not quietly blow past it.

## North star

- **Close the gap to DaVinci Resolve's visual output quality**, with zoom/pan (Ken Burns) as the current headline gap — non-zoomed footage is already close after the 2026-07-12 CQP switch (see Current state below).
- **Hard constraint: no severe render-time regression.** Any fix here must stay inside `docs/speed-goal.md`'s 5-min cold-render ceiling. A quality win that blows the speed budget is not a win — it needs to be scoped as a toggle/opt-in, not a default.
- **Make it testable, not vibes-based.** "Closing the gap" = passing a blind side-by-side TV-check against a DaVinci Resolve export of the *same project*, at 2160p, specifically on zoomed/panned footage. Same TV-check discipline already used for the CQP switch and the #85/#110 encoder work — just anchored against an external DaVinci reference instead of an internal before/after.

## Current state (as of 2026-07-12)

- **Non-zoomed 4K quality: materially improved.** The launch-plan #1.1/#1.2 switch from AMF VBR (40M/50M) to CQP (`qp_i=18/qp_p=20`) was TV-checked and confirmed sharper on non-zoomed pan-heavy footage.
- **Zoomed (Ken Burns) footage: now looks soft *by comparison*.** Filed as #116. Not yet established whether this is a regression the CQP change exposed (rest of the frame got sharper, so the zoomed region's pre-existing softness now stands out) or something that got worse. That's the first thing any #116 work should establish.

## Outstanding hypotheses / relevant open issues (ranked)

1. **[NEW, untested, not yet filed] Proxy QP30 as final 4K render source.** Background proxies (`generate_proxy_file_low_priority`/`_normal_priority` in `src-tauri/src/lib.rs`) encode 2160p at `-rc cqp -qp_i 30 -qp_p 30` — a speed-tuned setting meant for scrubbing preview, not delivery. The render.py proxy-reuse gate (`.claude/rules/pipeline.md`) accepts any 2160p+ proxy as a full substitute for normalise on **any** 4K render, zoomed or not — so whenever proxies are warm (the common case), the *entire film's* final-render source is already QP30-compressed before zoom or the final encode ever runs. Zoom then re-magnifies that compression on top, which is very plausibly the dominant contributor to #116. **Architectural framing:** the general NLE convention (DaVinci, Premiere, and others) is proxy-for-preview / full-resolution-source-for-final-render — final delivery is never gated by a preview-tier proxy. RushCut's gate treating a scrubbing-tier proxy as valid final-render input is a structural divergence from that convention, not just a tuning knob. This reframes it from "one more bitrate lever" to the most likely single biggest gap vs. the DaVinci north star. **Cheap test:** force one known-soft clip (e.g. the #49/#65/#78 "clip-4, not even zoomed" case) through fresh normalise instead of proxy reuse, re-render at identical CQP settings, A/B. Would also explain why #49/#65/#78's bitrate/CQP work only partly fixed clip-4.
2. **#116 (OPEN)** — Ken Burns output looks soft. Verified directly against this machine's real FFmpeg build (`ffmpeg -h filter=scale`): the `scale` filter's own `flags` option defaults to empty string, which falls through to the global `-sws_flags` default — confirmed **bicubic**, not bilinear (`-sws_flags <flags> ... (default bicubic)`). No code in `pipeline/` ever sets `-sws_flags` or `flags=` on any `scale=` call, so `zoom.py`'s Ken Burns per-frame scale (`zoom.py:147`) and static crop-then-upscale path (`zoom.py:193`/`264`) are genuinely running on FFmpeg's bicubic default today — not an unverified/implicit choice, and not bilinear. A/B an explicit `flags=lanczos` against that confirmed bicubic baseline. **Cost caveat:** lanczos is measurably slower than bicubic (commonly cited ~20% overhead on that specific operation) — but `t_zoom_s` is already <1% of total render time, so a 20% hit on that one stage is noise against the 5-min ceiling. That's "~20% overhead on the zoom stage specifically," not "near-zero cost in general" — don't misapply this reasoning to a filter on the hot path (e.g. the canvas pre-scale that runs on every clip).
3. **#102 (OPEN, correctness bug) — test-integrity blocker, fix before running #1/#2 A/Bs on any project with an intro card.** Intro card + per-clip zoom together silently misapply zoom to the wrong clip (off-by-one in `zoom_vfs` vs. `current_paths` after card insertion) and drop zoom from the last real clip entirely. This isn't just a feature bug — any zoom-quality A/B run on a card-using project would silently compare the wrong clips' zoom output, corrupting the result with no obvious symptom. Ranked ahead of the sharpen hypothesis below for exactly this reason: cheap to repro/fix, and removes a hidden confound from every A/B that follows it.
4. **[NEW, untested]** Post-zoom `unsharp` mask as a counter-measure/complement to #1/#2 — a common upscale-then-sharpen technique, never tried here. Needs its own A/B; risk of looking artificial (halo/ringing) if overdone.
5. **#103 (OPEN, tech debt)** — the parallel-array pattern (`current_paths`/`clip_volumes`/`zoom_vfs`) underlying #102. Worth fixing alongside #102 rather than patching the symptom again before mid-roll cards are ever built.
6. **#65 (OPEN, likely stale — verify before treating as live work)** — "raise bitrate within WebView2 limit + eliminate full-film double-encode for open/close." Lever 2 (double-encode) already shipped as boundary-segment-only re-encode (see `.claude/rules/pipeline.md` U1g section). Lever 1 (bitrate cap) is very likely superseded by the 2026-07-12 CQP switch, which replaced the bitrate-tier constants entirely. Action: confirm with a fresh render + log check, then probably close as superseded rather than picking it back up.
7. **#95 (OPEN, feature)** — Slow Pan: gradual directional pan across a clip's full duration, no zoom. A new motion type, not a fix to existing motion quality. Low priority per the issue itself.
8. **#72 (OPEN, feature)** — same-clip zoom state transition (intra-clip dual-keyframe crossfade). Another motion-quality feature, not yet scoped.
9. **#113 (OPEN, speed spike, not a quality fix itself)** — `-hwaccel d3d11va` GPU decode probe for U1g's Windows-native ffmpeg.exe inputs. Relevant here only as a *speed-budget enabler*: if it frees real CPU headroom, that headroom could pay for a slower/higher-quality scale filter (lanczos) or an unsharp pass without touching the 5-min ceiling. Sequence after the direct zoom-quality items above, not instead of them.
10. **#109 (OPEN)** — hevc_amf for proxy encoding. Not a quality lever on its own (same QP tier, smaller file), but if it's ever picked up, that's the natural moment to also revisit the QP30 constant — halved bitrate need at a fixed QP could allow lowering QP for a similar file size, directly helping hypothesis #1.

## Learning log (brief — tried, success/fail — don't re-propose without new evidence)

- **#49/#78 (closed, GO)** — 4K AMF bitrate cap raised in steps within the WebView2 crash ceiling (#5). Superseded 2026-07-12 by the VBR→CQP switch. Real, TV-checked win for non-zoomed footage — this is *why* zoomed footage now looks soft by comparison (#116).
- **#84 (closed, REJECTED)** — AMF `-quality "balanced"` at the (now-superseded) 40M/50M bitrate tier: zero speed gain, visible quality loss. Tested under a bitrate-starved-RDO constraint that may no longer hold now that the encode runs CQP instead of VBR. Don't re-propose blind, but a narrow retest under CQP is defensible if CQP-era quality still isn't sufficient.
- **#64/#86 (closed)** — AMF reliability fixes (yuv420p→yuv444p swscaler negotiation, WSL UNC-path permission denial). Not quality levers themselves, but prerequisite correctness fixes — without them, AMF renders silently fell back to libx264, making any quality A/B untrustworthy.
- **#65 Lever 2 / boundary-segment-only re-encode (shipped)** — replaced the full-film double-encode for open/close transitions. Real win, already live — don't re-propose the whole-film re-encode fix.
- **#66/#67/#79 (closed, shipped)** — embed-zoom-in-filter_complex, replacing the old pre-encode+cache model. Landed at ~zero render-time cost (`t_zoom_s` <1% of total). Establishes that iterating on the zoom filter graph itself (lanczos, unsharp) is cheap from the speed budget's perspective — the graph shape isn't the expensive part.
- **#101 (closed, GO/keep-as-is)** — 2160p (not tiered) proxy resolution validated for speed/reuse amortization. Scope was speed only — it never evaluated the QP30-as-final-source-quality tradeoff, so it is **not** evidence against hypothesis #1 above.
- **#92 (closed)** — proxy-gen `-hwaccel d3d11va` investigation. About proxy-gen *speed*, not proxy *quality* — orthogonal to this doc.
- **#110 (closed, shipped)** — hevc_amf for final render. A file-size win at equivalent visual quality (not a quality lever itself); real pipeline A/B showed ~50% smaller output, same visual result.
