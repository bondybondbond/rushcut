# Pipeline — invocation & FFmpeg rules

Applies when working on `pipeline/**`, `src-tauri/src/lib.rs` (pipeline spawn), or `src/pages/Output.tsx`.

## Proxy pipeline rules

- **`generate_proxy()` timeout must be 600s** — 120s is too short for 4K HEVC clips > 60s at software decode. FFmpeg killed by timeout leaves a corrupt partial file.
- **Validate existing proxies before skipping re-encode** — `Path(proxy_wsl).exists()` is not sufficient. Run `ffprobe -v quiet -show_format <file>` and check `returncode == 0`. A missing moov atom (FFmpeg killed mid-write) makes the file unplayable in WebView2 but `Path.exists()` still returns True. See `is_valid_proxy()` in `proxy.py`.
- **proxy.py manifest /tmp path must be a WSL path** — writing the manifest to Git Bash `/tmp/file.json` makes it invisible to WSL (`/tmp/` is a different filesystem). Always write manifests to a Windows path (`%TEMP%\rushcut\*.json`) and reference as `/mnt/c/Users/.../AppData/Local/Temp/rushcut/file.json` in WSL.
- **Proxy re-encode order: thumbnail → waveform → encode** — extract thumbnail and waveform from source first (fast: ~2s each), then run the slow H.264 encode. This ensures the TrimBar gets visual data within ~5s of proxy gen starting, regardless of how long the encode takes.
- **Proxy spec for normalise-compatible intermediates:** To use a proxy as a normalise substitute (skipping the 80s HEVC decode entirely on re-renders), it must match normalise output exactly: `-map 0:v:0 -map 0:a:0? -vf scale=-2:1080,format=yuv420p -r <native_fps> -fps_mode cfr -c:v libx264 -preset ultrafast -crf 23 -c:a aac -b:a 128k -ar 48000`. **Use the source clip's native fps, not `25`** — see Batch Q2. Missing `-c:a copy` passes 96kHz DJI audio through (bug); missing fps match causes 25fps proxy to be rejected by the render gate.
- **Proxy reuse gate checks BOTH height AND fps** — `_proxy_meta(proxy_wsl)` in `render.py` returns `(height, fps_int)`. A proxy is accepted only when `height >= required_proxy_h AND fps_int == target_fps_int`. Legacy 25fps proxies are automatically rejected when target_fps_int=30 (DJI 29.97 clips) and re-normalised. `_proxy_height()` no longer exists — use `_proxy_meta()`.
- **`target_fps_raw` vs `target_fps_int` naming convention** — `target_fps_raw` is the ffprobe rational string (`"30000/1001"`) passed directly to FFmpeg `-r`. `target_fps_int` is the rounded standard int (30) used ONLY for proxy comparison and log messages. Never pass `target_fps_int` to FFmpeg — the rational preserves the exact DJI framerate.
- **B-0 trim offset mutation — proxy clips need original `in_ms`/`out_ms` restored** — `_pretrim_worker` in `render.py` runs for ALL clips and rewrites `pipeline_clips[i]["in_ms"]`/`["out_ms"]` to be relative to the pre-trim window start (e.g. `15.569s → 2.0s` after subtracting `a_start = in_s - 2`). For clips that use a proxy (full-duration file), the Step 2 trim must apply the ORIGINAL absolute offsets, not the B-0-relative ones. Fix in `render.py` proxy loop: `pipeline_clips[i] = {**pipeline_clips[i], "in_ms": clips[i].get("in_ms"), "out_ms": clips[i].get("out_ms")}`. Without this, proxy clips are trimmed from the wrong segment (correct duration, wrong content).
- **`proxy_path` must be included in `start_job` manifest** — the Rust `start_job` function builds the clip JSON; `proxy_path` must be explicitly added (`"proxy_path": c.proxy_path`) or `run.py` always sees `null` and all clips fall through to the normalise path on every render.
- **Proxy height gate in `render.py` must be resolution-aware** — `height >= 1080` is correct for 1080p output, but a 1080p proxy used for a 4K render upscales from 1080p → 2160p and degrades quality. Use `required_proxy_h = 2160 if output_resolution == "4k" else 1080`; reject proxies below this threshold and fall through to normalise. A 2160p proxy qualifies for both 1080p and 4K renders.
- **Background proxy gen (`generate_proxy_file_low_priority`) must encode at 2160p** — encoding at `scale=-2:1080` permanently limits the proxy to 1080p reuse only. Always use `scale=-2:2160`; the extra encode cost (~20% more than 1080p at ultrafast) is negligible compared to the bug it prevents. `_proxy_height()` in `render.py` distinguishes old 1080p proxies from new 2160p ones and logs accordingly.
- **`get_clips_needing_bg_proxy` must return ALL `include=1` clips, not only `proxy_status IS NULL`** — filtering by `proxy_status != 'done'` prevents upgrading legacy 1080p proxies to 2160p. The Rust `run_bg_proxy_batch` function calls `proxy_height_native()` per clip and decides: skip (≥2160p), upgrade (existing but <2160p), or encode fresh. Height check in Rust is authoritative; DB filter is only for `include=1`.
- **h264_amf ignores `-color_primaries` — always writes `prim:reserved` regardless.** `-colorspace bt709` and `-color_trc bt709` ARE honored but `-color_primaries bt709` is silently ignored by the AMD AMF SDK. Confirmed via ffprobe: `color_space=bt709`, `color_transfer=bt709`, `color_primaries=reserved` after encoding with all three flags. Do NOT attempt to fix `prim:reserved` at the encode site — fix it in-graph with `setparams` (see U1g rule below). The three flags are still kept in lib.rs proxy gen for colorspace/trc correctness, but `color_primaries` is a no-op there.
- **U1g canvas chain MUST include `setparams=color_primaries=bt709:color_trc=bt709:colorspace=bt709` after `format=yuv420p` on every `[sv{i}]` node** — `format=yuv420p` pins pixel format but does NOT strip color metadata from frames. Swscaler still sees `prim:reserved` (from AMF-encoded proxy inputs) and tries to negotiate `yuv444p` for downstream AMF → `swscaler -129` / exit 127 on the segment encode → libx264 fallback → mixed-encoder concat → green frames / WMP freeze at `-c copy` boundary. `setparams` overrides the `prim:reserved` tag to bt709 at the filtergraph level before xfade sees it. Both `build_batch_video_fc` and `build_filter_complex` canvas strings in `transitions.py` must have this suffix. Also add it to `zoom.py` `apply_zoom()` so zoom-cache files don't carry inherited `prim:reserved` into U1g.

- **FPS stutter on constant-motion clips — FIXED (Batch Q2):** Pipeline no longer hardcodes `-r 25`. `render.py` probes `target_fps_raw` from `clip_paths[0]` via `_probe_fps()` at Step 1 start; passes to `normalise()`, `make_card()`. Rust proxy gen (`generate_proxy_file_low_priority`, `generate_proxy_file`) uses `probe_clip_fps()` for native fps. Output is `30000/1001` for DJI 29.97fps source. Legacy 25fps proxies are rejected by the reuse gate (fps_int mismatch) and re-encoded on next bg pass.

## AMF encoder fallback — silent-fallback detection pattern

- **`_run_with_amf_fallback()` pattern in `render.py`:** When `use_amf=True` and an AMF encode raises `RuntimeError`, retry with libx264 via a fallback command function. Track the fallback state with a list-closure (`amf_fallback_flag = [False]`) rather than a plain bool (lists are mutable inside nested functions in Python). Append `amf_fallback=0/1` to the ANALYSIS stdout line at pipeline end.
- **Any log/label that reports fallback status must read `amf_fallback_flag[0]`, never `amf_fallback_flag` bare** — a non-empty list is always truthy in Python regardless of its contents, so `if amf_fallback_flag:` reports "fallback happened" on every render, success or not. #69 was the opposite bug (the U1g segmented-render completion log at `render.py:1131` hardcoded "(no fallback)" and never read the flag at all); fixed by reading `amf_fallback_flag[0]` to pick the label, same as the existing `ANALYSIS:` line does.
- **`pipeline-done` event `analysis` field contract:** `src-tauri/src/lib.rs` captures the last `ANALYSIS:` stdout line per job into `last_analysis: Option<String>`. The `pipeline-done` Tauri event payload now includes `"analysis": last_analysis` alongside `jobId`, `outputPath`. React listeners: `listen<PipelineProgressEvent & { analysis?: string | null }>("pipeline-done", ...)`. Parse with regex: `/(^|,)key=value(,|$)/.test(analysis)` to extract fields without a full CSV parser.
- **Do NOT surface AMF fallback as an error** — the render succeeded; user just got libx264 quality instead of AMF. Toast is correct UX (informational, not blocking). The `amf_fallback=1` toast reads: "GPU encode unavailable -- rendered on CPU (standard quality)" (6s auto-dismiss; U4e reworded from the old "Fast render unavailable" copy now that there's no toggle).
- **AMF only works when the pipeline is spawned by Rust/Tauri — NOT from a manual `wsl ... python3 run.py` script.** When `run.py` is invoked from a PowerShell-spawned WSL context (e.g. a one-off comparison/benchmark script), the Windows `ffmpeg.exe` AMF encode fails with **exit 127 ("command not found")** and `_run_with_amf_fallback()` silently drops to libx264. The WSL→Windows-ffmpeg.exe interop bridge that AMF needs is only set up in the full Tauri spawn environment. Consequence: manual renders launched this way always produce libx264 output regardless of resolution. This is fine for a **bitrate comparison** (codec is identical across the runs) but useless for verifying AMF itself — to confirm AMF, render through the app (Rust spawn) and grep the job log for `encoder=h264_amf` with `amf_fallback=0`.
- **Resolution-adaptive final bitrate + AMF quality flags (#49/#78, shipped 2026-07-06).** `pipeline/encoder.py` now has two bitrate tiers: `FINAL_BITRATE="20M"`/`AMF_MAXRATE="24M"` (1080p) and `FINAL_BITRATE_4K="40M"`/`AMF_MAXRATE_4K="50M"` (4K). `video_encoder_args()`'s AMF final branch selects the pair via `output_resolution == "4k"`. The two AMF quality flags `-vbaq true` and `-high_motion_quality_boost_enable true` (VBAQ + high-motion-pan boost, both hardware-side/~zero render-time cost on ffmpeg 8.0.1) are **gated to 4K-only** (an `is_4k` check, added AFTER the bitrate select) — do not lift this gate without re-checking: a first implementation pass applied them unconditionally, which silently changed the rare 1080p `RUSHCUT_USE_AMF` opt-in path too, caught by `rushcut-qa-reviewer`. `pipeline/render.py`'s boundary re-encode `_boundary_codec_args()` (added #65, replaces the deleted `_apply_open_close_post`'s `_pp_codec()`) imports and shares the same 4K constants + the same 4K-only gate — single source of truth for the 4K final bitrate across both encode sites. Default 1080p (libx264) path is untouched at 20M.
- **`render-timing-log.jsonl`'s `t_render_s` bundles the main U1g encode AND the open/close post-pass into one number — no separate breakdown exists.** Confounds any A/B render-time comparison where one render has open/close enabled and the other doesn't: a `t_render_s` delta could be from a bitrate change, the post-pass's extra full-length encode pass, or both. Before attributing a time delta to a single variable (bitrate, resolution, etc.), confirm both renders being compared have identical `has_open`/`has_close` state — or isolate one variable at a time. See issue #88 (deferred) for a case where this confound blocked a clean conclusion.
- **#84 (closed, tested 2026-07-10): AMF `-quality "balanced"` gives zero render-time benefit at the current 4K bitrate (40M/50M) and produces visibly worse output — `quality` stays.** Real A/B on the Stagecoach 2025 project (19 clips, 4K, crossfade + dip-to-black close): baseline `quality` `t_render_s=483.0`, test `balanced` `t_render_s=486.0` — noise, not a real delta. Visual: user confirmed clear degradation on the pan-heavy ~13s section (same clip flagged in #65). Likely explanation (not re-verified, don't chase further): at this bitrate the AMF encoder is bitrate-constrained rather than RDO-effort-constrained, so the `speed`/`balanced`/`quality` preset gap collapses — `-quality` isn't a lever worth revisiting at 40M+ without also changing bitrate. Do not re-test `balanced`/`speed` at the current bitrate tier without new evidence the bottleneck has changed.
- **RESOLVED by #65 (2026-07-11):** the old `_pp_codec()`'s duplicate `-quality "quality"` literal (tracked as #105, found while testing #84) is moot — the whole-film `_apply_open_close_post()` function it lived in was deleted as part of #65's boundary-segment-only re-encode fix, not patched. The sole remaining codec-selection site is `_boundary_codec_args()` in `_render_segmented()`. #105 closed as resolved-by-deletion.

## Concurrent renders and log isolation

- **Exit code 15 = SIGTERM from WSL restart** — not a code bug. Check `wsl -- ls /tmp/<job_id>` before debugging; a missing dir means WSL killed everything (memory pressure or a concurrent heavy render). Re-run once WSL is stable.
- **U1g working directory must be NTFS, not WSL `/tmp`** — `_render_segmented()` uses `TMP_BASE = Path("/tmp")`. Under memory pressure from two sequential 4K encodes, WSL tmpfs can clear `/tmp/<job_id>/` between segment-batch completion and the concat-manifest write. `write_text()` then raises `[Errno 2]`; the fallback catches it and runs monolithic — defeating the whole purpose of U1g. Fix (Batch U4c): use `Path(os.environ["TEMP"]) / "rushcut" / str(job_id)` (NTFS). Segment paths in the concat file need the `/mnt/c/...` prefix. Same pattern as zoom-cache and `warm_zoom.py`.
- **AMF output targets (not just U1g) must be NTFS-anchored via the manifest path, not `USERPROFILE` (#86, fixed 2026-07-06)** — `_resolve_render_work_dir()`'s `USERPROFILE`-derived NTFS path is **not inherited into the WSL environment** from Rust's bare `wsl -d Ubuntu-24.04 -u root -- python3 run.py` spawn (no `WSLENV`/env forwarding), so it silently fell through to a `/tmp/rushcut-render/<job_id>` path. `to_win_path()` translates that to `\\wsl.localhost\Ubuntu-24.04\tmp\...` — a UNC share that Windows-native `ffmpeg.exe` (AMF) **cannot write to** ("Permission denied"), causing every AMF encode (single-clip, monolithic, U1g segments, U1g open/close post-pass) to silently fall back to libx264. Confirmed: this broke ALL AMF-quality verification since the 2026-07-04 WSL reinstall, not just U1g. **Fix:** `run.py` derives `ntfs_tmp_base = str(manifest_path.parent)` — the manifest is always written by Rust to `%TEMP%\rushcut\<job_id>.json`, so its parent is a guaranteed-existing `/mnt/c` path with zero env dependency — and passes it through `job["config"]["ntfs_tmp_base"]`. `_resolve_render_work_dir(job_id, base=None)` now prefers this explicit `base` over the old env lookup (which remains only as a legacy/test-env fallback). All AMF output targets (`output = tmp / "render.mp4"` at both call sites, and U1g's `seg_tmp`) were repointed at the new `render_work` dir; normalise/pre-trim/audio-treatment intermediates stay on tmpfs `tmp` (reads over UNC work fine — only writes fail). `run.py` cleanup now removes `render_work` (via the same `_resolve_render_work_dir` call, never hand-reconstructed) alongside the existing `/tmp/<job_id>` rmtree, so NTFS scratch doesn't accumulate. Verified via real 4K 9-clip xfade render: `encoder=h264_amf` confirmed by ffprobe, `amf_fallback=0` on all 3 U1g segments, `drift=0`. **Same `USERPROFILE` gap exists in `render_cache.py` `cache_dir()`** — not fixed as part of #86 (scoped out to keep the fix atomic); tracked on #81/#82.
- **`pipeline-latest.log` must be a symlink, not a shared file** — two concurrent `run.py` processes opening the same file with `mode="w"` produce a sparse file with null-byte gaps (second process truncates but retains old offset). Pattern: each job writes to `pipeline-{job_id}.log`; update `pipeline-latest.log` symlink atomically: `latest.unlink(missing_ok=True); latest.symlink_to(f"pipeline-{job_id}.log")`.

## Sync / performance fixes — logs first

Before writing any A/V sync fix or normalise speed fix, run a real render and read the `[sync-check]` log output. Identify WHERE drift enters (normalise output? post-trim? concat?) before touching code. `aresample=async` worsens DJI monotonic drift — see LEARNINGS.md.

**Normalise performance (post-Batch C):** render.py pre-trims each clip to `[in_s - 2s, out_s + 0.5s]` via FFmpeg copy before calling `normalise()`. Files land in WSL2 `/tmp` (tmpfs = RAM). `normalise()` uses `ThreadPoolExecutor(max_workers=min(4, os.cpu_count()))` with per-worker `-threads N` flag. Result: 10 min → ~3 min for a 1m26s 10-clip DJI 4K film. **Batch C proxy reuse:** on re-renders where 1080p H.264 proxies exist, normalise is skipped entirely — `TIMING:normalise=` drops from ~45s to ~2s (proxy_skip=N/N). The B-0 pre-trim still runs for all clips (fast copy) but normalise only runs for clips without a valid 1080p proxy. First renders are unchanged. The I/O-bound warning still applies to pre-trim (reads from NTFS): parallelising pre-trim does not improve total time.

**Music volume ducking (post-Batch B):** `music.py` `_build_filter()` now applies `[0:a]volume={movie_vol}[movaudio]` to the movie audio stream before amix. `movie_vol` is preset-driven: `{subtle: 1.0, balanced: 0.7, prominent: 0.3}`. The `mix_music()` signature takes `movie_vol: float = 1.0`. Callers in `render.py` pass `movie_vol = _MOVIE_VOL.get(round(music_volume, 1), 0.7)`.

## Segmented xfade render (U1g)

- **`_render_segmented()` activates when `len(current_paths) > BATCH_SIZE and has_xfade`** — batches of ≤4 clips (overlap-by-one), single audio pass, lossless `-c copy` concat + mux. **Open/close-to-black (`has_open`/`has_close`) is supported via BOUNDARY-SEGMENT-ONLY re-encode (#65, shipped 2026-07-11 — replaces issue #31's original whole-film `_apply_open_close_post()`, deleted):** `_boundary_reencode()` re-encodes ONLY `seg_files[0]` (open) and/or `seg_files[-1]` (close) with the fade baked in via `build_open_close_post_fc()` (video-only, `has_audio=False`), writing sibling files that get swapped into `seg_files` BEFORE the lossless concat — the middle segments and the concat itself never see a second encode generation. Audio open/close (acrossfade-to-silence, unchanged from #31) runs separately via `build_open_close_audio_fc()` on the cheap whole-project `audio_full` track, fully decoupled from video. **Encoder-consistency guard:** each boundary re-encode's `force_libx264` is derived from `encoder_outcome_by_idx[idx]` (per-batch AMF/libx264 outcome, captured during the batch loop) so it matches the SAME encoder its own segment already used — preventing a mixed-encoder `-c copy` concat (#64 green frames). A `log.warning` fires if a boundary re-encode's actual outcome still diverges (stale/mismatched `encoder_outcome_by_idx`). Guardrail: `_probe_frame_count(output)` vs `total_frames_expected + 0.1*has_open + 0.1*has_close` (±1 frame), using the planned pre-open/close total rather than probing an intermediate (there's no single "inner" file in this path). The remaining monolithic-only cases: small projects (≤ BATCH_SIZE), the `none`/concat path (no `has_xfade`), or any segmented failure (planner ValueError / encode error → caught → monolithic).
- **`BATCH_SIZE = 4` is the safe ceiling for 4K at 12 GB WSL** — actual peak for one 4K batch: ~6–9.7 GB (4 decoders + xfade buffers + libx264 medium). Do NOT raise to 5 or more without measuring peak RSS first.
- **Frame-count telescoping prevents drift** — batch frame count = `round(g_end * fps_f) - round(g_start * fps_f)` using the global frame grid. Using `round(batch_duration * fps_f)` instead causes progressive drift: each batch adds one sub-frame rounding error that accumulates across boundaries. Confirmed: `drift=0 frame(s)` on a 21-clip 4K render.
- **In-filter trim, not `-c copy` segment trim** — clips are trimmed inside `filter_complex` using `trim=start={t:.4f},setpts=PTS-STARTPTS`. Segment trimming with `-c copy` snaps to the nearest keyframe and can drift by whole seconds. This is NOT U1g-only — the ordinary Step 2 per-clip trim (`pipeline/trim.py`'s `trim()`) used to have the identical `-ss/-to -c copy` keyframe-snap issue (confirmed #96, 2026-07-09: a clip trimmed to a requested 10.236s window produced 10.644s of actual playable content — `nb_frames`/`r_frame_rate` proof — because the start didn't land on a keyframe).
- **FIXED (#96, 2026-07-10): `pipeline/trim.py`'s Step 2 trim now re-encodes instead of stream-copying** — `-c:v libx264 -preset ultrafast -crf 23 -c:a aac -b:a 128k -ar 48000`, keeping `-ss` before `-i` (FFmpeg's default `-accurate_seek` gives frame accuracy during transcode, no filter_complex rewrite needed). Uses `-t` (duration) instead of `-to` (end position) — `-to` is nominally input-relative when placed before `-i`, but `-t` avoids any ambiguity from output-timestamp-reset behavior some FFmpeg builds apply once transcoding (vs. copy) is involved. Verified live: clip 3 nb_frames 319→307 (exact match to `round(duration*fps)`), clip 5 518→490, `film_s` 62.1→60.8 on the "92 test" 4K project.
  - **Cost tradeoff: re-encoding Step 2 is ~10x slower than the old stream copy** — `t_trim_s` went from 3.0s to 31.0s for 6 clips at 4K/ultrafast (measured via `render-timing-log.jsonl`). Expected and accepted: `trim()` always runs on an already-lossy H.264 intermediate (normalise output or proxy), so a second re-encode generation is the same "intermediates re-encoded by render step, quality irrelevant" tradeoff `normalise.py` already makes. Don't be alarmed seeing `t_trim_s` in the 20-40s range on 4K renders going forward — that's the new normal, not a regression.
  - **#99 (2026-07-10): Step 2 trim jobs now run in parallel** (`ThreadPoolExecutor`, `MAX_PARALLEL_TRIM = min(4, os.cpu_count())`, mirroring `normalise.py`; `trim()` gained an optional `threads` param for the per-worker `-threads` cap). Delivers only ~1.1-1.3x in practice, not the `normalise.py`-precedent 3-4x — per-job time is dominated by a ~7-8s fixed FFmpeg-invocation floor that doesn't shrink with more workers/threads. See the LEARNINGS.md "parallelising short FFmpeg re-encode jobs" entry for the full investigation (two hypotheses tested and disconfirmed: CPU thread cap, WSL `/mnt/c` I/O contention). Real fix tracked as a separate, bigger issue (avoid the full re-encode — partial-GOP trim), not a parallelism tweak.
- **Boundary cut placement: inside the shared clip's solo region** — the clip that bridges two batches (overlap clip) appears in both. The concat cut point must land inside the solo region: `[offset[k]+xfade_dur, offset[k+1]]`. A cut outside this region overlaps two simultaneous xfades and corrupts the transition.
- **Mixed-encoder `-c copy` concat causes green frames / decoder freeze at segment boundaries** — when some U1g segments encode with AMF (`h264_amf`) and others fall back to libx264 (due to AMF `yuv444p` negotiation failure), the SPS/PPS headers differ across segments. A `-c copy` concat of mismatched encoders causes the decoder to produce green frames (YCbCr zero-fill ≈ RGB 0,255,0) or a freeze at the first cross-encoder boundary. WMP freezes at the boundary; scrubbing past and rewinding resets the decoder. **This is #64 (AMF reliability) — not a transition bug.** Isolate: extract the segment file at that timestamp; if it plays clean, the concat is the cause, not the xfade. **Two-level fix required:** (1) A terminal `[vpix]format=yuv420p[vout]` node (shipped #64, commit 07413cf) locks the buffersink pixel format — eliminates the *pixel-format-driven* yuv444p request and fixed segs with matched color metadata. (2) Per-node `format=yuv420p` on each `[sv{i}]` canvas pre-scale (`[{i}:v]{canvas},format=yuv420p[sv{i}]`) is still needed to prevent AMF requesting yuv444p when inputs have mismatched color primaries (`prim:reserved` vs `prim:unknown`). A single AMF-encoded intermediate file carrying `prim:reserved` is enough to trigger the swscaler error even with the terminal format node in place. Also consider normalizing color primaries on all intermediates with `-colorspace bt709 -color_primaries bt709 -color_trc bt709`.
- **`color` inline sources for `xfade` must carry `settb` matching the clip container timebase** — `color=c=black:r=30000/1001` generates timebase `1001/30000`; libx264 H264 output uses `1/30000` (tbn = fps numerator). FFmpeg 6.1.1 `xfade` rejects mismatched timebases with exit 234. Always append `,settb=1/30000` (derived via `_fps_to_tbn(target_fps_raw)` in render.py) to every inline `color` source paired with `xfade`. Applies to both the post-pass open/close blacks in `build_open_close_post_fc()` and the monolithic path in `build_filter_complex()`.
- **`plan_video_batches()` needs POST-normalise durations for ALL clips before batch 1 can start — this blocks any "start encoding early" optimization without a planner rewrite.** `clamp_xfade_dur()` (called before the planner, `render.py:855`) clamps a single global `xfade_dur` to the shortest clip in the *whole* project, and the planner itself (`transitions.py:90`) computes a global prefix-sum/offset table across all N clips to validate every batch boundary's solo region, raising `ValueError` up front if any boundary is invalid (the clean fallback-to-monolithic signal). Both depend on `get_duration()` on the post-normalise/post-trim file (`render.py:724`), so batch 1's encode cannot start until the *last* clip has been normalised — not just batch 1's own clips. A true "start batch 1 while later clips still decode" fix means switching the duration source to something available pre-normalise (manifest `in_ms`/`out_ms`, or a fast ffprobe on the source clip) and restructuring the planner to compute/validate incrementally, which also needs a new answer for what happens if a *later* batch's boundary turns out invalid after earlier batches already encoded (today's clean upfront-fallback guarantee no longer holds as-is). Investigated for #20 (V4.2) — not built; see issue comment for the measurement that showed it isn't worth building (proxy reuse already keeps `t_normalise_s` under ~5% of total in every real render logged).
- **Background proxy generation reliably beats a human-paced render to the punch — hard to get a genuinely "cold" (un-proxied) render for testing.** Creating a brand-new project and manually clicking through Upload→Trim→Arrange→Render gives background low-priority proxy gen (`generate_proxy_file_low_priority`) enough real time to finish on a handful of clips before the render starts, even when the goal is specifically to measure the un-proxied path. Confirmed in the #20 investigation: a fresh 9-clip 4K project showed `proxy_status='done'` for every clip by render time. If a genuinely cold measurement is ever needed, either render immediately (script-driven, no human navigation delay) or explicitly null `proxy_status`/`proxy_path` for the clips first — don't assume a "new project" alone stays cold.

## Invocation

```
wsl -d Ubuntu-24.04 -u root -- python3 /mnt/c/apps/rushcut/pipeline/run.py --job-id <uuid> --manifest-path <wsl_path>
```

Manifest JSON written to `%TEMP%\rushcut\<job_id>.json` by Rust. Contains clips array + settings + output_path.

## Python pitfalls

- **Relative imports:** `run.py` inserts the *parent* of `pipeline/` into `sys.path` and imports as `from pipeline.render import run_pipeline`. Never insert `pipeline/` itself — breaks all relative imports.
- **PowerShell `Out-File` writes UTF-8 BOM:** Python `json.loads()` raises `JSONDecodeError`. Write manifests via WSL or Python only.
- **Windows path to WSL:** `C:\clips\DJI_01.MP4` → `/mnt/c/clips/DJI_01.MP4`.
- **`run.py` config completeness:** Every `JobConfig` field needs `settings.get(key, safe_default)`. Missing fields silently use wrong defaults.
- **`or default` idiom silently promotes `0` / `0.0` to `default`** — `float(d.get("clip_volume", 1.0) or 1.0)` coerces Python's falsy `0.0` (muted clip) to `1.0` (full volume) because `0.0 or 1.0 == 1.0`. Rule: whenever the valid value range includes `0` or `0.0`, use an explicit None check: `float(v) if v is not None else default`. This applies to any numeric manifest field where zero is a valid non-default (volume, start offset, card opacity, etc.).
- **Multi-line diagnostic/repair scripts must be written to a temp `.py` file — never passed inline via `-c`** — both Git Bash `-c` strings and PowerShell inline strings corrupt Python quote characters in multi-line scripts, producing `SyntaxError: invalid syntax` or `unexpected token '('`. Pattern: write to `%TEMP%\rushcut\diag_NAME.py` (or `repair_NAME.py`) using the Write tool, then invoke via PowerShell: `wsl -d Ubuntu-24.04 -u root -- python3 /mnt/c/Users/Manasak/AppData/Local/Temp/rushcut/NAME.py`. Keep scripts on disk — useful for re-inspection. Confirmed in V1.1 session.

## FFmpeg rules

- **`crop` filter latches `iw`/`ih` at the first frame when upstream `scale=eval=frame` produces variable-size output** — `iw` in the crop expression is frozen at init. For gradual zoom-in this means `(iw-ow)*fx = 0` always (top-left lock). For zoom-out it means x overflows the valid range as zoom shrinks → rightward drift then hard left snap. Fix in `_kenburns_vf` (`zoom.py`): substitute Python-constant source dimensions with the same `zf(t)` expression directly in the crop origin: `crop=out_w:out_h:'(2*trunc(SRC_W*ZF/2)-out_w)*fx':'(2*trunc(SRC_H*ZF/2)-out_h)*fy'`. Bump `_KENBURNS_CACHE_VER` in `render.py` whenever the formula changes — the cache key does not auto-invalidate on code changes.
- **`aevalsrc` sample-rate param is `s=` not `r=` in FFmpeg 6.1.1** — `aevalsrc=0:c=stereo:d=5.0:r=48000` raises exit 8 "Option not found" under FFmpeg 6.1.1 (Ubuntu 24.04 apt). Use `s=48000`. All 3 occurrences in `transitions.py` `build_audio_only_fc()` must use `s=`.
- **DJI Osmo Pocket 3:** Two video streams — use `-map 0:v:0` (HEVC real clip). Stream 1 is an embedded MJPEG thumbnail.
- **Portrait clips:** 1728×3072 normalises to 608×1080 via `scale=-2:1080`. Correct — do not "fix" orientation.
- **Encoding:** Always `-c:v libx264 -pix_fmt yuv420p -profile:v main`. Omitting allows silent HEVC fallback (Windows Media Player rejects HEVC).
- **xfade name:** `xfade=transition=fade` (NOT `crossfade`). Dip-to-black = `xfade=transition=fadeblack`.
- **scale inside filter_complex:** Never mix `-vf` and `-filter_complex` on the same output stream.
- **Fixed-canvas pre-scale for portrait+landscape mixing:** `transitions.py` pre-scales every input to an exact canvas (`scale=W:H:force_original_aspect_ratio=decrease,pad=W:H:(ow-iw)/2:(oh-ih)/2`) with named labels `[sv0]`,`[sv1]`... before any concat or xfade. Do NOT use `scale=-2:{h}` after the chain — that was the crash (different widths entering xfade = FFmpeg exit 234). Both `"none"` and xfade paths use this approach.
- **normalise.py uses `ultrafast` preset for both modes** — normalised files are intermediates re-encoded by the render step. `fast` or better wastes CPU on motion estimation. Final mode changed from `fast` → `ultrafast` in Batch 13b.
- **xfade_dur clamp:** `transitions.py` clamps to `min(1.5, min_clip_dur / 2.0)`. Do not remove — prevents short clips (e.g. 3s cards) from being consumed.
- **`-map 0:a:0?` not `-map 0:a?`** — DJI clips can contain multiple audio streams.
- **Audio for xfade transitions (N>=2):** Pairwise chained `acrossfade` for ALL N>=2. `apad=whole_dur=durations[i]` per clip normalises audio to exact video duration, so crossfade start point (`dur - xfade_dur`) aligns with the xfade visual offset. Do NOT use `concat` for 3+ clips — hard cuts lag audio 1.5s behind video at every cut after the first (accumulates per cut).
- **Music loop: `asetpts=PTS-STARTPTS` before `atrim` when using `-stream_loop -1`** — loop assigns continuously rising PTS; trimming before reset cuts too early. Order: `[1:a]asetpts=PTS-STARTPTS,atrim=0:{dur},volume=...,afade=...`. See `music.py`.
- **`-ar 48000` at every re-encode site:** DJI records at 96kHz; force 48kHz at normalise, inject_silence, single-clip render, multi-clip render, music mix, and loudnorm. One missing site means 96kHz leaks into the output.
- **music_volume is now a string union:** `"subtle" | "balanced" | "prominent"`. `run.py` maps to floats `{subtle: 0.2, balanced: 0.4, prominent: 0.7}`. The old 0–100 integer slider no longer exists (changed Batch 12b).
- **pipeline/motion.py is DEAD CODE** — do not call from render.py. Motion scoring added >10 min overhead on 10 min footage. Kept for future premium AI feature only. Never re-enable without profiling total time on realistic footage first.
- **Per-clip review fields in manifest (Batch 14c):** Each clip in the manifest now carries `in_ms`, `out_ms`, `focal_x`, `focal_y`, `zoom_mode`. `render.py` Step 2 checks user `in_ms`/`out_ms` first (ms -> seconds); silence detection is fallback only. Step 3 checks per-clip `zoom_mode` + focal point before global `config.zoom`.
- **zoom.py focal offset clamping:** `x = min(max(iw*fx - iw/zoom/2, 0), iw - iw/zoom)` — without clamping, focal points near edges produce out-of-frame pans that crash or show black bars. Always log computed x/y/zoom values before passing to FFmpeg.
- **`crop` filter has NO `eval` option (FFmpeg 6.1.1)** — writing `crop=W:H:x_expr:y_expr:eval=frame` raises `Error applying option 'eval' to filter 'crop': Option not found`. The `crop` filter re-evaluates `x` and `y` expressions every frame natively; only `scale` needs `:eval=frame` for time-varying width/height. Drop `:eval=frame` from any `crop` filter — the time-varying expressions still work.
- **Gradual zoom (`kb_*`) FFmpeg approach** — use `scale=w='…':h='…':eval=frame,crop=W:H:x_expr:y_expr` (NOT zoompan). The constant crop window forces output to `W×H` every frame while scale changes; both run in a single pass at ~encode speed. Speed fractions: slow=1.0, med=0.75, fast=0.5 of trimmed clip duration. Comma-free smoothstep clamp: `clip(t/M,0,1)` = `(a+1-abs(a-1))/2` where `a=t/M`.
- **`start_job` filters `include==0` clips** at manifest time (Rust). Empty manifest (all clips skipped) returns error to UI. `out_ms` clamped to `duration_ms` to prevent FFmpeg crash on out-of-bounds trim.
- **Zoom is now embed-in-filter_complex — no pre-encode step, no cache files (#67/#79 shipped).** `build_zoom_vf()` in `zoom.py` returns a per-clip vf string (`scale=eval=frame,crop=...`) that `render.py` Step 3 injects into the `[sv{i}]` canvas chain in filter_complex. AMF absorbs `scale=eval=frame` at hardware speed with zero render-step overhead (benchmark: 19 clips 4K cold, -63% total time, Δt_render_s=0). `zoom_cache.py`, `warm_zoom.py`, and `warm_zoom_cache_cmd` no longer exist — do not attempt to recreate them.
- **`zoom_vfs` must stay 1:1 with `current_paths` through Step 4 card insertion — fixed #98, 2026-07-10.** `zoom_vfs` is sized in Step 3 (per-clip zoom) to `len(current_paths)` *before* intro/outro cards are prepended/appended. Unlike `clip_volumes` (already correctly extended at both card-insertion sites), `zoom_vfs` was not — U1g's segmented path does an unguarded `zoom_vfs[k]` lookup by global clip index (`render.py:917`), so any batch whose `idxs` includes a card position beyond the original `zoom_vfs` length raises `IndexError`. The monolithic path never crashed on this because `_sv_node` (`transitions.py:214`) has an inline `i < len(zoom_vfs)` bounds guard — it silently skips zoom for the out-of-range index instead, which is *also* wrong (an intro card shifts every real clip's index +1, so per-clip zoom settings get silently misapplied to the wrong clip). Fix: extend `zoom_vfs` alongside `clip_volumes` at both card-insertion sites (`if zoom_vfs is not None: zoom_vfs = [None] + zoom_vfs` / `zoom_vfs + [None]`), plus a length-match `assert` right before Step 5 as a tripwire. **Any future array added that is 1:1 with `current_paths` (built in Step 3 or earlier) must be extended at both card-insertion sites too** — this is a recurring class of bug, not a one-off (see #103, filed as tech debt to replace the parallel-array pattern before mid-roll cards are built).
- **Single-pass loudnorm — still exactly once, but relocated (V4.1 render cache, #19).** Loudnorm is no longer baked into any Step 5 encode site. Step 5 (monolithic `build_filter_complex`, U1g `_render_segmented`, and the single-clip shortcut) always produces a *treatment-free* clean intermediate — no loudnorm in any of the three. Loudnorm now applies exactly once in a shared `_apply_audio_treatment()` step that runs after the render-cache read/write point: when `music_on`, it fuses into the Step 6 `mix_music()` encode (`apply_loudnorm=(mode != "draft")`, unchanged); when music is off, it's a dedicated fast pass — `-c:v copy -af loudnorm... -c:a aac` — a remux, not a re-encode, since loudnorm is audio-only. Do not re-add loudnorm to any Step 5 site — that would double-compress (loud-soft-loud pump) whenever `_apply_audio_treatment` also fires. The `loudnorm_filter()` helper in `loudnorm.py` is still the single source of the filter string.
- **Render cache (`pipeline/render_cache.py`, V4.1/#19) — DaVinci-style clean-intermediate cache.** After Step 5 produces the treatment-free clean intermediate (video + transitions + zoom + cards + clip audio, no music/loudnorm), it's cached at `%APPDATA%\rushcut\render-cache\<sig>.mp4` (mirrors the proxy cache directory pattern). A later render with the same signature copies the cached file and skips Steps 1-5 entirely — only `_apply_audio_treatment` runs, so changing music mood/volume/track/fade/loop on a re-render is a ~5s remux instead of a full re-encode (confirmed live: 142s -> 7s, 8 clips/1080p).
  - **Signature source: `clips` (original manifest), NEVER `pipeline_clips`.** `pipeline_clips` is mutated by the B-0 pre-trim to hold pretrim-relative `in_ms`/`out_ms` offsets; hashing those instead of the absolute manifest values makes the signature drift between otherwise-identical renders and the cache never hits. Compute the signature before any B-0 mutation.
  - **In the signature (invalidates the cache):** per-clip `in_ms`/`out_ms`/`zoom_mode`/`focal_x`/`focal_y`/`clip_volume` + clip order/identity; global `transition`/`opening_transition`/`closing_transition`/`shuffle_between`/`output_resolution`/`mode`/`target_fps_raw`, and intro/outro card text+colour.
  - **NOT in the signature (cheap layer, reapplied on every hit):** `music_mood`/`music_volume`/`custom_music_path`/`music_fade_out`/`music_loop`, and loudnorm itself.
  - **Shuffle is cacheable (fixed #83).** `resolve_cut_names`/`build_filter_complex` now seed the shuffle RNG with `cache_sig` itself (the same stable hash used for cache lookup), not `job.id`. Identical clip-set+settings reproduce the identical shuffle cut order across renders, so `use_cache = mode == "final"` — no shuffle exclusion. Any settings/clip change that would invalidate the cache also changes `cache_sig`, so shuffle order and cache validity always move together.
  - **Cache write is atomic within the cache dir, not across filesystems** — the `.tmp` file is written directly inside `render-cache/` (NTFS) and swapped in with `os.replace()`. Do NOT `shutil.move()`/`os.rename()` from the WSL tmpfs work dir straight into `render-cache/` — that crosses filesystems (tmpfs -> NTFS) and raises `OSError: Invalid cross-device link` on WSL.
  - **Draft mode never reads or writes the cache** (480p throwaway output).
  - Deferred (not built in #19): Rust manifest wiring for the cache dir, a `render_cache_hash` DB column, and a robust cross-project vacuum (current `prune()` is a self-contained age+count bound in Python only).
